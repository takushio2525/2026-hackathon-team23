## 成果物（議事録 10-1）

本プロジェクトで最終的に提出する成果物を、第 4 回計画書で以下の通り明示した：

1. **Arduino R4 WiFi UNO を用いたオーケストラの実施**
2. **4 種類の音色を生成するプログラム**（金管・木管・弦・パーカッション想定）
3. **指揮棒を用いたゲーム**（BPM 計測 + フロントエンド UI）
4. **システム構成図**
5. **動作確認結果および評価資料**
6. **プロジェクト計画書および最終報告書**

## プロジェクトの対象範囲（議事録 10-2-1）

以下を **本プロジェクトの本実装スコープ** として扱う。

- Arduino を用いた指揮動作検出システムの開発
- 加速度センサを用いた指揮棒の振り動作の取得
- 指揮棒の振り周期に応じた音楽テンポ制御機能
- エッジ検出によるテンポ変化機能
- 指揮棒の振り速度に応じた音量変化機能
- ハードウェアとソフトウェアを組み合わせたシステム統合
- システムの動作確認および性能評価

## 必達ゴール（Must）

上記スコープから逆算した「最低限ここまで作れば成立」のライン：

1. **指揮者 1 台 + 楽器 3 台以上** が同期演奏できる
2. **輪唱曲** を演奏できる（主旋律 3 + 必要ならリズム 1）
3. **可変テンポに対応** する（指揮の速度に追従、≤ 2 拍で追従完了）
4. 同期誤差 **≤ 20 ms** を達成する（[ADR-0006](/decisions/0006-sync-error-moe-20ms/) /
   議事録 12-1 MOE 2）
5. **どの方向に振っても拍を検出可能**（振り方に依存しない）
6. **指揮棒練習用のゲーム要素**（BPM 計測 + フロントエンド UI）を含む

現状（2026-05-14）の達成状況: **1〜5 はテスト v2 で達成、6 は製造フェーズで実装予定**。
node_05（4 声目）は未実装。

## ストレッチゴール（議事録 11 章 表 3 ストレッチフェーズ）

設計・製造・テストの 3 フェーズが計画通り進んだ場合に着手する拡張：

7. **強弱（velocity）推定アルゴリズム** の実装（WBS 411、塩澤担当 1 週）
8. **CTRL パケットへの velocity 乗せこみ**（WBS 412、塩澤担当 1 週）
9. **強弱制御テスト**（WBS 413、塩澤担当 1 週）

これらは IMU の振幅から音量にマッピングする系統で、本実装スコープ内のテンポ追従が
安定してから着手する。

## プロジェクトの非対象範囲（議事録 10-2-2）

明示的に **やらない** と決めている事項。スコープ肥大化を防ぐためのガード：

- プロオーケストラレベルの高精度な指揮解析
- AI を用いた自動指揮認識機能
- 複数人による同時指揮への対応
- 市販音楽制作ソフトウェアとの連携
- 楽曲の自動生成機能
- 複雑なジェスチャ認識機能

加えて、過去の判断（ADR-0007）から：

- カメラ画像認識による指揮認識
- スマホ連動（BLE / アプリ）
- 照明同期（DMX / Hue 等）
- 楽譜のリアルタイム編集機能
- SD カードや外部ストレージへの依存（楽譜は内蔵）
- 1 台に複数声部を持たせる構成（1 台 1 声を堅持）

[ADR-0007](/decisions/0007-project-purpose-and-scope/) で「外的要因に作用されない」を
主目的に据えたため、外部機材・外部サービスに依存するものは原則すべて範囲外。

## 制約条件

| 項目 | 制約 |
|---|---|
| ハードウェア台数 | **Arduino UNO R4 WiFi × 5 台**（人数分 − 1） |
| パート差別化 | 各楽器は異なるパート（同一パート複数台は NG） |
| 機材調達 | 実験室の機材は貸し出し可。無いものはチームで購入 |
| はんだ付け | 各自で実施 |
| 期間 | 約 13 週（実装 6 週） |
| 発表 | 計画発表 2026-05-13、成果発表 2026-07-01 |

### 指揮者の IMU について

[ADR-0003](/decisions/0003-conductor-imu/) では「Arduino UNO R4 WiFi の本体内蔵 IMU」を
採用する想定だったが、実機検証で性能不足が判明し、`test_v2` 系では **XIAO ESP32-S3 Sense +
GY-521（MPU6050）** に切り替えている。

## 進捗の見方（議事録 12 章 V&V 計画）

第 4 回計画書で MOE / MOP / TPM の体系を以下に整理した。

### MOE（有効性指標）と対応 MOP・目標値

| 有効指標 (MOE) | 性能指標 (MOP) | 目標値 / 達成水準 |
|---|---|---|
| **MOE 1: 音楽的再現性と正確性**（単体テスト領域） | 拍検出の正解性 | 正解率 ≥ 90% |
|  | 音階の誤差 | セント単位の許容範囲内（自己相関ベース 5.52 cent） |
|  | 楽譜との相違 | 誤ノート発音数 0 |
| **MOE 2: 合奏の一体感とインタラクティブ性**（結合テスト領域） | 楽器間同期誤差 | ≤ 20 ms |
|  | 指揮から楽器への通信遅延 | ≤ 30 ms |
|  | テンポ追従の遅延 | ≤ 2 拍 |
| **MOE 3: システムとしての堅牢性と実用性**（システムテスト領域） | 起動時間 | ≤ 5 s |
|  | CPU 負荷（入力フェーズ） | ≤ 2 ms |
|  | パケロスの許容範囲 | 演奏継続が可能なレベル（聴感破綻なし、目安 5% 以内） |

### TPM（技術性能指標 = 開発中に追跡する内部指標）

- **同期誤差の推移監視**: 目標 20 ms に対し、乖離 10 ms 以上でバッファ制御を再検討
- **CPU 処理時間のマージン監視**: 制御周期 5 ms に対するタスク処理時間。目標値 2 ms の
  80% を超えた段階でコード最適化／タスク優先度を再設計
- **機能実装の完了率**: WBS ベース。指揮解析と通信基盤の統合が遅延したらクリティカル
  パス上のリスクとして要員配置・スコープを再検討
- **システム稼働の安定度**: 連続稼働中の意図しないリセット／ハングアップ頻度
