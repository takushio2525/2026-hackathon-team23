## 先に結論

各マイコンの時計は、起動した時刻も進む速さも少しずつ異なります。さらにWi-Fiでは、同じBEATが各楽器へ届く時刻も異なります。

そのため楽器は、受信した瞬間に音を出しません。**指揮者が指定した同じ未来の時刻へ、自分の時計を変換して待つ**ことで、発音時刻をそろえます。

## 記号と観測値

| 記号 | 意味 | 単位 |
|---|---|---|
| \(M_s\) | 指揮者が送信した時刻 | ms |
| \(L_r\) | 楽器が受信したローカル時刻 | ms |
| \(d\) | ネットワーク配送遅延 | ms |
| \(\Delta\) | 真の時計差、master-local | ms |
| \(T_p\) | `playAtMasterMs`（発音予定時刻） | ms |

楽器が観測できる値は次です。

\[
x = M_s - L_r = \Delta - d
\]

片方向通信だけでは、時計差\(\Delta\)と配送遅延\(d\)を完全には分けられません。ここでは、配送遅延が最も小さい観測を使い、真の時計差に近づけます。

## 最小遅延に近い時計差を選ぶ

SoftAPのマルチキャストは、省電力向けのバッファリングの影響で、約204.8 ms周期のまとまりとして届くことがあります。遅れて届いた観測を平均すると、楽器の推定時計まで遅れてしまいます。

productionでは、直近 **2秒** の観測窓で最大の`offsetSample`を採用します。`offsetSample = master - local`は、配送遅延が小さいほど大きくなるためです。

```text
offsetSample = timestampMs - localReceiveMs
masterNow    ≒ localNow + offsetMs

2秒の窓で最大の offsetSample
  = 最も早く届いた信号に近い観測
  = 配送待ちの影響が小さい時計差
```

- より大きいサンプルが来たら、すぐに時計差を更新する
- 窓が終わったら、その窓の最大値へ引き直す
- 5サンプルは診断上の収束目安で、最初の音を待たせる条件にはしない
- 指揮者再起動で差が1000 ms以上飛んだら、窓を待たず1パケットで追従する

## 220 msの発音予約

拍を検出した指揮者は、次の未来時刻をBEATに載せます。

\[
T_p = M_b + 220\ \mathrm{ms}
\]

楽器はこのマスタ時刻をローカル時計へ変換します。

\[
L_p = T_p - \widehat{\Delta}
\]

2 ms周期のループで`localNow >= Lp`になったらNOTEをPCへ送ります。受信時点ですでに予約に遅れていた場合も、音を捨てず即時発音します。

### なぜ220 msか

実測では、マルチキャストの配送待ちが最大約205 msでした。旧45 msの予約では、多くのBEATが予定時刻に間に合いませんでした。220 msは、この配送待ちと少しの処理余裕を吸収するための値です。

代償は、指揮から音までに約220 msの一定遅れが入ることです。タクトーンは「小さいが毎回違う遅れ」より、**全楽器が同じだけ遅れる安定性**を優先しています。

## タイムライン例

指揮者時計10000 msで拍を検出し、楽器時計が指揮者より120 ms遅れている場合を考えます。

```text
master beat      = 10000
playAtMasterMs   = 10220
estimated offset = +120
target local     = 10220 - 120 = 10100
```

この楽器は自時計10100 msまで待ちます。ほかの楽器のローカル時計が異なっていても、全員が指揮者時計10220 msを目標にできます。

## 4連送と重複排除

BEATは4回、2 ms間隔で送ります。4つは同じ`beatNo`と同じ予定時刻を持ちます。

- 楽器は同じ`beatNo`を1回だけ発音予約する
- 重複は時計同期の観測には使う
- 後から届く重複は通常より遅いので、最大offsetを押し上げず、発音予定を遅らせない

## 実測結果と限界

最終構成では、173拍×5楽器の発火時刻をデバイス側で計測しました。

| 指標 | 値 |
|---|---:|
| 楽器間発音差の中央値 | 7 ms |
| 平均 | 10.8 ms |
| 20 ms以内 | 90.8% |
| 最大 | 65 ms |
| 予約に間に合わない受信 | 3.1% |

最大値の外れは、Wi-Fiモジュール周辺の周期的なループ停止と見られる挙動に対応します。これは予約時間を増やすだけでは消えないため、計測手段を残して将来の改善対象としています。発表用の説明は[想定問答](/presentation/faq/)も参照してください。
