## 問題設定

各マイコンの`millis()`は起動時刻も発振誤差も異なります。BEATを受信した瞬間に鳴らすと、
Wi-Fiの到着ジッタがそのまま楽器間の発音差になります。

必要なのは時計表示を完全一致させることではなく、**同じマスタ時刻Tを各楽器のローカル時刻へ変換すること**です。

実装：

- 指揮者送信：`firmware/production/node_01/lib/OrcSenderModule/`
- 時計推定：`firmware/production/common/lib/OrcReceiverModule/`
- 発音判定：`firmware/production/node_02〜06/src/applyPattern.cpp`

## 記号

| 記号 | 意味 | 単位 |
|---|---|---|
| \(M_s\) | 指揮者が送信した時刻 | ms |
| \(L_r\) | 楽器が受信したローカル時刻 | ms |
| \(d\) | ネットワーク遅延 | ms |
| \(\Delta\) | 真の時計差、master-local | ms |
| \(\hat{\Delta}\) | 楽器が推定する時計差 | ms |
| \(T_p\) | `playAtMasterMs` | ms |

受信観測は次です。

\[
x = M_s - L_r = \Delta - d
\]

片方向通信なので遅延\(d\)と時計差\(\Delta\)を完全分離できません。ただし同じAP・同じパケットを使う楽器間では
遅延の共通成分が相殺され、重要な**楽器間差**を小さくできます。

## EMAオフセット推定

\[
\hat{\Delta}_n = (1-\alpha)\hat{\Delta}_{n-1} + \alpha x_n
\]

新しいCTRLまたは新規BEATは\(\alpha=0.20\)、同じBEATの2発目以降は\(\alpha=0.05\)です。

### なぜ重複を弱くするか

4連送はすべて同じ送信時刻を持ちますが、後続ほど2 msずつ遅く到着します。
同じ係数で4回更新すると「遅い観測」を過大評価するため、重複だけ0.05へ下げます。

### 収束の目安

初期誤差の残りは\((1-\alpha)^n\)です。alpha=0.20、5サンプルでは：

\[
(1-0.20)^5 = 0.32768
\]

初期誤差の約67%を除去します。`clockSyncMinSamples=5`は診断用の収束目安で、Playing開始の必須条件ではありません。

## マスタ再起動のスナップ

指揮者の再起動で`millis()`が大きく巻き戻ると、EMAだけでは復帰に時間がかかります。

\[
|x_n-\hat{\Delta}_{n-1}| \ge 1000\ \mathrm{ms}
\]

なら再起動とみなし、\(\hat{\Delta}_n=x_n\)を即採用します。通常の数十msジッタでは届かない閾値です。

## 45 ms先読み

拍検出時刻を\(M_b\)とすると：

\[
T_p = M_b + 45\ \mathrm{ms}
\]

楽器は次でローカル発音時刻を求めます。

\[
L_p = T_p - \hat{\Delta}
\]

2 msループで`localNow >= L_p`になった瞬間にNOTEを送ります。

## タイムライン例

指揮者時計10000 msで拍を検出し、楽器時計が指揮者より120 ms遅れているとします。

```text
master beat      = 10000
playAtMasterMs   = 10045
estimated offset = +120
target local     = 10045 - 120 = 9925
```

楽器は自時計9925 msまで待ちます。各楽器が異なるローカル時計でも、同じ10045 msを目標にできます。

## 4連送と先読み予算

productionではBEATを4回、2 ms間隔で送ります。

```text
送信1: t + 0 ms
送信2: t + 2 ms
送信3: t + 4 ms
送信4: t + 6 ms
送信API・無線処理を含む完了目安: 約8 ms
```

45 msから約8 msを引いても、Wi-Fiジッタと受信処理に約37 ms残ります。

## 誤差予算

発音差の主な成分を概算します。

| 要因 | 代表値・上限 |
|---|---:|
| 楽器ループ量子化 | 0〜2 ms |
| USB Serialの差 | 約1〜3 ms |
| offset推定残差 | ネットワーク差に依存 |
| Wi-Fi到着ジッタ | 先読み内なら発音時刻へ直接乗らない |
| Processingキュー・描画 | 音声スレッドへ渡すまでの差 |

先読み前にパケットが到着すれば、Wi-Fi到着時刻ではなく予約時刻で揃います。
目標20 msはこれらの合計と実機測定で評価します。

## 遅着

受信時点で`waitMs <= 0`ならNOTEを即送信します。遅着を破棄するとその声部だけ欠音になるためです。
この場合だけネットワーク遅延が発音差へ直接現れます。

## beatNoによる時間以外の同期

時刻が揃っても楽譜位置が違えば合奏になりません。BEATは`beatNo`も運び、楽器は毎拍：

\[
cyclePos=(beatNo-1)\bmod 56
\]

から位置を再計算します。時計同期と楽譜同期を別レイヤに分けています。

## PC途中起動

同期は楽器で完了してからNOTEを出します。PCはマスタ時計へ参加しません。
途中起動しても次のNOTEを鳴らすだけで、次のbeatNoが現在の楽譜位置を保証します。

## 32 bitラップ

`millis()`は約49.7日でラップします。未来判定は符号付き差分：

```cpp
if ((int32_t)(now - target) >= 0) {
  fireEvent();
}
```

を使い、単純な`now >= target`によるラップ境界の誤判定を避けます。

## デバッグ

- 全ノードの`beatNo`を比較
- `offsetMs`の分布とスナップ発生を記録
- 指揮者の`playAtMasterMs`と楽器NOTE時刻を突合
- 4連送のうち何発届いたかを確認
- MOP4では複数USB受信時刻の最大差を20 ms基準で判定

## 調整時の注意

- 先読みを短くすると体感は速いが遅着が増える
- 長くすると安定するが振りから音までの遅延が増える
- EMAを大きくすると追従は速いがジッタを拾う
- 冗長数だけ増やしてgapを増やしすぎると先読み予算を消費する
- 変更は`beatLookaheadMs`、`beatRedundancy`、`beatGapMs`を一組で測る
