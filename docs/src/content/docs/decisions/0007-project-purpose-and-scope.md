## 背景

第 1 回ミーティング（2026-04-15）では「Arduino 5 台で指揮者 + 楽器」「UDP 通信」
「独創性は指揮のジェスチャ」といった**構成の大枠**を合意したが、
**「何のために作るのか（目的）」「どこまで出来れば完成か（ゴール）」** の
言語化は未着手だった。

第 2 回ミーティング（2026-04-22、目的：案の一本化）で、各メンバーが持ち寄った
観点を集約し、プロジェクトとしての目的・ゴール・成果物範囲を確定した。

## 決定事項

### 目的（Purpose）

> **外的要因に作用されず、機器さえあれば（音楽未経験でも）演奏できること。**

各メンバーの観点を集約した結果：

| 提案者 | 観点 |
|---|---|
| 塩澤 | 演奏に人間が介入する |
| 地曵 | 音楽経験の有無に関係なく容易に演奏（テンポ変更やボリューム増減）できる |
| 梅澤・齋藤 | 時間帯や外音などに作用されず安定して演奏できる |
| 片岡 | 場所にとらわれず、音楽未経験でも演奏できる |

### ゴール（Goal）

> **ユーザーが指揮を振るテンポを変えたときに演奏速度も変えられる、簡単な演奏体験**。
> 具体的には「指揮の振り方が乱雑（どんなモーション）でもテンポを検出でき、
> 同期誤差 20 ms 以内なら同期できているとみなす」（塩澤）。

### 成果物の範囲

本プロジェクトで **実装する** 範囲：

1. **マイコンを振ってエッジ検出**し、**WiFi（UDP）で拍を送信**する。
2. **同期誤差 20 ms 以内**を満たす（[ADR-0006](/decisions/0006-sync-error-moe-20ms/)）。
3. **倍音データ**などを元に**根拠を持った音色作り**を行う。
4. **どの方向に振っても拍を検出可能**（振り方に依存しない）。

### エンタメ性の方針

> **「指揮を振る体験」を主なエンタメ性として据える**。

第 2 回で挙がった拡張案（ゲームモード・演奏モードの切替・指揮棒の LED 発光）は、
この主軸の上に乗せる**拡張**として位置づける（実装の優先度は別途）。

## 理由

- **目的**: 天候・照明・ネット環境等の外的要因に依存しない設計にすることで、
  デモ時の再現性を担保し、評価場面での不確実性を下げる。
  これはカメラ画像認識ではなく IMU ベースを採用した判断（[ADR-0003](/decisions/0003-conductor-imu/)）と整合。
- **ゴール**: 「誰が振っても動く」を担保することで、ユーザビリティと独創性の評価軸を両取りする。
- **スコープ限定**: 倍音ベースの音色作り・方向非依存のエッジ検出まで含めることで、
  「音が楽器らしいこと」「指揮らしいこと」の 2 軸で差別化する。
- **エンタメ主軸の言語化**: 機能の肥大化を防ぎ、優先度判断の基準を明確化する。

## 結果・影響

### ポジティブ

- FBS／PBS／WBS／MOP／V&V／TPM を次回確定するうえでの**上位要求**が揃った。
- 実装タスク（WBS 100/200/300 系）の採否判断に「本 ADR の成果物範囲にあるか」を使える。
- 明示的に範囲外となる機能（例: 照明同期・スマホ連動など）を気兼ねなく落とせる。

### ネガティブ / リスク

- 「倍音データを元に根拠を持った音色作り」は定量基準が未設定で、実装工数の見積りが甘い可能性。
- 「どの方向に振っても拍を検出可能」は IMU 信号処理の難度が高く、ゴール到達リスクが残る。

## 未決事項（本 ADR の範囲外）

- 楽譜（曲）の選定
- 5 声（[ADR-0004](/decisions/0004-ensemble-structure/)）の役割分担（どのノードがどの声部か）
- ゲームモード等の拡張要素の採否
- 音階の正確性の定量基準

## 次アクション
