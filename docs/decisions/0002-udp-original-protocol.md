# ADR-0002: 通信方式は UDP 独自プロトコルを採用

**ステータス**: Accepted
**決定日**: 2026-04-15（第1回ミーティング）
**関連議事録**: [`meetings/23_第1回議事録_24G1131-1776302030881.pdf`](../../meetings/23_第1回議事録_24G1131-1776302030881.pdf)

## 背景

Arduino オーケストラでは、指揮者マイコン（1台）から楽器マイコン（4台）へ
テンポや強弱などの演奏制御情報を**ほぼリアルタイム**で配信する必要がある。
さらに、楽器マイコン各々から音符情報（高さ・強さ・長さ・タイミング）を
Processing 側（PC）へ送って音色合成させる構成も前提となる
（[Kickoff 資料 p.62 "音の作成"](../../references/lectures/HCK1_01_Kickoff_A.pdf)）。

候補となった通信方式：

| 方式 | 長所 | 短所 |
|---|---|---|
| UDP（独自プロトコル） | 低遅延・ブロードキャスト容易・Arduino UNO R4 WiFi に標準対応 | パケロス対策を自前で設計 |
| TCP | 到達保証・再送自動 | コネクション管理コスト・遅延が読みにくい |
| OSC | 楽器制御の業界標準・Processing/MIDI連携しやすい | OSC over UDP を結局 UDP で実装することになる |
| ESP-NOW | 超低遅延・AP不要 | UNO R4 WiFi では非対応（ESP32 系の機能） |
| MIDI over Serial/BLE | 楽器制御の定番 | 5台同時配信には向かない |

## 決定事項

通信方式は **UDP ベースの独自プロトコル** とする。
去年度（関根先生の案）を参考にしつつ、同期誤差が許容範囲を超えた場合は
再検討する。

## 理由

- Arduino UNO R4 WiFi の WiFiUDP ライブラリが標準で使え、5台の同時配信に向く
- 評価軸の一つ「演奏タイミングの誤差」に対して、低遅延・ブロードキャスト容易な
  UDP が最適
- 独自プロトコル化することで、同期タイミング用フィールドやビブラート等の
  独自表現をパケットに乗せやすい
- パケロス時の挙動（次の同期パケットで復帰する等）をシステムに合わせて自由に
  設計できる

## 結果・影響

- **ポジティブ**: 低遅延・拡張性・実装量が予測しやすい
- **ネガティブ / リスク**: パケロス・ジッタ対策を自分たちで設計・実装する必要
- **次のアクション**: 具体的なパケット構造と同期方式を
  [`docs/design/protocol.md`](../design/protocol.md) に明文化する
- **計測指標**: 同期信号受信精度と演奏タイミング誤差は評価時に必ず計測
  （[Kickoff 資料 p.65 評価](../../references/lectures/HCK1_01_Kickoff_A.pdf)）

## 再検討条件

以下のいずれかに該当する場合、本決定を見直す：

- 実機測定で楽器間の同期誤差が **> 30 ms**（人間が聞いてズレを感知する目安）
  から改善できない
- パケロス率が **> 5%** を恒常的に超え、フォールバック機構でも救えない
