# ADR-0005: ファームウェア設計は Embedded-Module-Architecture に準拠

**ステータス**: Accepted
**決定日**: 2026-04-21
**決定の場**: チーム内オフライン検討（第1回と第2回の**間**）. 会議ベースではなく、第1回ミーティング（2026-04-15）で合意した 5 台構成（[ADR-0004](0004-ensemble-structure.md)）を前提に、firmware 実装方針としてリポジトリ上で確定したもの.
**関連**: [ADR-0004](0004-ensemble-structure.md)（5台構成）、[`firmware/README.md`](../../firmware/README.md)

## 背景

本チームは Arduino UNO R4 WiFi × 5 台（指揮者 1 ＋ 楽器 4）を複数メンバーで
分担開発する。担当者ごとに書き方がばらけると、以下の問題が起きやすい：

- ループ内で入力・制御・出力のコードが混ざり、テストと差し替えが困難になる
- ノード間で同じ機能（UDP 受信・LED 表示など）を別実装してしまう
- 状態管理が `global` 変数の山になり、担当者以外が読めなくなる

授業期間内に 5 台を結合・同期させる都合上、**共通の設計パターンを最初から
固定**し、責務分離とモジュール差し替えが効く形で書き始めたい。

## 決定事項

ファームウェア（`firmware/` 配下）のコードはすべて、塩澤既存の
リファレンス実装 **Embedded-Module-Architecture** に従って書く。

- リポジトリ: <https://github.com/takushio2525/Embedded-Module-Architecture>
- ローカル参照: `/Users/shiozawatakumi/Documents/Embedded-Module-Architecture/`
- 適用範囲: `firmware/node_01/` 〜 `firmware/node_05/` 全ノード
- 共通層（`IModule`、`ModuleTimer`、通信プロトコル定義など）は
  `firmware/common/lib/` に置き、各ノードの `platformio.ini` から
  `lib_extra_dirs = ../common/lib` で参照する

### 主要パターン（リファレンス準拠）

| パターン | 役割 |
|---|---|
| 3 フェーズループ | `loop()` を **入力 → ロジック → 出力** の順で構成 |
| `IModule` インターフェース | 各機能を `setup()`/`update()` を持つモジュールとして実装 |
| `SystemData` | ノード内でモジュール間共有する状態を 1 構造体に集約 |
| `ProjectConfig` | ピン配置・定数・閾値などノード固有設定を 1 ヶ所に集約 |
| `ModuleTimer` | 周期実行・非ブロッキングな時間管理 |

新規モジュールを追加する際は、リファレンスの `ARCHITECTURE.md` に
記載のチェックリストに従う。

## 理由

- **責務分離が明快**: 入力／ロジック／出力の 3 フェーズが強制されるため、
  複数人が同じノードを触っても境界が崩れにくい
- **差し替えやすい**: モジュール単位でテスト・モック化できる。
  実機が揃う前でもロジック部分を先行開発できる
- **説明コストが低い**: 塩澤が既に習熟しており、他メンバーへの
  オンボーディングを `ARCHITECTURE.md` 1 本に集約できる
- **マルチノード適合**: `SystemData`/`ProjectConfig` のノード固有化が
  自然で、5 台のハード構成差をコード差の最小化で吸収できる

### 代替案との比較

| 代替案 | 採用しなかった理由 |
|---|---|
| Arduino ベタ書き（`setup/loop` に全部書く） | 複数人開発で即破綻。テスト困難 |
| FreeRTOS タスクベース | UNO R4 でも可能だが学習コストが重く、6 週間の実装期間に対してオーバーキル |
| 他 OSS フレームワーク（Mozzi 等） | 本件は汎用ロジック＋UDP が主で、音色合成は PC 側。フィット感が薄い |

## 結果・影響

### ポジティブ

- ノード間で共通コード（UDP 層・状態表示 LED・時間管理）を `firmware/common/lib/` に
  集約でき、重複実装を避けられる
- 塩澤以外のメンバーもリファレンスリポジトリを読めば書き方が分かる
- 単体テスト（`test/`）がモジュール単位で書きやすい

### ネガティブ / リスク

- 初回学習コスト: Arduino を素で書いたことしかないメンバーは、
  `IModule`/`SystemData` の考え方に慣れる必要がある
- ボイラープレート: 小さな機能でもクラスを 1 つ切るため、
  極小プロトタイプでは冗長に感じる場面がある
  → プロトタイプは `example_sketch/` 等の簡易スケッチで試し、
  本実装に取り込むタイミングで本アーキテクチャに載せ替える運用

### 未決事項（本 ADR の範囲外）

- `SystemData` の具体的フィールド設計（ノードごとに別途設計）
- モジュール境界の切り方（指揮者／楽器で異なる）
- ノード間同期方式（時刻同期・シーケンス番号など。別 ADR で扱う）
