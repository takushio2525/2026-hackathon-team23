# MOP5 系統シフト −42.8ms の原因分析 — 2026-07-10

対象: 7/10 の MOP4/MOP5 再計測（M45R/M45F ログ、`logs/test_20260710_221146.log`、204 拍 × 5 ノード）で観測された

- BEAT 受信時点で beatLookahead 45ms の余裕がほぼゼロ（受信マージン平均 +2.2ms、**系統シフト −42.8ms**）
- 全 5 ノードの発火が一様に p95 ≈ 78ms 遅刻（ノード別 p95 76〜82ms で横並び）
- 一方で楽器間同期誤差（MOP4）は平均 9.8ms / p95 22ms と良好

という現象の原因調査。**本調査はログ・CSV の再集計とコード読解のみで、ファーム・スクリプトへの変更は一切していない**（集計スクリプトにバグは見つからなかったため修正も不要だった）。集計はすべて生ログから独自に再実行して検算した。

## 結論サマリ

1. **根本原因は「SoftAP からのマルチキャストが 204.8ms（= ビーコン間隔 102.4ms × 2）周期のバースト配送になっていること」**。BEAT は送信スタンプから平均 ≈102ms・最大 ≈205ms 遅れて届くため、45ms の発音予約は構造的に間に合わない。周期 204.9〜205.0ms はデバイス時計・PC 時計・7/9 の別ビルドのログの 3 系統で独立に一致検出した（§3）。
2. **系統シフト −42.8ms の直接の作り手は EMA クロック同期の「サンプル鮮度の非対称」**。オフセット EMA は 20Hz の CTRL（バースト内の最新品 = スタンプ後平均 ≈25ms の新鮮なサンプル）に支配される一方、拍ごとの BEAT サンプルだけが平均 ≈102ms 古い。この混合の結果、推定マスタ時計は「真のマスタより約 40〜55ms 遅れた位置」に座り、BEAT 受信マージンから見ると lookahead 45ms がほぼ食い潰されて見える。定常状態モデルの予測シフト −48.6ms は実測 −42.8ms と ±6ms で閉合する（§4）。
3. **計測式・集計スクリプトにバグはない**（仮説 A 棄却）。ただし解釈上の重要な注意として、**M45R/M45F の lateMs は「推定マスタ時計」基準であり、その推定時計自体が平均 40〜55ms 遅れているため、真のマスタ時計基準の遅刻は表示値よりさらに大きい**。最小遅延線基準の推定では発火の真の遅刻は平均 ≈+80ms / p95 ≈+145ms / 最大 ≈+168ms、受信時点の遅刻率は 45.4% でなく ≈79%（§5）。
4. **「全ノード横並びで遅れるのに楽器間は 10ms で揃う」のは、遅延が完全な共通モードだから**。5 台は同一のバースト（同一 DTIM フラッシュ）で BEAT を受け取り、推定時計のバイアスも同じ機構で同じ量だけ乗る。楽器間に残る 8〜22ms は発火ループ粒度（実測 +3〜4ms）・EMA の個体ノイズ・電波の個体差の和で説明できる（§6）。
5. 副産物として、**UNO R4 WiFi の millis() が指揮者比 −400〜−3200ppm（最大 0.32%）も遅い**ことを検出した（RA4M1 は内蔵発振器 HOCO 駆動でこの規模の誤差は仕様相当）。ただし EMA が吸収するため本件のシフトへの寄与は 2ms 未満（§7）。
6. 7/9 調査レポート（`MOP45_latency_investigation_20260710.md` §4.3）の「未解明の系統シフト 45〜55ms」は本調査で**同一原因として解明済み**。

## 1. データと検算

- 生ログ: `logs/test_20260710_221146.log`（M45R 1024 行 / M45F 1020 行、5 ポート、約 103 秒、平均拍間隔 504ms）
- 集計 CSV: `results/mop4/20260710_221532.csv` / `results/mop5/20260710_221532.csv` と各 summary
- ログ形式: `M45R/M45F,<partId>,<beatNo>,<playAtMasterMs>,<deviceMs>,<offsetMs>,<localMasterMs>`（localMasterMs = deviceMs + offsetMs）
- 検算: `mop5_comm_delay.py` と同一ロジックを独立実装し、受信マージン平均 +2.2ms / 最小 −98 / 最大 +90、発火 lateMs 平均 23.7 / p95 78 / 最大 101ms が summary と全一致することを確認した。

記号の整理（以降で使用）:

- `ts` = 指揮者の送信スタンプ。`OrcSenderModule.cpp:11,20,24` で `masterNow = millis()` を読み、`header.timestampMs = masterNow`、`playAtMasterMs = masterNow + 45` を**同じ時刻から**設定する。よって `ts = playAtMasterMs − 45`。
- `sample` = 受信側の時計同期サンプル。`OrcReceiverModule.cpp:20` の `sample = timestampMs − millis()`。BEAT 行からは `sample = (playAtMasterMs − 45) − deviceMs` で復元できる。
- `D` = 見かけの配送遅延（スタンプ→受信処理まで。真のオフセット θ に対し `sample = θ − D`）。
- `margin` = `playAtMasterMs − localMasterMs`（正 = 予定時刻より手前）。`margin = 45 + (sample − offset)`。

## 3. 決定的観測: 204.8ms 周期のバースト配送

### 3.1 受信→発火の間に時計が 1ms も動かない

M45F（発火時）と M45R（受信時）の offsetMs の差を同一拍で突合すると、**5 ノード × 1020 拍の全件で Δoffset = 0**だった。受信→発火の待ち時間は平均 26ms / p95 77ms / 最大 94ms あり、CTRL は 20Hz（`node_01/include/ProjectConfig.h:40` の `ctrlIntervalMs=50`）で流れているから、パケットが連続的に届いているなら大半の拍でこの間に CTRL 到着 → offset 更新が起きるはずである。全件ゼロという結果は、**パケットが「かたまり」でしか届かず、かたまりの間隔が発火待ち（最大 94ms）より長い**ことを意味する。

### 3.2 到着時刻は 204.9ms の格子に乗っている

連続拍の受信 deviceMs の差分を「T の整数倍への当てはまり」でグリッドサーチ（T = 60〜220ms、0.1ms 刻み、クロックスキュー一次補正後）すると、**全 5 ノードで最良 T = 204.9ms**、正規化残差 0.0004〜0.0014（到着が一様なら 0.083）という極端な集中を示した。同じ解析を**デバイス時計と独立な PC タイムスタンプ**（serial_logger の受信時刻）で行っても **T = 205.0ms** で一致。さらに **7/9 の別ビルド（SERIAL_DEBUG=1、MOP_TEST なし）のログ** `test_20260709_234302.log` の EVT BEAT 到着でも全 5 ポートで T = 205.0〜205.3ms を検出した。ビルド・日付・時計系をまたいで再現する構造的な実挙動である。

204.8ms = 802.11 のビーコン間隔 100TU（= 102.4ms、`WiFiAP.cpp` の arduino-esp32 実装が `beacon_interval=100` を設定）のちょうど 2 倍。SoftAP は省電力モードの STA が 1 台でもいると、マルチキャスト/ブロードキャストフレームを DTIM ビーコンまでバッファしてから一括送出する。UNO R4 WiFi の無線部は ESP32-S3 モジュール（省電力既定オン、WiFiS3 ライブラリに省電力制御 API はない — `WiFiS3/src/` 全域に power 関連 API なしを確認）であり、実測周期が 2 ビーコンなのは DTIM 周期または STA のリッスン間隔が 2 相当で動いているためとみられる（どちらかの特定は実機追加計測が必要）。

### 3.3 BEAT の見かけ遅延は一様 [0, 205ms]

拍はヒトの指揮で打たれるため 204.8ms 格子に対して位相がほぼランダムになる。このとき BEAT の配送遅延は「次のフラッシュまでの待ち」= 一様 [0, 204.8ms] になるはずで、実測（受信 deviceMs − playAt をスキュー補正し最小遅延線からの相対値にしたもの）は

| ノード | p50 | 平均 | p95 | 最大 |
|---|---|---|---|---|
| node_02 | 95 | 97 | 184 | 198 |
| node_03 | 99 | 103 | 192 | 209 |
| node_04 | 95 | 97 | 185 | 201 |
| node_05 | 95 | 97 | 184 | 199 |
| node_06 | 97 | 101 | 189 | 202 |

と、一様 [0, 204.8]（p50=102 / 平均=102 / p95=195 / 最大=205）に全ノードで一致する。PC 時計による同じ推定（p50 96〜100 / p95 188〜194 / 最大 202〜212）とも一致。

## 4. 系統シフト −42.8ms のメカニズム（仮説 C の精密化）

### 4.1 純粋な「EMA が平均遅延を吸収」ではシフトは出ない

オフセット EMA が BEAT サンプルだけで作られるなら、`offset ≈ θ − mean(D_BEAT)`、margin = 45 − D_BEAT + mean(D_BEAT) となり、**平均 margin は +45ms のまま**（遅延の平均成分は相殺し、ばらつきだけ残る）。−42.8ms の系統シフトは説明できない。仮説 C は元の形では棄却である。

### 4.2 実際は CTRL が offset を「新鮮な位置」に引き上げている

時計同期サンプルは BEAT だけでなく **CTRL（20Hz、α=0.20）** も使う（`OrcReceiverModule.cpp:62-66`）。ここに 2 つの非対称が重なる。

1. **鮮度の非対称**: CTRL は 50ms ごとにスタンプされるので、バースト内の最新 CTRL はスタンプ後 0〜50ms（平均 ≈25ms）しか経っていない。しかも `pollReceive` は同一サイクル内の CTRL を最後の 1 個で上書きするため（`OrcNetModule.cpp:142-144`）、**採用されるのは常にバースト内で最も新鮮な CTRL**。一方 BEAT のスタンプは拍時刻なので、バーストまでの待ち = 平均 ≈102ms 古い。
2. **頻度の非対称**: 平均拍間隔 504ms の間にバーストは ≈2.5 回来るので、EMA への入力は「新鮮な CTRL ×2〜3 : 古い BEAT ×1」。

この混合 EMA は「実効エイジ a ≈ 42ms」の位置（= 真のオフセットから見かけ最小遅延 + 42ms だけ下）に落ち着く。実測でも、M45R の offsetMs から当該拍の寄与を外した `offset_before = (offset − 0.2·sample)/0.8` と最小遅延線の差は**全ノード 43〜48ms** だった。

### 4.3 定量的な閉合

- **offset − sample の平均**: margin = 45 + (sample − offset) なので、シフト −42.8ms は「offset が BEAT サンプルより平均 +42.8ms 上に座っている」ことと同値。実測: node_02〜06 で +41.8〜+43.5ms（全ノード一致）。
- **BEAT のみ EMA シミュレーション**: ログの BEAT サンプルだけで α=0.20 の EMA を再現すると、実ログ offset はそれより恒常的に **+41〜+46ms 上**にある。差分 = CTRL の寄与そのもの。
- **定常状態モデル**: T=204.8 / CTRL 50ms（採用エイジ平均 25ms）/ 拍間隔 504ms / α=0.20 を入れて定常解を解くと、実効 EMA エイジ a = 41.6ms、予測 margin 平均 = −3.6ms（**予測シフト −48.6ms** vs 実測 −42.8ms）、margin 範囲予測 −86〜+78（実測 −98〜+90）。±6ms 程度で閉合する。
- 残る注意: M45R の offsetMs は**当該拍のサンプルを α=0.20 で適用した後**の値なので（`OrcReceiverModule.cpp:95-105` → `:116-133` の順）、表示上のシフトは真の値の 0.8 倍に圧縮されている（補正前 ≈ −53.5ms）。

**「シフトの大きさが lookahead 45ms とほぼ一致する」のは偶然**である。シフトは `−0.8 × (BEAT 平均エイジ 102ms − 実効 EMA エイジ 42ms) ≈ −48ms` から来ており、lookahead の値とは独立に、バースト周期 204.8ms と CTRL 周期 50ms の組み合わせで決まっている。

## 5. 重要な含意: 表示 lateMs は真の遅刻を過小評価している

localMasterMs = deviceMs + offsetMs の offset が「真のオフセットより 40〜55ms 下」に座っている、ということは、**推定マスタ時計そのものが真のマスタ時計より 40〜55ms（+最小配送遅延 d_min）遅れて進んでいる**ということである。lateMs はこの遅れた時計で測るため、真の遅刻より小さく出る。

最小遅延線（受信 x = deviceMs − playAt の下側 2 パーセンタイル包絡、スキュー補正済み）を基準に真のマスタ時計基準の遅刻を推定し直すと（d_min は含まれないので下限値）:

| 指標 | 表示値（M45 基準） | 真値の推定（最小遅延線基準） |
|---|---|---|
| 受信時に既に遅刻している拍の割合 | 45.4% | **≈79%** |
| 発火遅刻 平均 | 23.7ms | **≈+80ms** |
| 発火遅刻 p50 | 8ms | **≈+68ms** |
| 発火遅刻 p95 | 78ms | **≈+145ms** |
| 発火遅刻 最大 | 101ms | **≈+168ms** |
| 発火が遅刻する拍の割合 | 95.9% | **100%** |

（真値推定はノード別でも平均 +77〜+84 / p95 +141〜+150ms で横並び。）指揮ジェスチャ→発音の絶対遅延は `45ms + 真の発火遅刻` なので**平均 ≈125ms / p95 ≈190ms** に相当する。7/9 調査の「最大 ~160ms」推定とも整合する。

なお発火時刻には配送とは別に**発火ループ粒度**も乗る: `fireLate − max(0, recvLate)` は平均 3.5〜4.5ms / 最大 8〜14ms（設定 `loopIntervalMs=2` に対し実効ループ周期はその 2〜7 倍。UNO R4 の WiFi モジュール SPI ポーリングが 1 周期を延ばしているとみられる）。これは p50 の 8ms の主成分だが、p95 の 78ms に対しては誤差程度。

## 6. 「全員一様に遅れるのに楽器間は揃う」構造の説明

1. **配送の共通モード**: 5 台は同一のバースト（同一 DTIM フラッシュ、ほぼ同一の無線フレーム群）で BEAT を受け取る。バースト内の到着差は数 ms。
2. **時計バイアスの共通モード**: §4 の EMA バイアスは全ノードが同じ CTRL/BEAT 混合・同じ α で持つため、量までほぼ同じ（実測 43〜48ms）。localMasterMs が全員同じだけずれる。
3. よって MOP4（同一拍の localMasterMs レンジ）にはどちらも現れず、残るのは (a) 発火ループ粒度 ±4ms、(b) EMA の短期ノイズの個体差（margin の拍内ノード間レンジ実測 平均 12.4 / p95 23ms）、(c) 電波・受信処理の個体差。これが MOP4 の平均 9.8 / p95 22ms の正体である。

つまり **MOP4 の良好さと MOP5 の大幅 FAIL は矛盾ではなく、同じ機構（共通モードのバースト遅延）の表裏**である。発表で聴感上問題がなかったのも、楽器間の同時性が保たれ、指揮→音の絶対遅延（変動 ±80ms 程度）だけが犠牲になっていたためと説明できる。

## 7. 副産物: UNO R4 の millis() スキュー

x = deviceMs − playAt の一次回帰（および offsetMs のドリフトとの鏡像一致）から、指揮者（ESP32-S3）比のクロックレート差は node_02: −2926ppm / node_03: −401ppm / node_04: −3232ppm / node_05: −2317ppm / node_06: −1174ppm。水晶振動子では説明できない大きさだが、**RA4M1（UNO R4）は内蔵高速オンチップオシレータ（HOCO ±1% 級）駆動**なので仕様相当。EMA が追従するため本件シフトへの寄与は「EMA 遅れ × スキュー ≈ 0.6s × 0.003 ≈ 2ms 未満」で無視できる。ただし将来「時計同期の更新を止めて長時間フリーラン」させる設計にすると 1 秒あたり最大 3ms ずれる点は留意。

## 8. 対処案（ファーム修正案 — 実装はしていない）

| 案 | 内容 | 効果 | 代償・注意 |
|---|---|---|---|
| 1. lookahead 増（最小変更） | `node_01/include/ProjectConfig.h:42` の `beatLookaheadMs` 45 → **220ms 程度**（バースト周期 204.8 + マージン） | ほぼ全拍で予約が成立し、発火遅刻がほぼゼロに。楽器間同期もさらに安定 | 指揮→発音の固定遅延が ~220ms に増える（120BPM で約半拍）。音楽的に許容かは実機評価が必要 |

推奨: **案 1 を即効薬**（1 定数、発表運用にそのまま効く）、**案 2 を根治**として実機検証。案 4 は計測の正直さのために価値があるが優先度は下。ユニキャスト×5 への変更は、省電力 STA へのユニキャストも同様にビーコン待ちバッファされるため単独では解決の保証がなく、推奨しない。

## 9. MOP5 最終レポートへの記載案

- **数値**: 「発火 lateMs p95 = 78ms（判定 FAIL）」を主値として記載しつつ、「この値は推定マスタ時計基準であり、推定時計自体が平均 40〜55ms 遅れているため、真のマスタ時計基準の遅刻は p95 ≈ 145ms 程度（最小遅延線基準の下限推定）」と併記する。
- **原因**: 「SoftAP のマルチキャストが STA 省電力に伴う DTIM バッファリングで 204.8ms（2 ビーコン）周期のバースト配送となり、45ms の発音予約が構造的に不足していた」と記載（本レポートを出典に）。
- **MOP4 との関係**: 「遅延が全ノード共通モードのため楽器間同期（MOP4）は保たれる。MOP4 良好 / MOP5 FAIL は同一機構の表裏」と明記。
- **7/9 調査の訂正**: `MOP45_latency_investigation_20260710.md` §4.3 の「約 45〜55ms の系統シフトが未解明」は本調査で解明済みとして参照を張る。
- **将来課題**: 絶対片道遅延の実測（GPIO + ロジアナ / ping-ACK 同期)は引き続き将来課題。ただし遅延の支配項（バースト待ち 0〜205ms）は本調査で特定済みなので、優先度は下げてよい。
