## 結論サマリ

1. **受信側の劇的改善はすべて対処案1（lookahead 220ms）の効果**。対処案2（ビーコン 50TU）は
   `-> OK` 応答にもかかわらず**バースト配送周期を全く縮めていない**（対策前後とも実測 204.8ms、§2）。
   配送遅延の分布そのものは前後で不変（p50 ≈100 / p95 ≈193 / 最大 ≈215ms）。
2. **対処案4（min フィルタ）は狙いどおり機能**。推定時計は最小遅延線の近傍（BEAT 下側包絡 +13〜18ms、
   バースト内の新鮮な CTRL 相当）に座り、表示 lateMs が「真の遅刻の下限としてほぼ正直」になった
   （対策前は真値を 54〜59ms 過小表示）。階段状ジャンプは平常時 ±10ms 程度で無害（§3.2）。
3. **新現象①（発火 lateMs p95 47.3ms の高止まり）は見かけではなく実挙動**。原因は min フィルタでも
   受信遅延でもなく、**楽器ループがバースト到着から約 120〜165ms の位相区間でほぼ実行されない
   「周期ストール」**。この区間に発音予定時刻が落ちた拍（全体の約 1/3）が区間明け（位相 ≈160〜180ms）へ
   押し出されて 20〜79ms 遅刻する（§3）。
4. **新現象②（MOP4 の尾の悪化 p95 22→46ms）も同じストールが原因**。悪化 12 拍の playAt 位相は
   全件ストール窓内（117〜154ms）にあり、窓の端・当たり外れがノードごとに数十 ms 違うため拍内レンジが
   開く。測定値と包絡写像による実挙動推定が一致しており（両方 ≈50ms）、計測アーチファクトではない（§4）。
5. 推奨は **ビーコン 50TU 設定の撤去（効果ゼロ ＋ ストール新規誘発の疑い）→ 再計測**。ストールが
   50TU 起因なら発火 lateMs p95 は ≈9ms（ストール非暴露群の実測値）まで下がり MOP5 閾値 30ms を
   PASS、MOP4 p95 も ≈11〜20ms へ戻る見込み。lookahead はバースト周期が縮まなかった以上
   **220ms から下げられない**（70ms 化は不可と確定、§5）。

## 2. 問1: バースト周期はどうなったか — 変わっていない

### 2.1 ビーコン 50TU は「OK」でもフラッシュ周期 204.8ms のまま

`esp_wifi_set_config` は ESP_OK を返したが（起動ログ）、BEAT 到着周期のグリッドサーチは対策前後とも
**204.8ms** を示し、配送遅延分布（下側 2% 包絡からの相対値）も p50 ≈100 / p95 ≈193 / p99 ≈201〜207 /
最大 203〜215ms と**前後で統計的に同一**。つまりマルチキャストのフラッシュ周期は 1ms も縮んでいない。

解釈は 2 通りあり、ログからは区別できない: (i) ESP_OK でも実際のビーコン間隔はヘッダ記載レンジ
（100〜60000TU）で内部クランプされ 100TU のまま、(ii) ビーコンは 51.2ms 化されたが、フラッシュ周期を
決めているのは STA（UNO R4 の WiFi モジュール）側のリッスン間隔で、それが 204.8ms 相当のまま。
いずれにせよ**対処案2 は目的（バースト周期短縮）に対して無効**である。

### 2.2 受信マージン +102.7ms と系統シフト −117.3ms の整合

対策後の受信マージンは `margin ≈ 220 − D − (off − θ_env)` で説明できる。実測: 平均 D = 102.2ms、
min フィルタの offset は BEAT 下側包絡より平均 +12〜+18ms 上（§3.2）→ 220 − 102 − 15 ≈ +103ms（実測 +102.7）。
「系統シフト −117.3ms」は異常ではなく、**平均バースト待ち（≈102ms）＋ 時計が正直になった分（≈15ms）が
そのまま見えている**だけである。対策前の −42.8ms は「EMA 時計の遅れがバースト待ちの大半を隠した値」であり、
両者は同じ土俵の数字ではない。

## 3. 問2: 新現象①の機構 — 発火ループの「バースト位相同期ストール」（実挙動）

### 3.1 min フィルタ犯人説はデータで棄却

- 同一拍の Δoffset（M45F − M45R）は **915 拍中 850 拍で 0**、正 56 件（平均 +5.3 / 最大 +17ms）、
  負 9 件（最小 −14ms）。発火 late>10ms の 272 件のうち Δoffset>0 は 33 件だけで、
  「offset の階段ジャンプが予定時刻を過去化して即発火」で説明できる遅刻はごく一部。
- 真の発火遅刻（クロックスキュー補正済みの受信下側 2% 包絡でデバイス時刻→マスタ時刻へ写像、
  推定時計から独立）は平均 −2.9 / p50 −7.8 / **p95 +31.4 / 最大 +73.7ms**。包絡推定は最小配送遅延ぶん
  下振れする下限値なので、**p95 で数十 ms の遅刻は実在する**。なお表示 lateMs は `off ≤ 真のオフセット`
  より真の遅刻の下限であり、p95 47.3ms は「少なくともこれだけ遅い」と読める（見かけの悪化ではない）。

### 3.2 min フィルタ自体は設計どおり動いている

連続する受信観測間で offset は 70〜77% の区間で不変、上昇は平均 +4.5〜+7.6ms（最大 +32）、
下降（窓満了の引き直し）は平均 −2.5〜−7.7ms（最小 −14）。offset は BEAT 下側包絡の **+13〜+18ms 上**
（= バースト内の新鮮な CTRL サンプル相当の位置）に安定して座る。対策前の EMA は同じ量が
**−54〜−59ms 下**だった。計測の正直化という対処案4 の目的は達成されている。

### 3.3 決定的観測: 発火位相ヒストグラムの「空白帯」

全発火 915 件をバースト格子（受信位相中心）からの経過で 20ms ビンに集計すると:

```
位相[ms]  0-19 20-39 40-59 60-79 80-99 100-119 120-139 140-159 160-179 180-199 200-204
件数        82    82    83    92   104      55      13       1     246     138      19
（一様なら各ビン ≈83。200-204 ビンは幅 4.8ms なので少なくて正常）
```

**位相 120〜159ms で発火がほぼ消滅し（14 件）、160〜179ms に一様期待の 3 倍（246 件）が積み上がる**。
発音予定時刻（playAt）の位相分布はほぼ一様なので、これは「予定時刻が来ても発火せず、まとめて
160〜180ms で発火した」ことを意味する。ノード別の発火空白帯は node_02 (130,166) / node_03 (119,161) /
node_04 (118,160) / node_05 (125,167) / node_06 (130,166) — **全 5 ノードが同じ構造**を持つ。

発火条件（`applyPattern.cpp:164-169`）は `targetLocalMs = playAtMasterMs − offsetMs` と `millis()` の
比較で、offset が動かない限り（§3.1 で確認済み）発火を遅らせられるのは**ループ自体が回らないこと**だけ。
よって機構は: **バースト到着から約 120〜165ms の区間（約 40ms 幅）で楽器のメインループが実行されず、
そこに予定時刻が落ちた拍は区間明けへ押し出される**。

分割定量（playAt 位相がストール窓 105〜170ms 内/外）:

| 群 | n | 発火 lateMs 平均 | p95 | 最大 |
|---|---|---|---|---|
| 窓内 | 296（32%） | 25.5 | 53.0 | 79 |
| 窓外 | 619 | 6.0 | 18.0 | 31 |
| 受信→発火待ち <150ms（ストール非暴露） | 503 | 4.4 | **9.0** | 18 |

非暴露群の p95 9ms が「ストールが無ければ達成できる発火精度」（ループ粒度＋α）のベースラインである。

### 3.4 ストールの原因候補と、なぜ前回見えなかったか

- ストールは**バースト（DTIM フラッシュ）位相に固定**され、窓明けは次バーストの約 40ms 前。第一候補は
  **WiFiS3 モデム（UNO R4 内蔵 ESP32-S3）の省電力ウェイク処理**: ホストはループ毎に `udp.parsePacket()`
  で SPI 通信するため、モデムがビーコン受信・省電力遷移で応答を止めるとホストループごとブロックする。
  ビーコンが 50TU 化されていれば非 DTIM ビーコン位相（51.2×3=153.6ms）が窓（118〜167ms）に入るのは
  示唆的だが、モデム内部の確定はログからはできない（追加計測案は §5(b)）。
- **前回（lookahead 45ms）ログでこのストールが見えなかったのは非観測なだけ**。発火は到着後最大 94ms 以内に
  限られ、位相 100〜180ms を発火がサンプルしない（前回の発火位相ヒストグラムでも同区間はほぼ空）。
  よって「ストールが 50TU 化で新規発生した」のか「以前からあった」のかは現データでは判定不能。

## 4. 問3: 新現象②の機構 — MOP4 の尾は同じストールの拍内非一貫性

- MOP4 レンジ >20ms の悪化 12 拍の playAt 位相は **117〜154ms で全件ストール窓内**。
  窓内/窓外で分割すると: 窓内 n=59 平均 16.9 / p95 52.8 / 最大 79ms、窓外 n=124 平均 6.8 / p95 11.0 /
  最大 19ms。**窓外だけ見れば MOP4 は対策前より良い**。
- 悪化拍では測定レンジ（localMasterMs）と包絡写像による実挙動レンジが一致（例: beat=161 測定 79 /
  実挙動 75.4ms）。**min フィルタのノード間非一貫性による「見かけ」ではなく、実際に発火時刻が割れている**。
  当初仮説（窓リセット位相差で推定時計がノード間でズレる）は、offset 階段が ±10ms 程度しかない（§3.2）
  ため棄却。
- 割れ方はストールの「当たり外れ」: 同じ予定時刻でも、ストール窓の端（ノード間で ±10ms 程度違う）や
  その周期のストール発生自体の差で、**一部ノードだけが +30〜50ms 押し出される**。
  例: beat=9 は node_02/03/04 が真遅刻 +29〜34ms・node_05/06 は定刻、beat=161 は node_04 だけ +74ms。
  外れノード頻度は node_04: 6 / node_05: 3 / node_06: 2 / node_03: 1 回で、node_04 の空白帯がやや広い
  （発火 late 最大 79ms も node_04）。
- 対策前の MOP4 悪化拍（15 拍）は全ノードが +70〜170ms 共通遅刻した中の残差（実挙動レンジ ≈10〜27ms）で、
  機構が異なる。p50（8→7ms）とノード別平均偏差（±1.7ms 以内）が不変なのは、ストールが尾（約 1/3 の
  拍のうち一部）にだけ効くためで、共通モード同期の骨格は保たれている。

## 5. 問4: 次の一手

### (a) 設定調整のみで見込めるもの

| 項目 | 判断 | 根拠・期待効果 |
|---|---|---|
| lookahead 220ms の短縮 | **不可（220 維持）** | バースト周期 204.8ms が縮まなかったため、必要 lookahead = D 最大 ≈215ms ＋処理余裕。実測でも受信遅刻 17 件の D は 193〜215ms で 220 は既にギリギリ。70ms 化の前提（周期 ~51ms）は崩れた |
| lookahead 230〜240ms へ微増 | 非推奨 | 受信遅刻 1.9%（最大 14ms、即発火で吸収）を消せるが、全拍の固定遅延が +10〜20ms 増える。益が代償に見合わない |
| min フィルタ窓長 2000ms | **変更不要** | 階段 ±10ms 程度で両現象への寄与なし。窓を縮めても発火 p95 は改善しない（原因はストール） |

### (b) ファーム修正が要るもの（提案のみ・未実装）

1. **【第一推奨】ビーコン 50TU 設定ブロックの撤去**（`6642646` で `OrcNetModule.cpp` に足した 28 行）。
   周期短縮効果はゼロと実証された一方、ストールを 50TU が誘発している疑い（§3.4）が残る。撤去して
   同条件（lookahead 220 + min フィルタ）で再計測し、発火位相の空白帯が消えるかを見る。
   期待効果: ストールが 50TU 起因なら発火 lateMs p95 47.3 → **≈9ms**（§3.3 非暴露群）で MOP5 閾値
   30ms を PASS、MOP4 p95 46 → **≈11〜20ms**。効かなくても受信側（lookahead 220 の効果）は不変で損はない。
2. ストールが 100TU でも残る場合の対症療法: **発火判定を net ポーリングより先に実行する**
   （ループ順序の入替、または updateInput 冒頭に発火チェック追加）。ストールが「短いブロックの連なり」
   なら遅刻をブロック 1 回分（数〜10ms）に抑えられるが、「単一の長いブロック」なら効かない。
   ブロック粒度はログから判定できないため、ループハートビート計測（MOP8 の要領で loop 間隔 >10ms を
   1 行ログ化）で先に粒度と正確な窓位置を確定させるのが安全。
3. 根治（工数大・将来課題）: STA 省電力オフ（uno-r4-wifi-usb-bridge のモデムファーム改造）。
   バースト配送とストールの両方が消える見込みだが、前回調査どおり WiFiS3 に API がなく手を出しにくい。

### (c) このまま受け入れる場合に最終レポートへ書くべき内容

- **受信側（発音予約の成立）は解決**: 遅刻率 45.4→1.9%、受信 lateMs 最大 14ms。lookahead 220ms は
  バースト周期 204.8ms の実測に基づく設計値として妥当。
- **発火 p95 47.3ms は FAIL のままだが、値の意味が変わった**: 時計同期の min フィルタ化により表示値は
  「真の遅刻の下限」としてほぼ正直（対策前は同じ表示でも真値を 54〜59ms 過小評価）。対策前の
  「表示 p95 78ms＝真値 ≈145ms」に対し、対策後は「表示 p95 47ms ≈ 真値」で、**真の発火遅刻は 1/3 に改善**している。
- **指揮→発音の絶対遅延は固定化と引き換えに増加**: 対策前 平均 ≈125 / p95 ≈190ms（変動大）→
  対策後 220 ＋ lateMs で平均 ≈232 / p95 ≈267ms（120BPM の約半拍・ほぼ固定）。楽器間が揃った上での
  一定遅延なので聴感上は「指揮に半拍遅れてついてくる」形になる。許容可否は演奏評価で判断。
- **MOP4 は p50 7ms・ノード別平均偏差 ±1.7ms 以内を維持**。p95 46ms の尾はストール起因で、
  対処 (b)-1 の結果次第で 20ms 前後へ戻る見込み。

### 推奨の結論

**(b)-1（50TU 撤去）を実施して再計測**が最優先。1 コミットの revert 相当で済み、結果がどちらでも
情報が増える（消えれば原因確定＋全指標改善、残ればストールは既存問題と確定し (b)-2 のハートビート
計測へ進む）。lookahead と min フィルタは現状維持。
