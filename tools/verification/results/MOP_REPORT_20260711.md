## 全体ストーリー — 届くのはバラバラ、鳴るのは同時

本システムの同期設計の核心は「**パケットの到着時刻に頼らず、発音予約（playAtMasterMs）と
時計同期で発音時刻を揃える**」ことにある。7/10〜7/11 の計測でこの設計が数値で裏付けられた。

1. **届くのはバラバラ**: SoftAP の UDP マルチキャストは、省電力 STA 向けの DTIM バッファリング
   （802.11 の仕様由来）により **204.8ms 周期のバースト配送**になる。BEAT の生の配送遅延は
   一様分布 [0, 205ms]（平均 ≈100ms、p95 ≈195ms、最大 ≈215ms）で、3 回の計測すべてで再現した。
   「生の通信遅延 ≤30ms」は物理的に成立しない環境である。
2. **プログラムで吸収**: 指揮者は拍を 220ms 先の未来時刻として予約送信し（lookahead）、楽器は
   min フィルタの時計同期（窓 2000ms 内の最小配送遅延サンプル追従）で推定マスタ時計を持つ。
   バーストで遅れて届いても、予約時刻までに届きさえすれば発音は狂わない。
3. **鳴るのは同時**: 対策後、受信が予約に間に合わなかった拍は **45.4% → 1.9〜3.1%** に激減し、
   真のマスタ時計基準の発火遅刻（最小遅延線基準の推定）は **p95 145ms → 28〜31ms** に改善。
   楽器間の発音時刻差は中央値 7ms（目標 20ms の 1/3、聴感の同時性閾値 30〜50ms を大幅に下回る）。
   7/8 成果発表会の実演でも、チームメンバー・聴衆からタイミングのズレの指摘はなかった。

原因特定 → 対策 → 実測改善の流れ（詳細は各調査レポート）:

### MOP4: 楽器間同期誤差 — FAIL（実質達成・受け入れ）（7/11 検証）

計測方式: 同一拍の発火時推定マスタ時刻（M45F の localMasterMs）のノード間レンジ。
USB シリアル到着ジッタに依存しないデバイス側計測（7/10 再構築）。

最終構成（7/11、`results/mop4/20260711_154006_summary.txt`、173 拍 × 5 ノード）:

- 中央値(p50): **7ms**（目標 20ms の約 1/3）
- 平均: 10.8ms / SD: 12.6ms
- p95: 48.6ms / 最大: 65ms
- 20ms 超過: 16/173 拍（9.2%）
- ノード別平均偏差: −1.7〜+3.1ms（系統的な先行/遅延ノードなし）

3 回の実測比較（すべて生ログから再集計・summary と一致確認済み）:

| 指標 | 7/10 対策前 | 7/10 対策後 | 7/11 最終構成 |
|---|---|---|---|
| 集計拍数 | 204 | 183 | 173 |
| 平均 / p50 | 9.8 / 8ms | 10.1 / 7ms | 10.8 / 7ms |
| p95 / 最大 | 22 / 32ms | 46 / 79ms | 48.6 / 65ms |
| SD | 6.0ms | 12.1ms | 12.6ms |
| 20ms 超過率 | 7.4% | 6.6% | 9.2% |

**中央値・平均・超過率は 3 回とも同水準**で、対策（発音を予約時刻まで待つ方式が実際に機能する
ようになった）後も骨格は不変。一方 p95/最大の「尾」は対策後に 22→46〜49ms へ伸びた。

**尾の正体はデバイス内部の周期ストール**（`MOP5_countermeasure_eval_20260710.md` §3-§4）。
楽器のメインループがバースト到着から位相 ~120〜165ms の区間（204.8ms 周期に同期した約 40ms 幅）で
実行されず、そこに発音予定時刻が落ちた拍だけ、ストールに当たったノードが 30〜60ms 遅れて発火する。
対策前は発音が「到着即発火」だったためこの区間を踏まず、尾が見えていなかった（悪化ではなく露出）。
20ms 超過拍の測定レンジは包絡写像による実挙動推定と一致しており（例: 7/11 beat=149 測定 65 /
実挙動 68.7ms）、計測アーチファクトではない。ビーコン設定（50TU）を撤去しても尾は不変
（p95 46→48.6ms）で、ストールは WiFiS3 モジュール由来の既存挙動と確定。原因調査用の
ハートビート計測（M45S、`mop5_loop_stall.py`）を将来の調査手段として整備済み。

受け入れ判断の根拠:

1. **統計的には実質達成**: p50=7ms は目標の 1/3。90.8% の拍が 20ms 以内。
2. **聴感品質**: 人間の同時性知覚閾値は約 30〜50ms（河瀬 2014 等）。p50=7ms は完全に「同時」、
   尾（p95 48.6ms）も閾値帯の上端近傍かつ拍単位の単発事象（連続しない）で、「ズレ」の印象は
   生じにくい。7/8 成果発表会の実演でもズレの指摘はなかった。
3. **改善の限界**: ストールは UNO R4 の WiFi モジュール（ESP32-S3 ブリッジ）内部の挙動で、
   ホスト側ソフトの改善余地は限定的。根治（STA 省電力オフ）はモジュールファーム改造が必要。

**統計量設計の反省**: MOP4 の判定を「最大値 ≤ 20ms」としたため、中央値が目標の 1/3 でも
外れ値 1 拍で FAIL になる。ジッタを持つ通信系の品質指標は、最大値ではなく分位点
（例: p95 ≤ 閾値、または超過率 ≤ N%）で設計すべきだった。次回は「聴感閾値を p95 で下回る」の
ような、外れ値に頑健で知覚品質に対応した判定式を最初に置く。

### MOP5: 指揮→楽器の通信遅延 — 指標再定義（受け入れ）（7/11 検証）

**総括: 初期段階の指標設定ミス（スコープミス）だった。**

計画書の MOP5 は「指揮から楽器への通信遅延 ≤ 30ms」（生の片道遅延）と定義していた。しかし
本システムは **BEAT を lookahead ぶん未来の予約時刻付きで送り、楽器側が時計同期で予約時刻まで
待って発音する**設計であり、生の配送遅延は（予約に間に合う限り）演奏品質に影響しない。
指標がアーキテクチャと噛み合っていなかった。さらに実測の結果:

- **物理的に達成不能**: SoftAP のマルチキャストは DTIM バッファリング（802.11 仕様由来）で
  204.8ms 周期のバースト配送になり、生の配送遅延は一様 [0, 205ms]（平均 ≈100ms、最大 ≈215ms）。
  30ms 以下は原理的に成立しない。ビーコン間隔の短縮（100→50TU）も試したが、設定は受理されても
  バースト周期は 1ms も変わらず（3 回の実測すべてで周期 204.2〜205.1ms）、撤去した。
- **計測も不能**: 片方向の時計同期では絶対片道遅延は原理的に測れない（同期自体が遅延を吸収する。
  `MOP45_latency_investigation_20260710.md` §4.1）。

そこで 7/10 に、品質に実際に効く指標 **「発音予約に間に合うか」= 受信時点の遅刻 lateMs** へ
再定義した（`mop5_comm_delay.py`。受信 lateMs = max(0, 受信時推定マスタ時刻 − 予約時刻)）。

対策（lookahead 45→220ms + 時計同期の EMA→min フィルタ化）前後の実測:

| 指標 | 7/10 対策前 (LA=45ms) | 7/10 対策後 (LA=220ms) | 7/11 最終構成 |
|---|---|---|---|
| **受信遅刻率（予約破綻率）** | **45.4%** (465/1024) | **1.9%** (17/915) | **3.1%** (27/865) |
| 受信 lateMs 最大 | 98ms | 14ms | 33ms |
| 受信マージン平均 | +2.2ms | +102.7ms | +100.8ms |
| 発火 lateMs p50 / p95 / 最大 | 8 / 78 / 101ms | 6 / 47.3 / 79ms | 6 / 47.0 / 67ms |
| 真の発火遅刻（包絡基準・下限） p95 | ≈145ms | ≈31ms | ≈29ms |
| 真の受信遅刻率（同上基準） | ≈79% | 0% | 0% |
| バースト到着周期（グリッドサーチ） | 204.4〜205.0ms | 204.2〜204.9ms | 204.0〜205.1ms |

- **発音予約の成立（受信）は達成**: 遅刻率 1.9〜3.1%、遅刻しても最大 14〜33ms で即発火により
  回復する。lookahead 220ms はバースト周期 204.8ms + 余裕の実測設計値。
- 対策前の「表示上マシに見える」値（発火 p95 78ms）は、EMA 時計自体が 54〜59ms 遅れていて真の
  遅刻を過小表示していたもの。min フィルタ化で推定時計が最小遅延線に張り付き、**7/10 以降の
  表示値は真の遅刻の下限としてほぼ正直**になった（計測の信頼性も対策の成果）。
- 残る発火 lateMs p95 ≈47ms は配送でも時計でもなく、MOP4 の尾と同一の周期ストール由来
  （lookahead をこれ以上増やしても消えない）。
- 代償: 指揮 → 発音の絶対遅延は約 220ms + 発火遅刻（平均 ≈232ms、120BPM の約半拍）の固定遅延に
  なった。対策前は平均 ≈125ms / p95 ≈190ms の「小さいが暴れる」遅延であり、**ばらつきを固定遅延に
  置き換えて楽器間の同時性を確保する**トレードオフを選んだ。楽器間が揃っている限り、一定の
  追従遅れは指揮行為として自然に吸収される（実演で違和感の報告なし）。

**教訓**: 指標は「アーキテクチャが何を保証する設計なのか」に対応させて定義すべきで、
下位層の物理量（生の通信遅延）をそのまま目標化すると、達成不能かつ計測不能な指標を追いかける
ことになる。MOP5 の本来の意図（発音がもたつかないこと）は「予約成立率」と「発火遅刻」で測るのが
正しく、これは初期の V&V 計画段階で気づけたスコープミスである。

## 考察 — MOP4/MOP5 の一連の調査が示したこと

1. **設計は正しく機能している**: 「発音予約 + 時計同期」により、生の配送遅延が平均 100ms /
   最大 215ms でも楽器間の発音は中央値 7ms で揃う。「届くのはバラバラ、鳴るのは同時」を
   3 回の実測（延べ 560 拍 × 5 ノード）で定量的に確認した。
2. **指標は層に合わせて定義する**（MOP5 の教訓）: 予約発音アーキテクチャの品質は「予約成立率」で
   測る。生の片道遅延は本システムでは達成不能・計測不能・品質無関係の三重に不適切だった。
3. **判定統計量は外れ値に頑健にする**（MOP4 の教訓): 最大値判定は 1 拍のスパイクで FAIL になる。
   聴感対応の分位点判定（p95 等）を初期に設計すべきだった。
4. **計測の正直さ自体が成果物**: EMA 時計は真の遅刻を 54〜59ms 過小表示していた。min フィルタ化に
   より、以後の計測値は下限保証つきで解釈できる。将来の調査手段として、バースト位相同期ストールの
   粒度を特定する M45S ハートビート計測（`mop5_loop_stall.py`）も整備済み。
5. **残課題**: (a) 周期ストールの根本原因（WiFiS3 モジュールの省電力ウェイク疑い）の確定と対処は
   M45S 計測待ち。(b) 絶対片道遅延の実測（GPIO + ロジックアナライザ）は引き続き将来課題だが、
   遅延の支配項（バースト待ち 0〜205ms）は特定済みで優先度は低い。(c) MOP2 の録音検証。
