## MOP4/MOP5 考察 — 目標超過の受け入れ判断

### 超過の実態

**MOP4（楽器間同期誤差）**: 157 拍中 26 拍（17%）が目標値 20ms を超過した。ただし中央値は 13ms であり、大半の拍は目標内に収まっている。超過した拍も p95=28ms に留まり、最大 35ms のスパイクは稀な外れ値である。

**MOP5（楽器間到着ずれ）**: 全ペアの中央値は 5ms で、ほとんどの拍でノード間のずれは無視できる水準。最大 35ms は node_05 — node_06 ペアで発生した単発のスパイクである。

### 超過の原因

主因は WiFi（UDP マルチキャスト + SoftAP）固有のジッタである。

1. **SoftAP のビーコン間隔**: ESP32 の SoftAP は約 100ms 間隔でビーコンを発信する。ビーコン送出直後の短い期間にデータフレームが遅延するため、周期的な遅延スパイクが生じる。
2. **チャネル混雑**: 2.4GHz 帯の干渉（周囲の WiFi AP や Bluetooth）がバースト的な遅延を引き起こす。発表会場では多数の端末が同一帯域を使用しており、7/2 検証時（研究室）より干渉が大きかった可能性がある。
3. **マルチキャストの特性**: UDP マルチキャストは QoS 制御がなく、再送もない。各ノードへの到着タイミングは WiFi の内部バッファリングとスケジューリングに依存する。

ソフトウェア側ではすでに以下の最適化を実施済みであり、改善余地は限られている:
- EMA クロック同期（α=0.20 / 重複 α=0.05）で定常的なオフセットを吸収
- beatLookahead = 45ms で発音予約に余裕を確保
- ループ周期 2ms 化で発火判定ジッタを最小化
- BEAT パケットの連送冗長化でパケロスを吸収

### 聴感への影響

人間の聴覚における同時性知覚の閾値は約 30〜50ms とされる（河瀬 2014「聴覚フィードバックを用いた合奏訓練システムの評価」等）。

- **中央値 13ms**: 知覚閾値の半分以下であり、完全に「同時」と知覚される
- **p95 = 28ms**: 閾値の下端（30ms）をわずかに下回る。仮に知覚されても「微妙なゆらぎ」程度であり、不快感や「ズレ」の印象は生じない
- **最大 35ms**: 閾値内（30〜50ms の範囲内）。単発のスパイクであり、連続して発生しないため聴感上の影響は極めて限定的

実演（7/8 成果発表会）においても、チームメンバー・聴衆からタイミングのズレに関する指摘はなかった。

### 結論

MOP4/MOP5 ともに目標値を最大値で超過するが、以下の理由から**受け入れ**と判断する:

1. **統計的には良好**: p50=13ms（MOP4）/ 5ms（MOP5）であり、大半の拍は目標の半分以下
2. **聴感品質は確保**: 超過値（最大35ms）も聴覚の同時性知覚閾値（30〜50ms）の範囲内であり、実演で問題は観測されていない
3. **改善の天井**: WiFi ジッタの完全排除にはハードウェア変更（有線接続や専用 RF モジュール）が必要であり、Arduino WiFi の制約下ではソフトウェア最適化の限界に達している

## MOP2 が未検証である理由

MOP2（音階誤差 < 3.6 cent）は Processing の合成出力音を録音・周波数分析し、楽譜上の平均律理論音高と比較する検証が必要となる。マイク録音環境の整備と FFT による精密な周波数測定が必要で、今回の検証スコープ（ファーム側のシリアルログベース検証）の範囲外である。

なお、ファーム側が送出する noteNumber の正確性は MOP3 で全ノート一致を確認済み。Processing の加算合成は数式ベース（`f_n = n × f_0 × √(1 + B × n²)`）で理論周波数を直接生成するため、原理的に cent 単位の誤差は発生しない。音階の「誤差」が生じるとすればスピーカー・DAC の非線形性やエイリアシング等のハードウェア要因であり、ソフトウェア検証の対象外となる。
