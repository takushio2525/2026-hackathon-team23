# MOP3: 楽譜との相違 評価記録

## 目標値
誤ノート発音数 0（送出された noteNumber が楽譜と完全一致すること）

## 検証方法の妥当性

### 方法の概要
楽器ノード（node_02〜06）に `-DMOP_TEST=3` を設定してビルドし、演奏中に
Serial 出力される `M3,<partId>,<noteNumber>,<velocity>,<seq>` 行を Python
スクリプトで収集。パートごとに到着順序で score_data.cpp の期待楽譜と比較し、
noteNumber の不一致件数をカウントする。

### 評価

**妥当と判断できる点:**

- noteNumber の比較はパート別・到着順序ベースで行い、headRestBeats（輪唱の
  頭ずらし 0/8/16/24 拍）の影響を受けない。全金管ノードの楽譜が同一であるため正しい
- 周回（2周目以降）はモジュロ演算でラップアラウンドし、複数周回の連続演奏に対応
- ドラムノード（partId=6, node_06）は楽譜構造が金管と異なるため比較対象から除外
- 1周分に満たない演奏でも受信した分だけ比較するので、短い演奏でもテスト可能
- CSV に全ノートの期待値・実測値・一致判定を記録し、事後分析が可能

**レビューで修正した問題（3件）:**

1. **サブノート未捕捉**: フレーズ4（拍25-28）の8分音符ペアはメインスロットと
   サブスロットで2音ずつ発火するが、ファーム側の `#if MOP_TEST == 3` ブロックが
   メインスロットにしかなく、サブノート4音/周が検証漏れしていた。
   → ファームにサブスロットの M3 出力を追加、Python の期待楽譜を25→29音に修正

2. **SERIAL_DEBUG とのバイナリ混在**: MOP_TEST=3 + SERIAL_DEBUG=0 のとき、
   buildAndSend のバイナリ NotePacket と M3 テキストが同一 Serial に流れ、
   バイナリ中の `0x0A`（改行）でパースが破綻するリスクがあった。
   → MOP_TEST > 0 のとき SERIAL_DEBUG/buildAndSend を抑制する
   `#if MOP_TEST > 0` ガードを追加（MopTest.h の「MOP_TEST 有効時は
   SERIAL_DEBUG を抑制」の意図を実装レベルで強制）

3. **NOTE_SUB 未対応**: Python の RE_NOTE_DBG が `NOTE_ON` のみマッチし
   `NOTE_SUB` を無視していた。→ `NOTE_(?:ON|SUB)` に修正

**残存する制約（仕様として許容）:**

- velocity の一致は検証しない（MOP3 の定義が noteNumber の一致であるため）
- ドラムノード（node_06）の楽譜検証は対象外（楽譜構造・音域が完全に異なる）
- 推奨ビルド設定: `-DMOP_TEST=3`（SERIAL_DEBUG の値は問わない。MOP_TEST > 0
  のとき自動的に抑制されるため）

## 考察
- 全ノードでエラーパターンが似ており、実際の音符が系統的にズレている → 期待配列の問題が濃厚
- ノートナンバー自体は C4〜A4 の範囲で「かえるのうた」として妥当な音域
- 正しく検証するには production の score_data.cpp から期待配列を自動生成すべき
- CSV 生データ（partId, noteNumber, velocity, seq）は正しく記録されているため再検証可能
