## 検証方法

計画書の MOP2 は「合成出力音を録音・周波数分析し、楽譜の平均律理論音高と比較する」と
定義されているが、実音を録音するとサウンドカード・A/D・部屋鳴りの癖が混入し、
「JSON と合成方法に由来する音階誤差」だけを切り出せない。

そこで **Processing の加算合成アルゴリズムを Python で忠実に再現**し、
「現在の楽器 JSON + 現在の発音方法で鳴るはずの波形」を生成して周波数解析した。
録音経路を挟まないので、検出される誤差は 100% 音源データと合成式に由来する。

- 移植元: `pc_app/common/{SynthVoice,InstrModel,AudioManager}.pde`
- 楽器定義の正本: `pc_app/production/orchestra_resynth/data/*.instrument.json`
- 楽譜の正本: `firmware/production/node_02/src/score_data.cpp`（かえるのうた・全声部同一）
- 対象音: 楽譜が使う 6 音 C/D/E/F/G/A を、各楽器の `brassOctaveShift` 後の実発音オクターブで
  （トランペット/フルート C5–A5・ホルン C4–A4・オルガン C3–A3）
- 合成条件: 44100 Hz / 発音長 3.5 s / 解析窓 1.0–3.0 s

### 基音推定

基音だけを帯域抽出して解析信号を作り、**隣接サンプルの位相差**から瞬時周波数を求め、
ビブラート成分を最小二乗で分離した中心値を「平均音高」とする。

素直に「位相を unwrap して微分」すると、振幅が落ちた瞬間に位相が乱れて 2π の取り違え
（位相スリップ）が起き、周波数が数 cent 単位で狂う（フルートで実際に −4.8 cent の嘘が出た）。
位相差なら (−π, π] に収まるので原理的にスリップしない。さらに**振幅の 2 乗で重み付け**して、
振幅が落ちた瞬間（ノイズが支配して位相が信用できない瞬間）の寄与を自動的に下げている。

### セルフテスト（既知信号での分解能検証・10/10 PASS）

| テスト | 結果 |
|---|---|
| 純音 130.8 / 261.6 / 440 / 880 Hz | 誤差 ≤ 0.004 cent |
| デチューン純音 +7.3 cent | 復元 +7.3005 cent |
| ビブラート（20 cent p-p / 5.7 Hz）の中心 | 誤差 +0.016 cent |
| 同 振れ幅 | 20.001 cent（真値 20.0） |
| トレモロ（AM）混入時の中心 | 誤差 +0.0004 cent |
| FFT ピーク法（クロスチェック側） | 誤差 0.00001 cent |

**サブセント精度**（要求 < 0.05 cent）を満たす。

## 誤差の出どころ

合成の音高は `SynthVoice.pde` の次の式だけで決まる:

```
f = targetF0 * harmRatio[k] * sqrt(1 + inharmonicity_b * n^2) * pitchMul
targetF0 = 440 * 2^((midi-69)/12)     ← 平均律そのもの（ここに誤差はない）
```

MIDI → 周波数の変換は平均律の定義式そのままなので誤差ゼロ。よって音階誤差は
**`harmonics[n=1].ratio` と `inharmonicity_b` にしか宿らない**。

| 楽器 | ratio(n=1) | ratio 由来 | inharmonicity_b 由来 | 理論合計 | 実測との差 |
|---|---|---|---|---|---|
| トランペット | 0.999500 | **−0.866** | +0.000 | −0.866 | 0.002 |
| ホルン | 0.998900 | **−1.905** | +0.000 | −1.905 | 0.001 |
| フルート | 1.000000 | +0.000 | +0.042 | +0.042 | 0.134 |
| オルガン | 1.000000 | +0.000 | +0.081 | +0.081 | 0.011 |

**結論: 誤差の正体は `harmonics[n=1].ratio` が 1.0 でないこと**。トランペット 0.9995、
ホルン 0.9989 と、第 1 部分音がわずかに低い位置に測定されており、それがそのまま
「基音が −0.87 / −1.91 cent 低く鳴る」として出ている。音源録音を解析器にかけたときの
「宣言された基音」と「実際に測られた第 1 部分音の位置」のわずかな食い違いが残っている。
非調和性 `inharmonicity_b` の寄与は +0.04〜+0.08 cent で無視できる。
フルート・オルガンは ratio が 1.000000 に丸められているため誤差はほぼゼロ。

実測値は**瞬時周波数法と FFT ピーク法の独立 2 手法**で一致を確認している
（2 手法の最大差: トランペット 0.002 / ホルン 0.001 / フルート 0.101 / オルガン 0.011 cent）。
フルートだけ差が大きいのは、4 楽器で唯一 `noise.level = 0.022` の整形ノイズを持ち、
かつ包絡ループの折返しで基音の振幅が一瞬ほぼ 0 まで落ちるため（下記）。
それでも 0.1 cent 前後であり、基準 3.6 cent に対して無視できる。

### 補足: `fundamental_hz` は音高に影響しない

オルガンの JSON は `fundamental_hz = 263.141`（C4 = 261.626 Hz より **+10 cent 高い**）だが、
合成では `fundamental_hz` を使わない（倍音比 `ratio` は録音の基音に対する相対値なので、
平均律の `targetF0` に載せ替えた時点で録音側のズレは消える）。実測でもオルガンの誤差は
+0.08 cent であり、+10 cent は伝播していないことを確認した。

## ビブラートの扱い（平均音高とは分けて集計）

ビブラートは瞬時音高を周期的に振るが**平均音高は動かさない**ため、上表の「平均 |誤差|」
とは分けて「振れ幅 (peak-to-peak)」として集計した。振れ幅は JSON の
`modulation.vibrato.depth_cents` がそのまま出ている（トランペット 5 cent・フルート 20 cent）。

フルートは瞬時値が ±10 cent 振れるので「ある一瞬を切り取れば 3.6 cent を超える」が、
これは音階誤差ではなく狙って入れた音楽的な揺れである。

## 副産物: フルートの包絡ループに 88% の段差

解析中に見つけた**音高とは無関係の**所見。フルートの JSON は
`loop_start_sec = 1.425` / `loop_end_sec = 2.14` で、包絡値が loop_end で 0.054、
loop_start で 0.478 と **88.6% の段差**があり、ループ折返しのたびに基音の振幅が
一瞬ほぼ無音まで落ちる。

ただし **実演奏では発火しない**: 120 BPM の 4 分音符は約 470 ms で、`loop_start_sec`
（1.425 s）に到達する前に noteOff されるため、ループ区間そのものに入らない。
ロングトーンを鳴らしたときだけ問題になる。音高には影響しないので MOP2 の判定には
無関係だが、音色の作り込みで長い音を使うなら要注意。

## 限界・未検証

- **実音の録音では検証していない**。本解析は Processing の合成式を Python で再現した
  シミュレーションであり、「JSON と合成方法に由来する音階誤差」だけを見ている。
  Minim の内部処理・サウンドカードの D/A・リサンプリング（44100 Hz 以外の出力デバイス）で
  さらにズレが乗る可能性は否定できない。ただしこれらは通常 cent 未満であり、
  今回の余裕（基準の 20%）を食い潰す規模ではない。
- 合成は float64 で行った（Processing は float32）。位相は 2π で巻き戻されるため
  float32 の丸めは酔歩にとどまり、周波数の系統誤差にはならない（0.001 cent 未満）。
- 打楽器（`instrumentId` 4–7 / node_06）は音高を持たないため対象外。

## 再現手順

```bash
cd tools/verification
.venv/bin/python scripts/mop2_pitch_error.py            # セルフテスト → 解析 → グラフ
.venv/bin/python scripts/mop2_pitch_error.py --selftest # 推定器のセルフテストだけ
```

実機・録音・シリアル接続は不要（JSON と楽譜から決定論的に再現される。ノイズ生成の乱数も
シード固定）。`results/mop2/<timestamp>.csv` と `_summary.txt` が出る（どちらも .gitignore
対象。数値の正本はこのファイルと `results/graphs/mop2_pitch_error_slide.png`）。
