# MOP8: CPU 負荷 評価記録

> 対応する MOP 定義: MOP-6（入力フェーズ CPU 時間 ≤ 2 ms）
> 検証コード上の番号は MOP_TEST=8

## 目標値

- **主判定**: 入力フェーズ処理時間 ≤ 2 ms（2000 us）
- **副次チェック**: 3 フェーズ合計 ≤ 5 ms（5000 us）

根拠: MPU6050 の I2C サンプリング 200 Hz（周期 5 ms）に対し、EMA 3 フェーズ
（入力→ロジック→出力）が 5 ms 以内に収まる必要がある。入力 ≤ 2 ms なら
残り 3 ms をロジック＋出力に割り当てて周期を維持できる。

## 検証方法の妥当性

### ファーム側（`MOP_TEST=8` ブロック）

- **計測手段**: `micros()` を loop() 内の 3 フェーズ境界（t0/t1/t2/t3）で取得し差分を出力
- **計測精度**: ESP32-S3 の `micros()` 分解能は 1 us。フェーズ処理時間が数百〜数千 us の
  オーダーなので十分な精度
- **observer effect**: `mop_test::mprintf()` は t3 取得後に呼ばれるため、シリアル出力の
  オーバーヘッドは計測値に含まれない。ESP32 の Serial TX はリングバッファ経由で
  割り込み駆動のため、次ループの t0 にも実質影響しない
- **出力頻度**: 毎ループ（約 200 Hz）で全サンプルを出力。1 行 ≈ 25 バイト × 200 Hz ≈
  5000 B/s で 115200 baud の約 43%。帯域的に余裕はあるが、長時間計測では
  TX バッファ溢れの可能性を考慮して `--duration 30`（既定）程度が推奨
- **micros() ラップアラウンド**: `uint32_t` で約 71 分周期。差分計算は unsigned 減算で
  単一ラップを正しく処理する。既定 30 秒では問題なし

### Python 側（`mop8_cpu_load.py`）

- **ウォームアップ**: 先頭 20 サンプルを除外して初期化コスト（I2C 初回読取、WiFi 初回
  パケット処理等）の影響を排除
- **統計**: 各フェーズの平均・p99・最大を報告。p99 は外れ値に頑健な指標として、
  最大値は worst-case 保証として機能する
- **合計チェック**: 入力 ≤ 2 ms の主判定に加え、合計 > 5 ms を WARNING として報告

### 既知の制約

- `MOP_TEST=8` ビルドでは `SERIAL_DEBUG` のダンプ（`dumpPeriodic` 等）が
  `#if SERIAL_DEBUG && MOP_TEST == 0` ガードで無効化されるため、計測時のシリアル負荷は
  MOP8 出力のみ。通常運用（`MOP_TEST=0, SERIAL_DEBUG=1`）ではデバッグ出力の
  オーバーヘッドが加算されるが、それは出力フェーズ外のコストなので入力フェーズには影響しない
- 楽器ノード（node_02〜06）の CPU 負荷は本 MOP の対象外（指揮者 node_01 のみ）

## 考察
- 入力フェーズ最大 0.72ms は目標 2ms に対して 2.8 倍の余裕
- 3フェーズ合計最大 1.50ms はループ周期 5ms に対して 3.3 倍の余裕
- ロジック（applyPattern）は平均 4us と極めて軽い
- 出力フェーズの最大 912us は UDP 送信のジッタによるスパイクと推定
- 入力フェーズの p99=620us は安定しており、724us の最大値は I2C 読み取りの単発遅延と推定
