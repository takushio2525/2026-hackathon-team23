## 3. レポート公表値の出所解明

### 3.1 公表値は ahead ベースで完全再現できる

`logs/test_20260709_234302.log` の楽器側 `[Nx EVT BEAT] ... ahead=<ms>` 行（各ノード 157 拍、初回受信のみ）から:

- **MOP4 = 同一拍の ahead のノード間レンジ**: n=157、平均 **14.3** / p50 **13.0** / p95 **28.0** / 最大 **35** ms、20ms 超 **26** 件 → レポート記載（14.3 / 13 / 28 / 35、26/157）と**全一致**
- **MOP5 = 同一拍の ahead のノードペア差**: 全ペア n=1570、平均 **6.8** / p50 **5.0** / p95 **18.0** / 最大 **35** ms、最大は **node_05–node_06** ペア → レポート記載（6.8 / 5 / 18 / 35、node_05–06）と**全一致**

つまり 7/9 の公表値は、レポートに書かれた「NOTE_ON の PC タイムスタンプベース」ではなく、**7/2 と同じ ahead（デバイス時計）ベース**で算出されている。実際、7/2 の MOP4 も旧スクリプトが `localMasterMs = playAt − ahead` の同一拍ノード間レンジを取っており（旧 `mop4_sync_error.py`、コミット `7973294` 以前）、playAt は全ノード共通なので**数学的に同じ量**である。したがってレポートの 7/2→7/9 比較表（14.1→14.3 等）は、結果的には方式が揃った公正な比較になっている。

### 3.2 一方、改修スクリプト（NOTE_ON ベース）の実出力は破綻している

コミット `7973294`（7/9 23:01）で MOP4/MOP5 スクリプトは NOTE_ON の PC タイムスタンプ方式に改修されたが、その直後の実測（23:39〜）の出力は使い物になっていない。

- `results/mop4/20260709_233957_summary.txt`: 平均 2586.3ms / 最大 5263.3ms / FAIL（ノード別偏差 node_06 +1644.1ms など）
- 原因（コード読解）:
  1. **ライブ計測の逐次ブロッキングポーリング**: `mop4_sync_error.py:99-105` は 6 ポートを for 文で 1 行ずつ読む。`common.py:131` の `timeout=0.1` により、無データのポートで最大 100ms ずつ順にブロックするため、受信タイムスタンプ `pc_ts = time.time()`（読み出し時刻）が実到着から最大数百 ms〜秒単位で遅れる。バックログが溜まるほど遅れが累積する（node_06 = 毎拍発音するドラムで偏差最大、と整合）
  2. **`--log` 解析モードは形式不一致で全滅**: `mop4_sync_error.py:154` は行をタブで分割するが、`serial_logger.py:72` は**スペース区切り**（`ts port text`）で書く。このため既存ログを `--log` で解析すると pc_ts が取れず NOTE_ON が全て捨てられ、結果ゼロ件になる
  3. **NOTE_ON のノードプレフィックスが全ノード `[N2` 固定**: `NoteSenderModule.cpp:63` が共通ライブラリ内で `"[N2 NOTE_ON ]"` をハードコードしており、行頭プレフィックスからノード判定できない（part= フィールドのみ有効）
  4. **直前 EVT BEAT 紐付けの誤対応**: §2 主張 2 の通り
- `mop_graphs.py` の `graph_mop4()` も同じ CSV から再計算する構造（`mop_graphs.py:148-177`）で、さらに 1 台のみの拍を誤差 0 として混ぜるため（`:173`）、平均を実態より下げる。なおコミット済みの `results/graphs/mop4_sync_error.png` / `mop5_comm_delay.png` は 23:01 時点の生成で **7/2 データ**（最大29.0/平均14.1/121拍、最大37.0/平均5.4/4162サンプル）を描いており、7/9 公表値とも別物

## 4. 計測手法・公表値の総合評価

### 4.1 ahead レンジ（公表値）は何を測っているか

`ahead = playAtMasterMs − (millis() + offsetMs)` を BEAT 初回受信時に計算した値（`node_0X/src/main.cpp:91-101`）。同一拍では playAt が全ノード共通なので、ノード間レンジ = 「各ノードの (受信時刻 + 推定クロックオフセット誤差) のばらつき」。

- USB シリアル遅延は乗らない（デバイス側で計算された値）。この点で NOTE_ON PC タイムスタンプ方式より筋が良い
- ただし「発音時刻の楽器間差」そのものではない。ahead>0 の拍では発音はスケジュール発火（差はクロック誤差のみ）、ahead<0 の拍では到着即発火（差は到着差）になるため、ahead レンジは発音差のやや保守的（大きめ）な代理指標
- 発音時刻をより直接推定する device_t + EMA offset 方式（Codex 方式の再現）では p50=8 / p95≈30 / 最大 61ms。オフセットを別時刻の status 行から取るため数 ms の推定誤差を含むが、「**最大 35ms**」を上限として断定できないことは確実

### 4.2 USB タイムスタンプ方式（改修スクリプトの方式）は原理的に不適

同一拍の楽器 EVT BEAT（デバイス時計上はほぼ同時受信）の PC 受信時刻スプレッドを雑音フロアとして実測すると、**平均 19.6 / p95 29.0 / 最大 35.4ms**（スレッド化された serial_logger のログでも）。目標値 20ms と同桁の雑音が乗るため、ログ収集をスレッド化しても NOTE_ON PC タイムスタンプ方式で MOP4 を判定するのは不適切。逐次ポーリングのライブ計測（現行スクリプト）では前述の通り秒オーダーまで破綻する。

### 4.3 新発見: 45ms lookahead は半数の拍で間に合っていない

7/9 ログの全 785 サンプル（157 拍 × 5 ノード）で、BEAT 初回受信時点の遅れ `lateMs = max(0, −ahead)` は:

- **遅刻 419/785 件（53%）**、遅れの p95 = 91ms、最大 116ms
- ahead は全ノードほぼ共通に振れる（拍ごとのノード平均の SD ≈ 52ms、振れ幅 −110〜+84ms）。共通モードなので**同期は壊れない**（全員一緒に遅れて発火する）。マルチキャストは単一無線フレームを全ノードが同時受信するため、と整合する
- 振れの周期・幅は SoftAP のマルチキャスト配送（DTIM ビーコン間隔 ~100ms でのバッファリング）と整合するが、確定には至っていない
- 理論上 ahead の平均は lookahead の 45ms 付近になるはずだが、実測平均は −9.6ms（7/2 の M5I データでも ≈ +2ms）で、**約 45〜55ms の系統シフトが未解明**。EMA オフセットが平均遅延を吸収する構造（サンプル = `timestampMs − millis()`、`OrcReceiverModule.cpp:18-43`）だけでは説明が付かず、追加調査（B 案の lateMs 計測）が必要

これは「発表で問題なかった」事実と両立する: 発音の同時性（楽器間）は保たれ、犠牲になっているのは指揮棒→発音の絶対レイテンシ（最大 ~160ms = lookahead 45 + 遅れ 116）と拍ごとの揺らぎであり、単独では知覚されにくい。

### 4.4 公表値はどこまで信用できるか（見解）

- **信用できる**: 「楽器間の相対ずれの中央値は十数 ms で、目標 20ms の水準感の中にある」という定性的結論。ahead レンジ（p50=13）と発火差推定（p50=8）が独立にこれを支持する
- **信用できない**: (a)「最大 35ms」という上限（発火差推定の最大は 61ms、かつ計測時間は約 80 秒 × 1 回のみ）、(b)「計測方式: NOTE_ON の PC タイムスタンプベース」という方法記述、(c) MOP4 の出典 CSV、(d) MOP5 を「指揮→楽器通信遅延」の検証と見なすこと（未検証）
- **条件付き**: FAIL（受け入れ）という判定自体は維持可能。ただし根拠数値の出典と方式の記述を訂正し、上限断定を避けた表現に直す必要がある
- 補足: 7/9 の計測は SERIAL_DEBUG=1（NOTE バイナリ送信を抑止しテキストを流すモード、`NoteSenderModule.cpp:61-71`）で行われており、本番構成（バイナリ送信 + Processing 発音）そのものの計測ではない点も最終レポートでは明記すべき

## 5. 今後の対応方針

### A. 最終レポートの記述修正のみ（コード変更なし）— **推奨・必須**

- 内容: §3.3 の 4 点を訂正。MOP4 の方式を「EVT BEAT ahead のノード間レンジ（デバイス時計ベース、7/2 と同方式）」、出典を `logs/test_20260709_234302.log` に改める。最大値は「35ms（ahead レンジ）、発火差推定では最大 61ms 程度まで幅がある」と併記。MOP5 は「計画書定義の通信遅延は未検証、楽器間相対ずれに指標を変更した」ことを明示
- 期待効果: 振り返り・最終レポートの数値が検証可能になり、7/2 比較の整合も取れる
- 工数: 本レポートを根拠に文章修正のみ（~1 時間）。**7/15 までに確実に実施可能**

### B. 再計測(ソフト変更要): Codex 提案の専用ログ + lateMs 計測

- 内容: Codex 提案どおり (1) 発火箇所で `beatNo, playAtMasterMs, millis(), offsetMs, localMasterAtFire` を 1 行出力し、`localMasterAtFire` のノード間レンジで MOP4 を確定、(2) BEAT 受信時と発火時の `lateMs` を専用モードで集計して「45ms 予約に間に合うか」を定量化、(3) M5I 出力を共通モジュールか main の一方に統一、(4) `analyze_log` の区切り文字を `serial_logger.py` の形式（スペース区切り）に合わせる
- 評価: 提案は妥当。特に (1) は USB 雑音とログ紐付け誤りを同時に排除する正攻法で、(2) は §4.3 の未解明シフトの解明にも直結する
- 期待効果: MOP4 の確定値（推定でなく実測）と lookahead 設計の検証
- 工数: ファーム数行 + スクリプト修正で半日、実機 6 台での再計測 1 時間程度。**7/15 までに実施可能だが実機アクセスとメンバー時間に依存**。実施する場合も「実機未テストコードへの介入禁止」ルールに従いユーザー主導で
- 位置づけ: 任意。振り返りの説得力を上げたい場合のみ

### C. ハード計測: GPIO トグル + ロジックアナライザ / 往復同期 (ping/ACK)

- 内容: 送信・受信で GPIO をトグルして外部計測器で絶対片道遅延を測る、または往復時刻同期を実装して計画書 MOP5（通信遅延 ≤30ms）を検証する
- 評価: 絶対片道遅延の検証にはこれしかない（Codex の指摘どおり片方向同期では原理的に不可能）
- 工数: 機材・配線・ファーム改修で数日。**7/15 までは非現実的**。最終レポートには「片道遅延の実測は将来課題」と明記することを推奨

**推奨**: A を必須で実施（本レポートがその根拠資料）。B は時間が許せば (2) lateMs 計測を最優先で 1 項目だけでも実施（未解明の系統シフトと lookahead 妥当性が同時に判る）。C は将来課題として記載に留める。
