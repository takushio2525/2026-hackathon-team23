# meetings — 議事録・進捗管理（例）

ミーティングの記録と進捗管理ドキュメントを置く場所。

## 構成

| ファイル / ディレクトリ | 内容 |
|---|---|
| `_template.md` | 議事録テンプレート（コピーして使う） |
| `wbs.md` | WBS（作業分解）の記入例 |
| `gantt.md` | ガントチャート（Mermaid）の記入例 |
| `MMDD_N回/` | 各回ごとのフォルダ（例: `0415_1回/`, `0422_2回/`）．正規議事録 PDF・AI 要約・当日のメモ等をまとめて置く |
| `MMDD_N回/23_第N回議事録_*.pdf` | 本チームで回している正規の議事録（PDF） |
| `MMDD_N回/第N回議事録_AI要約byClaude.md` | AI（Claude）が PDF を読み解いて生成した補足要約 |
| [`0422_2回/wbs_table/`](0422_2回/wbs_table/) | 第2回議事録 表1 WBS の再構成版（HTML 表で 1 ページに一覧）。チーム共有用 |

## 議事録の作り方

本チームは**PDFの正規議事録をコミットする運用**。議事録担当者が LaTeX 等で
作成した PDF をこのフォルダに置く。

正規議事録に抜けていた決定事項（例: IMU採用）は、議事録を書き直さず
[`docs/decisions/`](../docs/decisions/) に ADR（設計判断記録）として残す。

1. 担当者が議事録 PDF を作成（ファイル名規則: `23_第N回議事録_<学籍番号>.pdf`）
2. `MMDD_N回/` ディレクトリを作ってその下に置いてコミット
3. 補足や PDF で漏れた決定事項は ADR（[`docs/decisions/`](../docs/decisions/)）に記録
4. 必要なら Claude に依頼して同じディレクトリに **AI 要約 Markdown**（`第N回議事録_AI要約byClaude.md`）を生成させてもよい

`_template.md` は markdown で議事録を取りたい場合の参考テンプレ（汎用）。

## 進捗管理のコツ

- **WBS** は最初のうちに作る。粒度は「1タスク = 数時間〜3日」が目安
- **ガントチャート** は WBS と連動させる。Mermaid は GitHub で自動描画される
- 毎週のミーティングで進捗確認し、WBS の「状態」列を更新する

## 不要な班は

進捗管理を別のツール（Notion, Trello, GitHub Projects 等）で行うなら、
このディレクトリごと削除してよい。
