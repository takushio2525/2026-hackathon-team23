## 2. 良かった点

### 2.1 プロジェクト計画の「見直し」というスタンスを明示している

- §3 冒頭「前回作成した役割分担、WBS 図に割り振りのアンバランスさ、不要な項目の追加、項目の不足など不備が多くあるため、大幅な見直しが必要である。また FBS、PBS 図にも不足が見られるため、項目を追加する必要がある」と、第2回議事録（[`meetings/0422_2回/23_第2回議事録_24G1075.pdf`](../../0422_2回/23_第2回議事録_24G1075.pdf)）の WBS / FBS / PBS の **改訂が必要であることを正面から認めて作業に入っている** 点は良い。
- 第2回議事録 表1 の WBS は、後に `meetings/0422_2回/wbs_table/wbs_table.md`（HTML 表 + rowspan で再整理）と `work/shiozawa/work-0422/wbs_proposal/main.pdf` 第4章で塩澤主導の再構成が進んでおり、本書の「見直し」モチベーションはチーム全体の流れと一致している。

### 2.2 新 WBS（§3-1 表 1）の構造化はおおむね妥当

- WBS 表 1 は「時間的フェーズ」「番号」「要素成果物」「活動タスク」「担当者」の 5 列構成で、ルーブリック A 要件「8/80 ルールに従った粒度」に近い形式。
- 設計フェーズ（100）配下に 110 基本設計（4 項目）と 120 指揮棒（10 項目）の 2 つの要素成果物に分けた点は授業資料の WBS 例と整合。

### 2.3 新 FBS 図（§3-2 図 2）の階層化

- 図 2 の FBS は「指揮棒を振る速度に合わせて BPM を変化させるシステム」を最上位に置き、第 2 階層「指揮棒 / 音 / 同時演奏」、第 3 階層「速度変化 / 音の大きさ / 金管楽器 / パーカッション / リズム / 各旋律」、第 4 階層「同じタイミング / 楽譜を元に演奏」の 4 階層に分解されている。
- ADR-0004「金管楽器」+ 塩澤 PDF「金管 3 + ドラム 1」 + ADR-0007「同期誤差 ≤ 20 ms」 + ADR-0007「テンポ追従」を、機能側の言葉で整理した結果として読める。

### 2.4 新 PBS 図（§3-2 図 3）が物理構成を 3 系統に整理している

- 図 3 の PBS は「指揮棒の作成 / Processing による音作り / 通信システム」の 3 系統で、それぞれ下位に「ピーク検知 / 加速度の大きさ解析」「金管の音色作成 / パーカッションの音色作成」「UDP 通信 / 有線通信」「テンポ検出 / 音の高さ・長さ」を配置。
- ADR-0002（UDP）／ ADR-0003（IMU）／ ADR-0004（金管 + ドラム）と矛盾しない。

## 3. 改善・確認すべき点

### 3.1 【最重要】§3-1 WBS 表 1 が `wbs_table.md` の正式版と番号がズレている

リポジトリの正式 WBS（`meetings/0422_2回/wbs_table/wbs_table.md`）と本書 §3-1 表 1 の番号体系が異なっている:

本書は **第2回議事録 表1（旧 WBS）の 17 行版を再編集した状態** で、塩澤主導の再構成版（51 タスク）には追従できていない。第3回でこの番号体系の差を解消し、本書の改訂内容をリポジトリ正式版にマージするか、本書を破棄して `wbs_table.md` を採用するかを決める必要がある。

### 3.2 【最重要】WBS 表 1 から本人（片岡）・地曵・御代川の名前が抜けている

- §3-1 表 1 の担当者列は **塩澤 / 齋藤 / 梅澤 の 3 名のみ** で、24G 勢（片岡・地曵・御代川）の名前が一切現れない。
- これは `wbs_table.md` の整理「24G 勢は議事録持ち回りと 300 テストフェーズ以降の実機検証・発表準備に合流するため、本表の活動タスク列には記載しない」と整合してはいるが、**本書の §3 タイトル「プロジェクト計画の見直し」で WBS を改訂しているのに、見直し後も自分（片岡）の役割が空白** という結果になっており、これでは「見直しの実効性」がない。
- 第3回で以下のいずれかに決める必要がある:
  1. 24G 勢は 300 テストフェーズ以降に集中（現状維持）。代わりに **「議事録・計画書編纂・調達・発表準備」を 100/500 系に明示的なタスクとして追加**。
  2. 24G 勢のうち一部（例: 御代川）が計画書編纂、片岡が発表準備、地曵が Sync-Loss フェイルセーフ等のサブタスクを引き受ける、と分業を確定。
- 例えば「511 発表準備・デモ調整 → 全員」「521 成果報告書 → 全員」「**112 PBS の最終化 → 全員**」「**113 MOE/MOP/TPM の最終化 → 全員**」のように、`wbs_table.md` で既に「全員」タスクと書かれている部分から 24G 勢の貢献領域を明示するのが最小コストの解。

### 3.3 【重要】200 実装フェーズと 300 テストフェーズが「未定」のまま

- 表 1 の最下部 3 行「200 実装フェーズ / 未定 / 未定 / 未定 / 未定」「310 単体テスト / 未定 / 未定」「320 結合テスト / 未定 / 未定」は、§3-1 末尾の「実装フェーズや、テストフェーズについてはまだ未定であるため、次回の授業内で議論し、必要となる項目を追加する必要がある」と整合しているものの、**リポジトリ `wbs_table.md` には既に 200 系（211-254 の 19 項目）／300 系（311-337 の 15 項目）／400 系（411-413 の 3 項目）／500 系（511-521 の 2 項目）が書かれている**。
- つまり「未定」は本書独自の状態であって、チーム全体の状態ではない。第3回で本書の WBS をリポジトリ版に置換するか、リポジトリ版を本書に転記する必要がある。
- ルーブリック A 要件「WBS とクリティカルパスを考慮した役割分担とスケジュールが計画されている」は、200 / 300 が「未定」のままだと達成不可能。

### 3.4 【重要】アローダイアグラム / クリティカルパス / ガントチャートが無い

- マネジメント計画書ルーブリック「作業計画と役割分担」項目の **A 要件 / B 要件いずれにもアローダイアグラム（とクリティカルパス）が含まれる** ため、本書の §3-1 のように WBS 表のみではこの項目で C 評価止まり。
- 例えば以下のような粗い依存関係を 1 図にまとめるだけで C → B / B → A に上がる:
  - 113 MOE / MOP / TPM 確定 → 114 / 115 / 116 / 117 基本設計 → 121-130 詳細設計 → 200 実装 → 300 テスト → 500 発表
  - 並走可能: 116 楽譜データ形式 と 114 アーキ設計 は並走可能、127 FFT 解析は 117 音色合成と並走可
- ガントチャートも、4/15 〜 7 月末までを週単位で書き出すだけで「実行可能性」項目の評価が上がる。

### 3.5 【重要】§2 詳細設計の本文が無い

- 目次相当の §2「詳細設計」が **章タイトルのみで本文ゼロ**。基本設計（§1, 1 段落）と見直し（§3）の間に §2 が空いている。
- 24G 勢が WBS 上「設計フェーズに担当タスクを持たない」とはいえ、本書の節タイトルで §2 を立てた以上、何かしらの詳細設計の記述（少なくとも担当範囲：300 テストフェーズに対する実機検証手順の素案）は欲しいところ。
- §2 を残すか削除するかを第3回で確定。

### 3.6 【重要】3 つの図のキャプションが全て同じ無関係な文字列になっている

- 図 1（WBS 全体ツリー、p.1）、図 2（新 FBS 図、p.2）、図 3（新 PBS 図、p.3）の **3 つの図キャプションが全て「それぞれの搭載可能の最大次元数」** と書かれている。
- これは別文書（おそらく講義資料か別レポートの図キャプション）からのコピペ残りで、本書の図内容とは完全に無関係。LaTeX なら `\caption{}` のテキストを差し替え忘れた状態。
- 本来は以下のようなキャプションが必要:
  - 図 1 → 「WBS の階層構造（設計／実装／テストフェーズ）」
  - 図 2 → 「新 FBS（機能分解構造）」
  - 図 3 → 「新 PBS（製品分解構造）」
- ルーブリックの「実行可能性と客観性」項目で「**主観的・誇張的な表現が多く、内容の妥当性に疑問がある**」（C 要件）または「**根拠が示されておらず、主張のみで記述されている、または誤解を招く表現が多い**」（D 要件）に該当しかねない致命的なケアレスミス。提出前のセルフチェックを推奨。

### 3.7 §3-2「FBS と PBS の対応をさせることができた」の根拠が示されていない

- §3-2 末尾「今回の修正によって、前回作成した FBS と PBS ではできていなかった FBS と PBS の対応をさせることができた」と記述されているが、**FBS と PBS の対応関係を示す表・図・本文は本書中に存在しない**。
- 例えば次のような対応表を 1 つ追加するだけで C → B に上がる:

  | FBS（機能） | PBS（部品） | 担当 WBS |
  |---|---|---|
  | 速度変化（指揮棒） | 加速度の大きさ解析（指揮棒の作成） | 122 / 222-224 |
  | 音の大きさ（指揮棒） | ピーク検知（指揮棒の作成） | 122 / 223 |
  | 金管楽器（音） | 金管の音色作成（Processing） | 117 / 252 |
  | パーカッション（音） | パーカッションの音色作成（Processing） | 117 / 252 |
  | 同じタイミング（同時演奏） | UDP 通信（通信システム）+ テンポ検出 | 115 / 213 |
  | 各旋律（同時演奏） | 音の高さ・長さ（有線通信） | 130 / 233 |

### 3.8 §1 基本設計の独自性・新規性が不足

- §1 は本書唯一の本文記述で「我々が今回、提案するシステムとして、加速度センサを用いた指揮棒を振り、振る速度や周期に合わせてテンポや音の大きさを変化させるシステムを提案する」と書かれているが、**ADR-0007 の「外的要因に作用されず音楽未経験でも演奏できる」という上位目的への言及や、既存技術（テルミン / Wii Music / バーチャル指揮者）との差別化点が示されない**。
- マネジメント計画書ルーブリックの「プロジェクトの概要」A 要件は「技術的な課題、既存の技術、作成するシステムの目的と独自性について、参考文献を交えながら十分に記述されている」を要求しており、現状は D 評価相当。
- 御代川 PDF §1.1.1〜§1.1.3 は同じ要件を概ね満たしているので、第3回でこの 3 名（御代川・地曵・片岡）の §1.1 部分を統合する作業が望ましい。

### 3.9 §4 AI の仕様について — 申告内容の粒度

- 「表の整備に使用」だけでは、AI 利用申告書（`work/shiozawa/ai_declaration/` 配下）が要求する「使用したモデル / プロンプト / 該当箇所 / 介入度」の粒度に届かない。
- 提出物としての形式整合性のために、テンプレに沿って **どの章のどの図表を、どのモデルで、どの程度修正したか** まで具体化すべき。

### 3.10 細かい指摘

- §3-1「以下の通りに、大幅の修正を行った役割分担表 1 を記載した。また、新たに WBS 図 1 を作成した」 — 「役割分担表」と「WBS 図」が別物として書かれているが、表 1 に両方が含まれている（役割分担 = 担当者列）。「役割分担表（WBS 表 1）」または「役割分担を含む WBS 表 1」と表記を統一。
- §3-2「修正をした FBS 図が以下の図 2 であり、PBS 図が以下の図 3 である」 — 図 2 / 図 3 のキャプションは前述の通り別文字列なので整合しない。
- WBS 表 1 の担当者列「齊藤」と「斎藤」が混在（113 = 齊藤、128/129/130 = 斎藤）。実際は「齋藤」（旧字「齋」）が正しい表記。
- 表 1「125 4 種類の楽器パートを鳴らす仕組み（梅澤）」 — `wbs_table.md` の対応タスクは「125 Processing 詳細設計：4 種類の楽器パートを **同時に** 鳴らす管理（どの音をどの発音枠で鳴らすか）」。「同時に鳴らす管理」という核（ボイスマネージャ）が落ちている。

## 6. 総合所見

本書は「プロジェクト計画の見直し」というタイトル通り、第2回議事録の WBS / FBS / PBS の **改訂作業の中間報告** である。マネジメント計画書ルーブリック軸での評価対象として完成した文書ではなく、計画書本体（御代川 PDF）に統合される素材としての位置づけが妥当。

良かった点としては「見直しの必要性を正面から認めた」「WBS / FBS / PBS の 3 セットを更新する」というスタンスが明確であり、第2回議事録の旧 WBS（17 行）の不備（実装フェーズ・テストフェーズが粗い、24G 勢の役割が不明確）に対する問題意識は妥当。

一方、致命的な問題が 3 つある:
1. WBS 表 1 の番号体系がリポジトリ正式版（`wbs_table.md`）と異なり、塩澤主導の再構成（51 タスク）に追従できていない。
2. 24G 勢の名前が見直し後の WBS にも出てこない（「見直し」の実効性がない）。
3. 図 1 / 2 / 3 のキャプションが全て無関係な文字列（コピペエラー）。

第3回ではこれら 3 点を解消した上で、本書の改訂内容を **御代川 PDF の §1.3 章立てに統合** することを推奨したい。本書単独では計画書として完成しないが、御代川の章立ての中身を埋める素材としては有効に機能する。地曵 PDF（システム設計書寄り）と本書（計画書改訂寄り）と御代川 PDF（計画書骨格）を 3 名で分担して 1 本の計画書本体に統合できれば、24G 勢の貢献領域も WBS 上に立つ。

### 7.3 システム設計書としての伸ばしどころ

24G 勢の役割（300 テストフェーズ以降の実機検証・発表準備）から見て、本書を **システム設計書として伸ばす方向はそもそも筋が悪い**。地曵 PDF（システム設計書 6/10）と御代川 PDF（システム設計書 3/10）が既に存在するため、3 本の設計書を並走させるより、片岡 PDF は **「24G 勢が担当する WBS 改訂 / 役割再分配 / 300 テスト計画」に専念** するほうがチーム全体の労力配分として効率的。

具体的には:

1. **§1 基本設計と §2 詳細設計を削除**（または地曵 PDF に統合）
2. 本書の §3「プロジェクト計画の見直し」を主軸として、`wbs_table.md` 正式版にマージするための **差分提案 PDF** に書き直す
3. 24G 勢の貢献を WBS 上に明示するセクション（議事録 / 計画書編纂 / 機材調達 / 300 テスト実機検証 / 発表準備 / ポスター作成）を追加
4. 図 1 / 2 / 3 のキャプションを正しい内容に修正（コピペエラーの解消）

こうすれば、本書は **マネジメント計画書ルーブリック側の §1.3「作業計画と役割分担」評価** で D → C / B に上がる素材として有効に機能する。一方、システム設計書ルーブリック側で点を取るのは、3 名の役割分担として地曵 PDF に任せるのが現実的。
