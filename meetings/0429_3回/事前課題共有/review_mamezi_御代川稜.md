## 2. 良かった点

### 2.1 マネジメント計画書テンプレートの章立てを完全に踏襲している

- 目次（p.3〜p.4）が **§1.1 概要 / §1.1.1 題目 / §1.1.2 目的 / §1.1.3 既存技術と独自性 / §1.2 スコープ / §1.2.1 成果物 / §1.2.2 対象範囲・非対象範囲 / §1.2.3 FBS / §1.2.4 PBS / §1.2.5 FBS と PBS の対応関係 / §1.3 作業計画 / §1.3.1 WBS / §1.3.2 アローダイアグラム / §1.3.3 役割分担 / §1.3.4 ガントチャート / §1.4 V&V / §1.4.1 MOE / §1.4.2〜§1.4.5 各層の MOP / §1.4.6〜§1.4.7 TPM / §1.5 資源・調達管理 / §1.6 リスク管理** と、ルーブリック 5 項目を網羅する正しい章立て。
- 他の事前課題（地曵 PDF / 片岡 PDF）はマネジメント計画書としての章立て自体が無く、本書の構成は最も計画書ルーブリックに対応している。中身を埋めれば最も高得点を狙える土台。

### 2.2 §1.1.2 プロジェクトの目的と課題列挙が具体的

- ADR-0007 の目的「外的要因に作用されず、機器さえあれば（音楽未経験でも）演奏できる」と整合した記述で、6 つの課題（指揮動作の正確検出 / リアルタイム反映 / 誤検出防止 / 経験不問の操作性 / 視覚的状態提示 / 環境非依存）に分解されている。
- 各課題に「なぜなら〜である」と理由付きで書かれている点は、ルーブリック A 要件「技術的・社会的課題が参考文献を交えて十分に記述されている」に近い水準。

### 2.3 §1.1.3 独自性の主張が成立している

- **「複数マイコンによる分散演奏と、身体動作によるリアルタイム指揮制御を両立するシステム」** という独自性の主張は、テルミン（単機・身体動作）／Wii Music（単機・コントローラ動作）と比較して立っており、論理的に矛盾がない。
- 既存技術 2 件（テルミン論文・Wii Music 公式）との対比軸も明示されている。

### 2.4 §3.2.1 関数設計のモジュール分割が EMA と整合

- §3.2.1 表「関数名 / 機能」の `SensorModule:update()` / `TempoModule:update()` / `NetworkModule:update()` / `SoundModule:update()` / `LedModule:update()` / `ModuleTimer:isReady()` / `setup()` / `loop()` という命名は、`firmware/common/lib/` の Embedded-Module-Architecture（ADR-0005）と整合。
- 「保守性・再利用性・役割分担の明確化を目的として、入力・ロジック・出力の 3 フェーズ構成でソフトウェアを設計する。各機能は独立したモジュールとして実装し、共有データを介して連携する。周期処理にはタイマ制御を用い、`delay()` によるブロッキング処理は行わない」（§3.2 冒頭）は、塩澤 PDF §1.2.4 / §2.4.1 と完全に同じ思想。

### 2.5 §3.3 テスト設計が階層化されている

- 単体テスト（§3.3.1）を入力・ロジック・出力の 3 モジュール別に分けて、それぞれにテスト方針（境界値・条件分岐網羅・形式とタイミング検証）を書いている。
- 結合テスト（§3.3.2）／タイマ制御テスト（§3.3.3）／システムテスト（§3.3.4）と階層も適切。`wbs_table.md` の 311-337 と直接マップできる粒度。

### 3.2 【重要】参考文献 [2] Wii Music の論拠が弱い

- §1.1.3「Wii Music では、コントローラの動作から拍を検出し、音楽のテンポを制御している」と書かれているが、**Wii Music は実際にはテンポ検出ではなく振り動作のタイミングに合わせて事前に決められた音符を再生する仕組み** で、本書の指揮・テンポ追従とは挙動が異なる。
- 引用 [2] が公式サイト URL のみであることに加え、引用文の事実関係も精査されておらず、ルーブリック A 要件「参考文献を交えながら十分に記述」を満たしにくい。
- 学術論文側（[1] 川合・伊藤「非接触型電子楽器の開発を通じたサウンドデザイン」, 東京工科大学メディア学部, 先端芸術音楽創作学会会報 Vol.14 No.1 pp.14-19, 2022）は適切な引用形式。これに加えて Wii Music の代わりに以下のような既存研究を追加すると独自性の主張がさらに強くなる:
  - VirtualConductor（バーチャル指揮者システム）の論文
  - 加速度センサベースの拍検出に関する MIR 系論文
  - PTP / IEEE 1588 / NTP の時刻同期に関する文献

### 3.3 【重要】§3.1 必要機材表の KXR94-2050 が ADR と整合しない

- 表 3.1「必要機材一覧」で **「3 軸加速度センサ KXR94-2050、1 個」** と記述。
- KXR94-2050 は **アナログ出力（電源 5 V、出力レンジ 0.6V〜4.4V）** の加速度センサで、Arduino の ADC（10bit、0〜5V）に直結する想定。一方、塩澤 PDF §1.3.1 / §2.2 と既存議事録は **MPU6050（I2C 接続、±2g/±4g/±8g/±16g 切替、6 軸 IMU）** を採用している。
- KXR94-2050 を採用するなら以下のリスクを §1.6 に書く必要がある:
  1. **アナログ出力 → ADC 変換のノイズ**（MPU6050 の I2C デジタル出力より精度が劣る）
  2. **3 軸のみ**（角速度ジャイロが無いため、塩澤 PDF §2.4.2.4 の「強い振り検出」フォールバック条件が成立しない）
  3. **ADR-0003「指揮者は IMU でジェスチャ」** と整合させるなら 6 軸が望ましい
- 第3回で「センサは MPU6050 / KXR94-2050 のどちらを採用するか」の合意が必要。
- また数量「1 個」は指揮棒ノード 1 台分のみで、楽器ノード 4 台にはセンサ無し（楽器側は LED マトリクスのみ）という構成として正しいことを明示すべき。

### 3.4 §1.1.1 の題目がルーブリック A 要件にやや届かない

- 題目「**指揮用 Arduino から指揮をするオーケストラ**」は、課題（指揮）と技術（Arduino）と目的（オーケストラ）のキーワードを含んでいる点でルーブリック B 相当。
- ただし「指揮用 Arduino から指揮をする」が同じ語の繰り返しになっており、A 要件「過不足なく的確に表現した題目」としては冗長。
- 例: **「Arduino と加速度センサを用いた指揮ジェスチャによる分散電子オーケストラの実現」** のように、技術（Arduino + 加速度センサ）・課題（指揮ジェスチャ検出）・目的（分散電子オーケストラ）を 1 文で揃えるとルーブリック A に近づく。

### 3.5 §2.1 FBS 図が分解粒度として浅い

- §2.1 の FBS 図は 3 階層（最上位「指揮棒を振る速度に合わせて BPM を変更するシステム」 → 第 2 階層「演奏 / 指揮棒 / 旋律」 → 第 3 階層「金管楽器 / パーカッション / 木管楽器 / 速度変更 / 音の大きさ / 主旋律 / リズム」 → 第 4 階層「同じタイミング / 楽譜を元に演奏」）と分解されているが、以下の問題がある:
  - **「指揮棒」を入力インタフェースと位置づけながら、指揮棒の機能（拍検出・テンポ算出・送信）が分解されていない**
  - **「木管楽器」が登場するが、ADR-0004 / 塩澤 PDF / WBS では金管 + ドラム** で確定済み。木管は範囲外
  - 塩澤 PDF §1.1.1 の F1〜F5（指揮情報抽出 / 同期配信 / 楽譜進行 / 音色合成 / 共通基盤）と粒度が揃わない
- ルーブリック A 要件「FBS と PBS が機能と部品に適切な粒度で分解されており、対応関係が明確に示されている」を満たすには、**機能 ID（F1〜）と部品 ID を付与し、対応表 1 つで一覧化** すべき。

### 3.6 §3.1 PBS 図と FBS 図の対応関係が示されない

- §2.1 FBS 図と §3.1 PBS 図がそれぞれ独立しており、どの機能がどの部品で実現されるかの対応表（FBS-PBS 対応表）が無い。
- ルーブリック A 要件の核は「FBS と PBS の対応関係が明確」なので、表 1 行 = 1 機能の単純なマッピング表を §1.2.5 に追加するだけで C → B / B → A に上がる。

### 3.7 第3章 §3.3「テストの設計」と §1.4 V&V の重複・整合性

- §3.3 単体（入力・ロジック・出力）／結合／タイマ制御／システムの 4 階層は良いが、これらに **対応する MOP（合否基準の数値）が一切ない**。
- 例えば §3.3.4 システムテスト「最終的に実環境においてシステム全体の動作確認を行う。長時間の連続動作による安定性の確認や、入力異常などを想定した異常系テストを実施し、実運用に耐えうることを検証する」は **何分連続動作するか／何 % のパケロスを入れるか** の数値が無い。
- ADR-0006 の MOE = 20 ms、塩澤 PDF §1.1.2 の MOP-1〜MOP-8、`wbs_table.md` の TP-1〜TP-7 を §1.4 に転記すれば一気に B 評価以上になる。

### 3.8 §3.4 AI 利用について — 申告内容の粒度

- 「文章構成、頭の中のアイデアを箇条書きにし、その内容の要約、、や。や語尾の統一のために使用」という記述は、本リポジトリ `work/shiozawa/ai_declaration/` 配下のテンプレに沿って **どのモデル（ChatGPT / Claude）を、どの章のどの粒度の作業で利用したか** まで具体化する必要がある。「、や。の統一」と「文章構成」は介入度が大きく異なるので分けて記述する。

### 3.9 細かい指摘

- p.6 §2.3 状態遷移図 — 図 2.4 のキャプションが「**本プロジェクトのハードウェアの状態遷移図**」となっているが、図の中身はシーケンス図（加速度センサ → 指揮用 Arduino → 楽器用 Arduino → Processing の時系列）に見える。状態遷移とシーケンス図は別物なので、キャプションを「シーケンス図」に直すか、状態遷移図（Initialization / Sync / Ready / Playing / Stop など）を別途追加する。
- p.7 §3.1 「2.1 は本プロジェクトの PBS 図である」 — 図番号が 2.1 と書かれているが、実際の図番号は 3.1。番号誤記。
- p.8 §3.2.1 関数設計表 — `setup()` / `loop()` は Arduino 標準関数だが、`SensorModule:update()` 等の `:` は C++ 構文ではなく `::`（スコープ解決演算子）が正しい。`SensorModule::update()` に修正。
- 表紙の Version `x.x` は提出版なら `0.1` 等の確定値に。

## 6. 総合所見

提出用計画書テンプレートの **章立てを過不足なく踏襲している** 点で、3 つの 24G 勢提出物の中では最もマネジメント計画書ルーブリックに正しく対応している。中身が埋まれば、合計 11〜12 / 15 と十分に B 以上の評価が見込める。

ただし、本書最大の問題は **§1.2〜§1.6 の本文がほぼ全節で空** であること。これは「下書きの章立て骨格を提出した状態」に近く、ルーブリックの「成果物の対象範囲」「作業計画」「実行可能性」3 項目で得点できていない。第 2 章・第 3 章で部分的に補えている内容はあるものの、計画書の核（スコープ・WBS・V&V・リスク）が章タイトルのみという状態は、評価対象としては D 寄りになる。

幸いなことに、空欄を埋める材料はリポジトリにすべて揃っている（`wbs_table.md` / ADR-0004〜0007 / 塩澤 PDF §1.1.2 / `meetings/0422_2回/第2回議事録_AI要約byClaude.md` 等）。第3回までに転記＋整合作業を行うだけで本書は計画書本体として機能する。本書を **チームの計画書本体（v0.1）の母版** として位置づけ、御代川を「マネジメント計画書の編纂責任者」とする運用に倒すのが、本書の構造を最も活かす道筋になる。
