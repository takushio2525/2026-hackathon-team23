## 2. 良かった点

### 2.1 楽譜データを「相対表現」で持つ方針

- §1 で「絶対時間（ms）配列ではなく『音高 × 拍数』の相対表現」とした判断は妥当。
- 指揮テンポが可変（ADR-0007 のゴール「指揮を振るテンポを変えたときに演奏速度も変えられる」）である本プロジェクトでは、絶対時間表現は破綻するため、この方針は必須要件と整合。
- 各 Arduino ノード上にメモリ保持する方針も `docs/design/score_format.md` の「楽器マイコン内部で保持」「SD・外部ストレージに頼らない」と一致。

### 2.2 ノード割当ての明示

- node_02 = 主旋律、node_03/04 = 輪唱、node_05 = ベース・リズム伴奏 という割当が明記されており、議論のたたき台としては機能する。
- ADR-0004 の「主旋律 3 + リズム 1」「輪唱曲」という制約は担っている。

### 2.3 課題曲を「カエルの歌」で具体化

- 抽象論で終わらず、`{60, 62, 64, 65, 64, 62, 60, -1}`（ドレミファミレド休）というデータ例まで踏み込んでいる。
- 議論を進めるサンプル素材として有効。

## 3. 改善・確認すべき点

### 3.1 【重要】塩澤 PDF の楽譜データ仕様と完全に乖離している

- `arduino_塩澤.pdf` §2.4.3 では同じ楽譜データを以下の構造で詳細定義済み:
  ```
  ScoreEvent { beatAt(uint16_t), noteNumber(uint8_t),
               velocity(uint8_t), durationQ8(uint16_t), flags(uint8_t) }
  kScore[]; kScoreLength;
  ```
  - `durationQ8` は **拍の 1/256 単位の固定小数**（256 = 1 拍）
  - 輪唱の入り遅れは各ノードの `startBeatNo`（0/4/8/0）で表現
  - ノード間差分は `score_data.h` のみ
- 本書 §2.1 の `int score[][2] = { {60, 1.0}, ... }` は:
  1. **型エラー**: `int` 配列に `1.0`（float）を入れている。実際にコンパイルすると `1` に切り捨てられる。本気で float 拍数を持つなら `float score[][2]` か構造体定義が必要。
  2. **休符の表現が揺れている**: 表 §2.1.3 では「-1」、本文 §2.1.2 では「0 や -1 などの特定の数値」と曖昧。塩澤側は `noteNumber == 0` を休符としており、決め打ちが必要。
  3. **輪唱の入りタイミングが未定義**: 本書 §2.1.1 で「輪唱パート 1/2」と書いたまま、各パートが何拍遅れて入るかの仕様がない。塩澤 PDF 表 2.5 では `startBeatNo = 0/4/8/0` と確定済み。
- **第3回ミーティングで楽譜データ仕様を 1 案に集約する必要**。 担当者は齋藤なので、塩澤の `ScoreEvent` をそのまま採用するか、自案を採用させて塩澤の applyPattern() / OrcProtocol を寄せさせるかを決める。

### 3.2 【重要】Arduino → PC 通信仕様が塩澤 PDF と矛盾

- 本書 §2.2 は「**カンマ区切りなどで送信**」と記述。
- 塩澤 PDF §2.3.3 では **20 バイト固定長のバイナリパケット**（`magic 0x4F52`／`type=3 NOTE`／`partId`／`noteNumber`／`velocity`／`gate`／`durationMs`）と既に確定。
- カンマ区切りテキストとバイナリ固定長は両立しない。**WBS 130 は齋藤の担当**なので、齋藤側がバイナリ仕様に同意するか、再交渉するかを明確に。
- なお梅澤 PDF §2.1.1 では「通信フォーマットの詳細は 115 と 130 が確定してから決める」と明示しており、今が決め時。

### 3.3 通信内容に欠けている要素

- 本書 §2.2「送る内容」は **「楽器 ID／Pitch／Velocity」の 3 要素** のみ。塩澤 PDF の NOTE ペイロード（`partId`／`noteNumber`／`velocity`／`gate`／`durationMs`／予約）と比較すると以下が抜けている:
  - **発音停止の指定方法** （`gate` ビットによる NoteOn/NoteOff の区別、または `durationMs`）
  - **Note OFF タイミング** の所在（本書では「PC 側で行うか別途送る」と保留）
  - **シーケンス番号**（パケロス検出用）
- 「PC 側で発音長を管理する」とした場合、テンポ変更時に PC が Arduino 側のテンポを把握する経路が必要（CTRL パケット転送など）。この設計は本書には書かれていない。

### 3.4 課題曲が WBS 241 として未着手

- 本書は「カエルの歌」を例として使っているが、`meetings/0422_2回/wbs_table/wbs_table.md` の WBS **241「課題曲 4 パート選定」は齋藤担当の未着手タスク**。
- 「カエルの歌」を正式採用するなら、第3回でチーム合意 → ADR 追加（または既存 ADR-0004 の改訂）が必要。
- 主旋律 7 音 + 休符 1 拍程度の長さでは演奏時間が短すぎる可能性もあるため、輪唱 4 声で何小節演奏するかも併せて決定したい。

### 3.5 ADR-0004 との楽器構成の不一致

- ADR-0004 は「**金管楽器**」を採用、本書 §2.1.1 のパート割当は「主旋律／輪唱／ベース・リズム伴奏」と楽器名（金管・ドラム）を意識していない。
- 塩澤 PDF は「金管 1 / 金管 2 / 金管 3 / ドラム」と明示。
- 本書のパート名は「役割名」、塩澤は「楽器名」で表現しているが、両者の対応関係を明示しないと、梅澤の音色合成（4 パート全て金管想定）と擦り合わない。

### 3.6 ハードウェア（Arduino）側のメモリ実装方針の欠落

- 本書 §1「Arduino の限られたメモリ容量を圧迫しないよう」と書いているが、具体的な対策（PROGMEM / `const PROGMEM` / Flash 配置）への言及がない。
- WBS 242「4 パート楽譜のヘッダ配列化（**PROGMEM 形式**）」が齋藤担当として既に書かれているので、基本設計段階でも触れておくのが望ましい。
- 塩澤 PDF §2.3.1 のファイル構成では `score_data.h` で前方宣言・`score_data.cpp` で本体定義としており、`PROGMEM` の有無は言及なし。ESP32-S3 / RA4M1 のフラッシュ容量を踏まえた決定が必要。

### 3.7 図 1 のフローが粗い

- §2.2 の処理フロー図は「待機 → 読み込み → 計算 → 待機 → 休符判定 → パケット → 送信 → ループ」だけ。
- 以下の分岐が抜けている:
  - **テンポ変更を受信したときの処理**（指揮者からの CTRL を待って待機時間を再計算するか）
  - **輪唱の入りタイミング待ち**（曲頭から N 拍待ってから演奏開始）
  - **曲が終わったときの処理**（ループするか・停止するか）
  - **指揮者ノードからの BEAT 信号との同期**（時刻基準）

### 3.8 細かい指摘

- §1 末尾「指揮棒の入力による動的なテンポ変更にも柔軟に対応可能とする」は良いが、その「対応」を本書の処理フロー §2.2 で示せていない。基本設計の方針 → 詳細設計の手順 へつながっていない。
- §2.1.2「真ん中のド = 60」は MIDI 規格と整合（C4 = 60）。良い。
- §2.1.4 のコード例：1 行目 `int score[][2] = {` は前述の通り型ミス。`struct Note { uint8_t pitch; uint8_t durationQ8; };` のような構造体定義のほうが拡張性が高い。

## 5. 推奨アクション（第3回ミーティングで決めたいこと）

1. **楽譜データ構造の一本化**（最優先）。 塩澤 PDF §2.4.3 の `ScoreEvent`（`durationQ8`、`startBeatNo`）を採用するか、本書の {pitch, duration_beats} 形式を採用するか、変換層を挟むかを決める。
2. **Arduino → PC 通信フォーマットの一本化**（WBS 115/130）。バイナリ／テキストどちらにするか、塩澤 PDF §2.3.3 のパケット定義を採用するか。
3. **課題曲の正式選定**（WBS 241）。「カエルの歌」を正式採用するか、より演奏時間の長い曲（例: 「もみじ」「ぶんぶんぶん」「メリーさんのひつじ」など輪唱可能な小品）を探すか。
4. **輪唱の入り拍の確定**。各パートが何拍遅れて入るかを本書 §2.1.1 に明示する（塩澤側の 0/4/8/0 を採用するか、別の値にするか）。
5. **次バージョンでの追記項目**:
   - PROGMEM 等のメモリ配置方針
   - 休符の特殊値の確定（推奨: `noteNumber == 0`、塩澤側と揃える）
   - テンポ変更受信時の処理フロー
   - 輪唱パートの入り拍待ち処理

## 6. 総合所見

「たたき台」というタイトル通り、議論を始めるに足る素材は揃っている。ただし、塩澤 PDF が同じ領域（楽譜データ詳細・通信仕様）に深く踏み込んだ詳細設計を出してきているため、本書のままだと塩澤側の設計に押し負けて、齋藤の WBS 116/128/129/130 の成果物が形骸化するリスクがある。

第3回ミーティングでは、楽譜担当として **「自分の案を残すか／塩澤案に乗るか／変換層で両立するか」を齋藤自身がスタンスを取る** ことが重要。齋藤案を残す場合は、以下の弱点（型エラー・休符の揺れ・輪唱入り拍の欠落・通信仕様の粗さ）を次バージョンで解消する必要がある。
