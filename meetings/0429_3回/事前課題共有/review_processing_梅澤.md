## 2. 良かった点

### 2.1 倍音とエンベロープの方針が明確

- §1.1.3 表 1 の倍音構成（基音 = 1.00 / 2 倍音 = 0.70 / 3 倍音 = 0.50 / ... / 7 倍音 = 0.05）は金管の倍音特性として妥当な初期値。
- §1.1.4 の Attack 20 ms / Release 60 ms は金管楽器のエンベロープとして概ね妥当（実際のトランペット等は Attack 10〜30 ms 程度、Release は奏法による）。
- 「Minim の Line では 1 本の直線変化しか持てないため、音の出来次第で ADSR クラスを使う」という代替案を持っているのも実装リスクへの備えとして良い。

### 2.2 既存授業資料の活用

- `WavetableGenerator.gen10()` と `HackInstrument` という授業で習得済みのクラスを起点にしているため、実装着手のハードルが低い。
- `out.playNote()` までの呼び出し構造を意識しており、最小実装で動く絵が描けている。

### 2.3 FFT 解析（WBS 127）への接続

- §2.4 で「`scipy.fft` または `librosa` で実音をフーリエ解析 → 倍音比率を §1.1.3 表 1 に反映、立ち上がり時間を `attackTime` に反映」と、現在の暫定値を実データで上書きする経路が示されている。
- ADR-0007 の「**倍音データなどを元に根拠を持った音色作り**」を成果物範囲とする要件と直接整合。

### 2.4 タスク間の依存を理解している

- §2.1.1 で「通信フォーマットの詳細は 115 と 130 が確定してから決める。ここでは大まかな流れのみ示す」と明示しており、自分の担当範囲と前提タスクの境界を意識できている。

## 3. 改善・確認すべき点

### 3.1 【重要】楽器構成が他メンバーの想定と乖離

- 本書 §2.2 で「4 つの楽器パート（**現段階では金管楽器**）をそれぞれ独立して鳴らせるように管理」とあり、4 パート全て金管想定。
- 一方:
  - `arduino_塩澤.pdf` §1.2.1 では **「金管 3 + ドラム 1」**
  - `music_齋藤.pdf` §2.1.1 では **「主旋律／輪唱 1／輪唱 2／ベース・リズム伴奏」**
- 特に **node_05 のドラム（打楽器）を金管音色で鳴らすと表現が破綻** するため、第3回でドラム音色（パーカッション系：ノイズ + 急速減衰のエンベロープ）を追加実装するか、それとも全パート金管で進めるかを合意する必要がある。
- ADR-0004 は「金管楽器」を採用としつつ、授業制約「主旋律 3 + リズム 1」を満たす構成として塩澤案（金管 3 + ドラム 1）が出ている状態。整合の取り方を全員で握る必要あり。

### 3.2 【重要】「4 パート発音管理」（WBS 125）の記述が 1 段落のみ

- §2.2 全文が「授業で習った melody・duration・startTime のような配列を使って音の情報を管理」「パートごとの音色（Waveform）を配列で管理」の 2 文だけ。
- ルーブリックの A 要件（クラス設計・関数設計・データベース設計など実装に必要な設計が具体的）に対して粒度が不足。次バージョンで以下を補う必要がある:
  - **ボイス（同時発音オブジェクト）のクラス／配列構造**
  - **パート ID → 発音枠マッピング**（`partId 0x02-0x05` をどの Waveform / HackInstrument にバインドするか）
  - **同時発音数（ポリフォニー）の上限**と、上限超過時の挙動（古い音を切るか／新しい音を捨てるか）
  - **NoteOff の管理**（duration が経過した時点で停止する処理、または Arduino からの明示的 NoteOff 受信処理）
- WBS 253「4 パート同時発音管理（ボイスマネージャ）」が梅澤の製造フェーズタスクとして既にあるので、設計段階でも形を持たせておきたい。

### 3.3 【重要】Mac × 4 台構成（塩澤 PDF）が前提になっていない可能性

- 塩澤 PDF §1.2.1 と §2.2.2 では **「各 Arduino UNO R4 WiFi が専用 Mac へ 1 対 1 で USB Serial 直結。Mac 側 Processing は対応する 1 ポートだけを開いて受信」** という構成を採っている。
- 一方、本書 §2.1.2 の「Processing の `serialEvent()` 関数が自動で呼ばれ、データを受け取る」という記述は **PC 1 台 ですべての Arduino からの NOTE をマージして受信** する旧構成（ADR-0004 / `docs/design/architecture.md`）にも適合する。
- 仮に塩澤案（Mac × 4 台、各 Mac は 1 パート専属）が採用されると、本書 §2.2「4 パート発音管理」の意味が変わる:
  - 1 台の Processing が 1 パートだけ鳴らす設計に縮小される
  - 4 パート同時発音管理は不要になり、各 Mac はモノフォニックでよい
  - 代わりに「4 台の Mac 上の Processing が時刻ずれなしに発音する」という別問題が浮上
- **第3回で PC 構成（1 台 vs 4 台）を確定させてから本書の §2.2 を書き直すかどうかを判断**。

### 3.4 NOTE パケットの中身が梅澤側で未理解

- §2.1.1 で「通信フォーマットの詳細は 115 と 130 が確定してから決める」と保留しているが、**塩澤 PDF §2.3.3.3 で既に NOTE ペイロード仕様は確定**:
  ```
  partId(1B) / noteNumber(1B) / velocity(1B) / gate(1B) /
  durationMs(2B) / 予約(2B)  // 共通ヘッダ 12B + ペイロード 8B = 20B
  ```
  - 「音名」=`noteNumber`（MIDI ノート番号）
  - 「長さ」=`durationMs`（参考値）
  - 「振幅」=`velocity`（0-127）
  - 加えて `gate` が 1=NoteOn / 0=NoteOff の区別を持つ
- 梅澤 PDF §2.1.2 の「音名・長さ・振幅を取り出す」と概念は一致するが、**バイナリパケットからの同期マーク `magic = 0x4F52` 検出と type 判別** という前処理が必要なことが本書には書かれていない。
- 次バージョンでは以下の処理を `serialEvent()` 内のフローとして明記する:
  1. バイト列を内部バッファに溜める
  2. `magic 0x4F52`（リトルエンディアン: `0x52 0x4F`）を探してフレーム同期
  3. `type` を読み、`type == 3` (NOTE) のときペイロード 8 B を読み取り
  4. シーケンス番号で重複排除（`seq` フィールド）
  5. `partId` で発音枠を選び、HackInstrument に渡す

### 3.5 ドラム音色の不在

- 塩澤 PDF が node_05 = ドラムを採用すると、本書のエンベロープ（Attack 20 ms + Release 60 ms）と倍音構成（金管系倍音）はドラム音色には適合しない。
- ドラムは典型的に:
  - 倍音は非整数倍を多く含む（ノイズ + 数本の共鳴ピーク）
  - Attack < 5 ms、Release はキック/スネア/ハイハットで大きく異なる
- 第3回で「ドラムは金管 4 構成に変えてしまう」「ドラムは別音色を作る（梅澤の追加タスク）」「ドラムは音源データを再生する（サンプラ方式）」のどれにするか合意必要。

### 3.6 §2.3.3 エンベロープの設計が duration 全体に張り付いている

- §2.3.3「`ampEnv.activate(duration, maxAmp, 0)` で **音符の長さ全体をかけて最大音量からゼロへ直線的に減衰**」とあるが、これは:
  - 鳴り始め（Attack）が瞬時に最大音量
  - 持続中ずっと音量が線形減衰
  - Release（息を止めた後の減衰）と Sustain（持続）の区別なし
  - という非常に単純化された ADSR 近似
- 結果として全ての音符が「叩いた音 → だんだん消える」という打楽器寄りの音色になる。金管らしさ（Sustain 中もエネルギーを持ち続ける）は出ない可能性大。
- §1.1.4 で言及した本来の ADSR（Attack 20 ms / Decay → Sustain / Release 60 ms）を §2.3.3 で具体化すべき。「音の出来次第で ADSR クラスを使う」ではなく、最初から ADSR クラス前提で書いたほうが品質が上がる。

### 3.7 細かい指摘

- §2.3.2 Listing 1 のコメントが LaTeX レンダリング崩れ：「**第倍音（基音）：最大1**」「**第倍音：金管は倍音が強い22**」「**第倍音3**」など、数字が末尾にずれている。意図は「第 1 倍音 (基音): 最大」「第 2 倍音: 金管は 2 倍音が強い」「第 3 倍音」のはず。`listings` のリスト内日本語文字幅の問題。次バージョンでは `plistings` パッケージを使うか、コメントを英語にすると安全。
- §2.4.2 手順 5「`makeBrassWaveform()` の配列と `attackTime` に反映」だが、`attackTime` の宣言が本書中になく、どこに保持される変数なのかが未定義。HackInstrument のメンバ変数として追加するなら明示が必要。
- §2.1.2 手順 4「HackInstrument を生成し、`out.playNote()` で音を鳴らす」 — 1 音ごとに HackInstrument を生成すると GC 負荷・レイテンシが心配。プールから再利用する設計（ボイスマネージャ）に言及があるとよい。

## 5. 推奨アクション（第3回ミーティングで決めたいこと）

1. **PC 構成の確定**（1 台 vs Mac × 4 台）。本書の §2.2「4 パート発音管理」の設計方針が大きく変わるため、まずここを決めたい。
2. **楽器構成の確定**（金管 4 vs 金管 3 + ドラム 1）。ドラム採用なら音色実装タスク（梅澤）を 1 つ追加する。
3. **NOTE 受信処理の詳細化**（塩澤 PDF §2.3.3 のバイナリパケットを前提にした `serialEvent()` 内のフロー設計）。WBS 124 を次バージョンで具体化。
4. **ボイスマネージャの設計**（WBS 125/253）。クラス・配列・同時発音数・NoteOff 管理を明文化。
5. **ADSR クラス採用の判断**。`ampEnv.activate()` の 1 直線エンベロープは金管らしさが出にくいため、最初から ADSR 前提で詳細設計を書いたほうが手戻りが少ない。
6. **§2.3.2 Listing 1 のコメントレンダリング修正**（次バージョンで `plistings` または英語コメント）。

## 6. 総合所見

音色合成の **方針** は明確で、倍音テーブルと ADSR の値も具体的。FFT 解析（WBS 127）で実データを取得 → 暫定値を上書きするという PDCA も設計に組み込まれており、ADR-0007 の「根拠を持った音色作り」要件は満たせる見込み。

ただし、**「4 パート発音管理」「NOTE 受信処理」の 2 つは設計粒度が概念レベルにとどまっており**、実装フェーズに入るには WBS 125/253/124/251 の詳細を次バージョンで補う必要がある。特に PC 構成（1 台 vs 4 台）と楽器構成（金管 4 vs 金管 3 + ドラム 1）は、決まらないと本書の §2.2 が書き直しになるため、第3回で必ず合意したい論点。
