## 2. 良かった点

### 2.1 ルーブリック A 要件をほぼ全面で満たす記述粒度

- §1.1.1 の FBS（F1 指揮情報抽出 〜 F5 共通基盤）が機能ID付きで列挙され、§1.1.2 の MOP 表（MOP-1 〜 MOP-8）と紐付いている。
- §1.4 の状態遷移図と表 1.6／1.7 で、各状態の遷移条件と動作が一意に決まる。
- §2.3.3 の `OrcProtocol` パケット定義はオフセット・型・意味すべて表で示され、受信実装が即座に書ける粒度。

### 2.2 設計上の判断根拠を明示している

- §2.2.2「採用判断」で UNO R4 WiFi の WiFiS3 が AT bridge 経由のため SoftAP を node_01 から ESP32-S3 直制御の XIAO に置換した理由を明記。
- §2.3.3.6「マスタクロック方式と時刻同期」で、4 台の WiFi ジッタを発音タイミングに直接乗せず、時計オフセット推定の系統誤差に押し付ける設計判断を数式付きで記述。
- §2.5.4 の「実機で確定する仮値」テーブルで、現時点の数値が暫定であることを正直に明示し、実機で握る基準（T2/T3 など）を併記。

### 2.3 EMA 準拠の徹底

- 全モジュールが `IModule` を実装、`SystemData` 経由でしか結合しない、`ModuleTimer` で周期判定、`{Module}Config`／`{MODULE}_CONFIG` の命名規則まで一貫。
- ADR-0005「EMA 準拠」と整合しており、リポジトリの `firmware/README.md` のコーディング方針とも齟齬がない。

## 3. 改善・確認すべき点

### 3.1 【重要】MOE と MOP-1 の数値が乖離している（同期誤差）

- ADR-0006（`docs/decisions/0006-sync-error-moe-20ms.md`）で **MOE = 同期誤差 20 ms 以内** と確定済み。
- 本書 §1.1.2 の MOP-1 は **「楽器間 同期誤差 ≤ 30 ms」**、§2.5.2 の TP-3 合否基準も「最大差 ≤ 30 ms（マスタクロック方式により実機では 5 ms 以下を目標）」となっている。
- MOE 20 ms より緩い MOP は通常許容されない。「実機目標 5 ms」を持っているなら **MOP-1 を 20 ms に揃え、TP-3 の合否基準も 20 ms** に合わせるか、もしくは **「実装目標／合否基準／MOE」の三層構造を本書冒頭で明記** すべき。
- 現状のままだと、MOP-1 を満たしてもプロジェクトの MOE を満たさない逆転が起こり得る。

### 3.2 【重要】楽譜データ仕様が齋藤担当（WBS 116/128/129/130）と完全に二重定義になっている

- 本書 §2.4.3 で `ScoreEvent`（`beatAt`／`noteNumber`／`velocity`／`durationQ8`）と `kScore[]`、各ノードの `startBeatNo = 0/4/8/0` まで具体定義している。
- 同日提出の `music_齋藤.pdf` では同じ楽譜データを `int score[][2] = {{60, 1.0}, ...}`（音高 + 拍数 float）として全く別形式で定義しており、両者は互換性がない。
- WBS では 116/128/129/130 が齋藤担当。本書 §1 末で「楽譜担当との合意で確定」と注記してあるのは良いが、**実際には齋藤の事前課題と整合していない** ため、第3回ミーティングで以下のいずれかに決着させる必要がある:
  1. 塩澤の `ScoreEvent`（`durationQ8`、輪唱は `startBeatNo` で実現）に齋藤側を寄せる
  2. 齋藤の {pitch, duration_beats} 形式に塩澤の `applyPattern()`／`OrcProtocol` を寄せる
  3. 双方を残し、ヘッダ→中間表現→ScoreEvent への変換層を齋藤が担当する
- 越境して書くこと自体は問題ないが、**本書冒頭で「楽譜担当の最終仕様で上書きされる暫定」と明記** するか、事前に齋藤と擦り合わせて整合した一案として提出すべきだった。

### 3.3 【重要】PC 構成が ADR-0004 から変わっている（合意必要）

- ADR-0004 では「**Processing（PC）側**で音色合成・再生」と単数形で記載され、`docs/design/architecture.md` のブロック図も PC 1 台。
- 本書 §1.2 では **「Mac × 4 台、各楽器ノード ↔ Mac は 1 対 1 で USB Serial 直結」** という新構成を採っている。
- メリット（パケロス・ネットワーク輻輳の排除）は理解できるが、**ADR-0004 の改訂か新 ADR の追加が必要**。第3回での合意 → ADR 化を経ずにこの前提で詳細設計を進めるとリスク（梅澤側の Processing 仕様、片岡らの発表時の機材構成にも波及）。
- 特に Mac 4 台の手配・接続性検証は地曵／片岡／御代川のリソースに直接効く。早期合意が望ましい。

### 3.4 ハードウェア設計に「回路図」が無い

- §2.2.1 のピンアサイン表は明確だが、ルーブリック A は「回路図などのシステムの構成要素の設計図」を要求。
- I2C 2 本＋電源＋LED 構成なので簡素ではあるが、**MPU6050 と XIAO ESP32-S3 Sense の結線図（Fritzing 等で 1 枚）** を追加すれば疑義が消える。
- USB ハブ経由の給電と、各 Arduino UNO R4 WiFi → Mac の USB 接続トポロジも図で示すと、§2.2.3 の物理配置と合わせて A 評価が確実になる。

### 3.5 細かい指摘

- §2.5.4 表 2.10 の「拍検出閾値（MPU6050）」`仮 1.80 g`：腕の自然な振りは加速度ノルムで 1.5 〜 3.0 g 程度になるため概ね妥当だが、§2.4.2.4 の「強い振りでないと検出しない」欠点は、本プロジェクトの目的「**指揮の振り方が乱雑でもテンポを検出可能**」（ADR-0007）と弱い緊張関係にある。鉛直成分ゼロクロス方式へのフォールバック条件を §1.1.2 の MOP-3（拍検出 正解率 ≥ 90%）と紐付けて明記しておくと安全。
- §2.3.3.3 共通ヘッダ `magic = 0x4F52` は ASCII "OR"（リトルエンディアンで `0x52 0x4F` → "RO"）。「ASCII で見たときの並び順」を明示しておくと、Mac 側の Processing 実装者（梅澤）が同期マークを探すコードを書く際に迷わない。
- §2.3.3.7 ログ書式 `[ModuleName] msg` は良い規約。`#ifdef DEBUG` でビルド分岐するなら、`platformio.ini` の `build_flags = -DDEBUG` 例も付録に置くと共有しやすい。

## 5. 推奨アクション（第3回ミーティングで決めたいこと）

1. **MOP-1 / TP-3 の同期誤差基準を 20 ms に揃える**（または三層基準を明文化）。 ADR-0006 と本書のどちらを優先するかを最初に握る。
2. **「PC = Mac × 4 台、1 対 1 接続」の採否決定 → ADR 追加**（仮: ADR-0008）。地曵・片岡・御代川の機材手配にも関わるため最優先。
3. **楽譜データ仕様の一本化**（齋藤と塩澤で 1 案に集約）。本書 §2.4.3 と齋藤 PDF §2 のどちらをベースにするか、または変換層を挟むかを決定。
4. **回路図 1 枚の追加**（§2.2.1 のピン表をベースに Fritzing 等で 1 枚作成）。次バージョンでルーブリック A 評価を確実化。
5. **本書 §2.5.4 の「実機で確定する仮値」のうち、第3回時点で暫定確定可能なもの**（WiFi チャネル、SoftAP マルチキャスト到達性方針など）を絞り込み、未確定リストを縮小。

## 6. 総合所見

担当範囲を大きく超えて、システム全体が手戻り少なく実装フェーズに入れる粒度まで書き込まれている。ルーブリックの A 要件はほぼ全項目で満たしている。

ただし「越境して書いた部分」（楽譜データ詳細・PC 構成変更・MOP 数値設定）は、**他メンバーの担当成果物や既決 ADR と整合していない**。技術的な完成度の高さと引き換えに、合意形成プロセスをスキップしたリスクを抱える。第3回でこれらを「合意」に格上げできれば、本書はそのままチームの基礎設計書として機能する。逆にここで合意が取れないと、塩澤と他メンバーが別仕様で並行実装して結合フェーズで衝突する危険がある。
