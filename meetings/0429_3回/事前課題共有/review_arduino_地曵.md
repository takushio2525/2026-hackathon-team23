## 1. ルーブリック評価サマリ

本書のタイトルは「ハッカソン 〜計画書と設計書〜」だが、実際の本文は **第1章 基本設計／第2章 詳細設計** の二部構成で、マネジメント計画書としての必須要素（題目・目的・独自性・参考文献／成果物の対象範囲／FBS・PBS／WBS・スケジュール・役割分担／MOP・TPM）が **ほぼ全項目で欠落** している。指定された評価基準（マネジメント計画書ルーブリック）に対しては、形式・記述ともに不足が大きい。

| 評価項目 | 配点 | 評価 | 根拠 |
|---|---|---|---|
| プロジェクトの概要（題目・目的・独自性・参考文献） | 3 | **D (0)** | 題目「ハッカソン 〜計画書と設計書〜」は成果物を表すキーワードを含まず汎用。プロジェクトの目的・既存技術との比較・独自性・参考文献の記述が一切ない |
| 成果物の対象範囲と機能（成果物・対象範囲／非対象範囲・FBS・PBS・対応関係） | 3 | **D (0)** | 成果物の定義・対象範囲／非対象範囲・FBS／PBS・対応関係いずれの記述もない。§1.1 の機能一覧は 5 項目あるが、機能 ID 付与もなく PBS との対応も取られていない |
| 作業計画と役割分担（WBS・8/80 ルール・アローダイアグラム・クリティカルパス・スケジュール） | 3 | **D (0)** | WBS・アローダイアグラム・クリティカルパス・スケジュール・役割分担いずれの記述もない |

## 2. 良かった点

### 2.1 「予約実行型 UDP」と「マスタクロック方式」の発想

- §1.1 で **「確定した BPM 情報と実行予定時刻 (T_exec) を含む独自プロトコルのパケットを生成」**、§2.4.3 で **「指揮棒が『現在時刻 + 2000ms』を実行時刻としたパケットを送信 → 楽器側は受信後キューに保存 → if (myTime >= T_exec) となった瞬間に発音指示を送出」** という方式を提案している。
- これは塩澤 PDF §2.3.3.6 の「マスタクロック方式」（マスタが拍時刻を t = lookahead 先で配信、各楽器がローカル時計で発音）と思想的に同じで、**WiFi ジッタを発音タイミングに直接乗せず、時計オフセット推定の系統誤差に押し付ける** 設計判断としては妥当。
- §2.3.3 で `EventQueue` クラス（予約時刻付きで命令を保持し指定時刻に実行）まで踏み込んでおり、実装に近い粒度。

### 2.2 同期維持（Sync-Loss）状態を明示している

- §1.4 の状態遷移図に **5. Sync-Loss（一時的な通信途絶時、直前の BPM と自身の内部時計に基づき自立的に演奏を継続するフェイルセーフ）** を独立した状態として置いた点は良い。
- `docs/decisions/0006-sync-error-moe-20ms.md` の「同期誤差 ≤ 20 ms」を、瞬断時にも維持するための運用設計として位置づけられる。
- 他メンバーの提出物（塩澤 / 齋藤 / 梅澤）には Sync-Loss 状態の明示がなく、本書独自の貢献。

### 2.3 ファイル構成と関数設計の粒度

- §2.3.1 の `Conductor_Main.ino` / `Instrument_Main.ino` / `SyncManager.h/cpp` / `ProtocolDefinition.h` / `SoundEngine.pde` の分割は妥当で、`firmware/common/lib/`（EMA 共通層）に乗せる前提なら最小構成として機能する。
- §2.3.3 の関数シグネチャ（`float calculateBPM()` / `int quantizeTempo(float rawBPM)` / `void sendRedundantPacket(Packet p)` / `void checkAndExecute()`）は引数・戻り値型まで型付けされており、実装着手のハードルは低い。

## 3. 改善・確認すべき点

### 3.1 【重要】マネジメント計画書としての必須要素がほぼ全項目で欠落している

本書は表紙のサブタイトルに「計画書と設計書」と銘打っているが、実体は **設計書のみ**。マネジメント計画書ルーブリックが要求する以下の項目が一切記述されていない:

| ルーブリック要求項目 | 本書の記述 |
|---|---|
| プロジェクトの題目（成果物のキーワードを含む） | × タイトルが汎用 |
| プロジェクトの目的・最終ゴール・解決すべき課題 | × |
| 既存の技術 / 作成するシステムの独自性 | × |
| 参考文献 | × |
| 成果物の対象範囲・非対象範囲 | × |
| FBS / PBS / 両者の対応関係 | × |
| WBS / 8/80 ルールに沿った分解 | × |
| アローダイアグラム・クリティカルパス | × |
| 役割分担・スケジュール | × |
| MOE / MOP / TPM | × |
| 単体／結合／システムテスト計画 | × |
| 資源・調達計画（Mac × 4 台等の調達） | × |
| リスク管理 | × |

第3回ミーティングまでに、`work/shiozawa/work-0422/提出用計画書/plan_template.tex` のテンプレートに沿った各章を新規作成する必要がある。
特に **24G 勢（片岡・地曵・御代川）の役割は WBS 上「議事録持ち回り＋ 300 テストフェーズ以降の実機検証・発表準備への合流」と整理済み** であり、地曵自身がこの位置づけのままで良いのか、それとも設計フェーズに踏み込み直すのかを第3回で握る必要がある。

### 3.2 【重要】120/140 BPM の 2 値ヒステリシスが ADR-0007 のゴールと乖離している

- §1.1「テンポ量子化・ヒステリシス制御機能」で **「算出した BPM を音楽的な標準テンポにマッピングする。130 BPM（上昇時）および 125 BPM（下降時）の閾値を用いたヒステリシス処理」** と記述。
- §1.4 状態遷移図 4. Playing でも **「120 BPM で演奏中、計測値が 130 BPM を超えたら 140 BPM へ遷移パケットを送信」** と、演奏テンポを **120 / 140 の 2 値** に量子化する設計。
- これは ADR-0007「ゴール」**「ユーザーが指揮を振るテンポを変えたときに演奏速度も変えられる、簡単な演奏体験」** と整合しない。チームのゴールは「振りに応じてテンポが連続的に追従する」体験であり、塩澤 PDF も連続値 BPM を `applyPattern()` で適用している。
- なぜ 2 値量子化なのかの根拠（指揮の判定が荒い人向けの安定化策？／演奏体験の演出？）が本書には書かれておらず、設計判断の理由を §1 末尾あたりに 1 段落で添えるか、第3回で「ゴールを 2 値量子化に変える」「連続値に揃える」のいずれかに揃える必要がある。
- 仮に 2 値を残すなら、TP-4「テンポ追従（80→120 BPM 階段変化）」（`wbs_table.md` 334）との整合（テンポは 2 値ではなく階段変化を測る）も再設計が必要。

### 3.3 【重要】PTP-lite 方式が塩澤 PDF と二重定義になっている

- §1.1「時刻同期・キャリブレーション機能：演奏開始前に指揮棒と各楽器間の内部時刻を **PTP-lite 方式** で同期し、ネットワーク遅延に伴うオフセットを解消する機能」。
- §2.4.1「時刻同期フロー：指揮棒が SYNC_REQ をブロードキャスト → 各楽器が受信時刻と共に ACK 返信 → 指揮棒が往復遅延を計算し補正値を返信 → 各楽器が内部時計（millis()）にオフセットを適用」。
- 塩澤 PDF §2.3.3.6 にも同じ思想で時刻同期方式が定義されており（マスタクロック offset 推定）、**両者は別仕様の二重定義**。担当範囲（WBS 114/115/121/122/123 が塩澤、24G 勢は議事録・テスト合流）から見ると、地曵側は塩澤の時刻同期仕様を **採用 or 補強する立場** が自然。
- 第3回で「時刻同期は塩澤の仕様に乗る」「地曵は Sync-Loss 状態のフェイルセーフ部分だけを担当する」など分担を再整理したい。

### 3.4 【重要】PC 構成変更（Mac × 4 台）が ADR を経ずに前提化されている

- §1.2 ハードウェア構成で **「音響出力・処理：ノート PC（Mac）× 4 台」「各楽器 Arduino と Mac は USB シリアル接続」** が前提として書かれている。
- これは塩澤 PDF §1.2.1 と同じ構成だが、ADR-0004 / `docs/design/architecture.md` は **「PC 1 台」** で確定済みであり、本書時点で **新 ADR は発行されていない**。
- メリット（パケロス・ネットワーク輻輳の排除）は理解できるが、地曵 PDF と塩澤 PDF が共に「Mac × 4 台」前提で書かれることで「合意済みの既決事項」のように見える状態は、議事録上の合意プロセスをスキップしている。第3回で **正式に新 ADR-0008（仮）を起票し、Mac × 4 台の調達計画（誰が貸し出すか）まで決める** 必要がある。

### 3.5 §1.1 の独自プロトコルが §2 で定義されない

- §1.1「予約実行型 UDP 通信機能」と書きつつ、§2.3.1 の `ProtocolDefinition.h` の中身（パケット構造体・byte レイアウト・magic 値）が §2 内で一切示されない。塩澤 PDF §2.3.3.3 の `OrcProtocol`（共通ヘッダ 12 B + ペイロード 8 B = 20 B 固定長、`magic = 0x4F52`）レベルの記述粒度が必要。
- 「同一 UDP パケットを短間隔で 3 回送信」（§2.3.3）の冗長送出間隔（何 ms 間隔か）と重複排除方法（シーケンス番号）も未定義。

### 3.6 §2.5「テストの設計」が目次のみ

- 目次（p.4）に §2.5「テストの設計」「2.6 AI の利用について」が 11 ページ目として記載されているが、本文は §2.4「処理フローの設計」の次が **§2.5 ではなく §2.5「AI の利用について」** になっており（番号も §2.5 で重複）、テストの設計の本文が存在しない。これは脱稿漏れと推測される。
- 本来は単体（拍検出精度・量子化挙動・キュー実行精度）／結合（マスタ−スレーブ間時刻同期誤差）／システム（`wbs_table.md` 331-337 の TP-1〜TP-7）まで記述する必要がある。

### 3.7 機能一覧（§1.1）に機能 ID が無く FBS と接続しない

- §1.1 は 5 機能（時刻同期・指揮動作解析・テンポ量子化・予約実行型 UDP・分散音響合成）を列挙しているが、機能 ID（F1〜F5 等）が無いため、後段の MOP / テスト計画 / 部品 PBS との対応関係が引けない。
- 塩澤 PDF §1.1.1 のように **「F1: 指揮情報抽出 / F2: 同期配信 / F3: 楽譜進行 / F4: 音色合成 / F5: 共通基盤」** のように ID 付与しておくと、ルーブリックの「成果物の対象範囲と機能」評価が C 以上に上がる。

### 3.8 細かい指摘

- §2.2「全 Arduino は共通の SSID を持つ WiFi アクセスポイントに接続し、固定 IP アドレスを割り当てる」 — 塩澤 PDF §2.2.2 では UNO R4 WiFi の WiFiS3 が AT bridge 経由で SoftAP の安定性に難があるため node_01 を XIAO ESP32-S3 Sense に置き換える方針。本書では UNO R4 WiFi のまま。第3回で機材構成を揃える必要。
- §2.1 部品一覧「6 軸加速度・ジャイロセンサ（**BMI270** 等）」 — 塩澤 PDF / リポジトリの ADR は **MPU6050** を採用。BMI270 は新提案だが採用根拠（精度・単価・在庫）の記述なし。
- §2.3.1 `SoundEngine.pde` — Processing のメインスケッチ拡張子は `.pde` が正しいが、Mac × 4 台構成だと音色合成ファイルは梅澤担当（WBS 117/124/125/126/251/252/253/254）。本書のファイル構成は梅澤の担当成果物に踏み込んでいる。
- §1.3 操作インターフェース「LED マトリクス（120/140 等）の表示」 — 量子化テンポを LED マトリクスで数字表示する案だが、UNO R4 WiFi 内蔵 LED マトリクスは 12×8 ピクセルのため、3 桁数値は読みづらい。スクロール表示か、状態色（120 = 青／140 = 赤）に倒すかの設計判断が要る。

## 5. 推奨アクション（第3回ミーティングで決めたいこと）

1. **本書の位置づけを再定義する**（最優先）。マネジメント計画書として書き直すか、システム設計書（地曵版）として残すか、塩澤の設計書に一部寄せる（Sync-Loss 状態のフェイルセーフ部分だけ独立貢献として残す等）かを決める。

3. **時刻同期方式の一本化**：本書の PTP-lite と塩澤 PDF のマスタクロック方式のどちらを採るかを決め、もう一方は破棄するか「フォールバック方式」として併記する。

5. **マネジメント計画書としての §1.1 概要・§1.2 スコープ・§1.3 作業計画・§1.4 V&V・§1.5 資源調達・§1.6 リスクの追加**。`work/shiozawa/work-0422/提出用計画書/plan_template.tex` のテンプレートに沿って形式を整える。
6. **§2.5 テストの設計の本文追加**（脱稿漏れの補完）。`wbs_table.md` 331-337 の TP-1〜TP-7 に対応する MOP の合否基準を書く。

## 6. 総合所見

技術設計書としての記述粒度は決して低くなく、特に **「予約実行型 UDP × 時刻同期 × Sync-Loss フェイルセーフ」** の 3 点セットでチーム内で重複しない独自貢献を出せる位置にある（Sync-Loss 状態は他メンバーの提出物に無い）。`EventQueue` / `TempoController` / `TimeSyncManager` のクラス分割と関数設計まで踏み込んだ点も評価できる。

ただし **指定された評価基準（マネジメント計画書ルーブリック）に対しては、形式・記述要件ともに不足が大きい**。題目・目的・独自性・参考文献／対象範囲・FBS・PBS／WBS・スケジュール・役割分担／MOP・TPM・テスト計画／実行可能性のいずれもが計画書として求められる粒度に達していない。

加えて、24G 勢が WBS 上「議事録持ち回り + 300 テスト以降に合流」と位置づけられている中で、本書のように設計フェーズに深く踏み込むこと自体が役割再定義を伴う。塩澤 PDF と二重定義になっている時刻同期・通信プロトコルをそのまま並走させると結合フェーズで衝突するため、第3回で「地曵の独自貢献ポイントは Sync-Loss 状態と冗長送出に絞る」「マネジメント計画書部分（§1.1〜§1.6 概要〜リスク）は地曵が担当して、システム設計書部分は塩澤に寄せる」のような棲み分け案を握ることを推奨したい。

### 7.3 システム設計書としての伸ばしどころ

現状 6 / 10 を 8 / 10 以上に引き上げるための具体的アクション:

1. **基本設計: 機能一覧とシステム構成図 B → A**（+1）
