### 2.2 細分音符（subdivision）の予約スロット上書き（Low）

`firmware/test_v2/node_02/include/SystemData.h:42-50` の `ScoreProgressData`:
```cpp
bool     pendingSub = false;
uint32_t pendingSubAtMs = 0;
...
```

`firmware/test_v2/node_02/src/applyPattern.cpp:50-62` の `fireScoreEvent()` で
新しい BEAT のたびに `pendingSub` を **無条件上書き** する。
コメントは「後続の BEAT で新しい予約が来たら上書きされる（1 BEAT につき高々 1 個の subdivision）」
としているが、**前ループまでに未発火だった旧予約は静かに捨てられる**。

きらきら星全行で `subNote == 0` のため現状は発火しないが、将来 8 分音符を含む曲に拡張した時、
拍裏の音符（subOffsetQ8 = 128）はその拍の中で発火するので問題ないが、
仕様が `subOffsetQ8 > 256`（次拍以降に予約）を許す書き方になっているのに対し、
それは確実にロストするので「**`subOffsetQ8` は `[0, 256)` のみ許容**」とコメントに明記すべき。

### 2.3 `int32_t` への uint32_t 差分キャストでの wraparound（Low）

`firmware/test_v2/node_02/lib/OrcReceiverModule/OrcReceiverModule.cpp:15`:
```cpp
const int32_t sample = (int32_t)(timestampMs - millis());
```

`uint32_t millis()` は約 49.7 日でラップする。短期セッションでは問題ないが、
demo 当日の長時間稼働中に指揮者と楽器どちらか片方だけがラップした瞬間に
`sample` 値が約 ±24 日ぶん（≒ 2×10^9 ms）に跳んで EMA が壊れる。

対策案:
- 差分計算前にもう一度 `(int32_t)` キャストを明示してオーバーフロー時の挙動を
  「処理系定義」から「2 の補数の連続性」に置き換える、または
- `(int32_t)((uint32_t)timestampMs - (uint32_t)millis())` の形に書き換えて
  unsigned 演算の結果を最後に signed に解釈する明示化を行う
（2 番目のほうが C/C++ 上、未定義動作回避として安全）。

ハッカソン本番（数十分稼働）では現実的に踏まない。要観察。

- B: `rehype-mermaid` + Playwright で SSG 時に SVG を生成して埋め込み

採用案のトレードオフ:
- ◯ 依存追加ゼロ・ビルド速度に影響なし・既存ページの Mermaid も同時に動く

- × expressive-code の DOM 内部に依存しているため、Starlight アップグレード時に
  `.ec-line .code` のクラス変更で容易に破綻する

短期的には合理的だが、**サイト公開先が GitHub Pages に確定した時点で SSG 化を検討**
