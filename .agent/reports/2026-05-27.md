## 2. バグ・矛盾レビュー

### 2.1 梅澤の Processing が NotePayload の `instrumentId`（byte 18）を読み飛ばしており、楽器音色を選択できない【重要度: High】

`work/umezawa/hck/processing/P03_NoteProtocol.pde:36-51` の `decodeNote()` は `durationMs`（byte 16-17）までしか抽出しておらず、`instrumentId`（byte 18）を完全に無視している。canonical の `pc_app/test_v2/orchestra_resynth/orchestra_resynth.pde:270` は `int instrumentId = u8(buf[18])` で受け取り `triggerNote(partId, instrumentId, ...)` に渡しているのと対照的。

設計上「instrumentId が楽器番号で、PC 側が data/*.json から音色を選ぶ」（`firmware/test_v2/common/lib/OrcProtocol/OrcProtocol.h:60`）アーキテクチャなので、梅澤の Processing では楽器切替が一切できない（NotePacket の `partId` で固定的に Brass / Rhythm 波形を選んでいるだけ）。

加えて `makeNoteFrame()`（同ファイル:53-68）も byte 18-19 を 0 埋めしているので、テスト送信フレームは canonical Processing 側で `instrumentId=0` 固定として表示される。

修正案（梅澤側の今後の作業）:
- `NotePacket` クラスに `int instrumentId` を追加し `frame[18] & 0xff` を読む。
- `PartManager.handleNote()` / `noteOn()` に instrumentId を引き渡し、`instrumentId` ベースで波形（あるいは data/*.json 解析結果）を選ぶ構造に書き換える。
- 同時に `makeNoteFrame()` に instrumentId 引数を追加。

### 2.7 梅澤 Processing の重複 seq だけ落とすロジックが順序逆転を吸収していない【重要度: Low】

修正案: `noteNumber == 0` 条件を外す。実害はほぼ無いが「過剰防御」は将来のトリガを潰す。
