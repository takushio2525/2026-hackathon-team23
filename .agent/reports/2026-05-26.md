### 2.3 `beatRedundancy=4` でも `beatGapMs=0` で実質効果が出ていない可能性 [High]

`ac8c5ff` の意図はコメントから読み取ると「ESP32-S3 SoftAP の radio がタイトループ連送だと
全滅しやすい → 各回の間に `beatGapMs` の delay を挟む」というロス対策。

しかし `firmware/test_v2/node_01/include/ProjectConfig.h:35` で `beatGapMs=0` が設定されている。
`OrcNetModule.cpp:135-140` のロジックでも `beatGapMs > 0` のときだけ `delay()` が挟まる:

```cpp
if (cfg_.beatGapMs > 0 && (i + 1) < reps) {
    delay(cfg_.beatGapMs);
}
```

つまり現状は単に「2 連送 → 4 連送に増やしただけのタイトループ」になっており、新規パラメータが
何も効いていない。コメント（`ProjectConfig.h:35` の `// 0 = タイトループ連送 (旧挙動)。
一時的に切り分け用`）と合致する切り分けフェーズの可能性は高いが、
**「切り分けが終わったか／推奨値（コメントでは 1-5ms 推奨）に切り替えたか」が
`.agent/progress.md` から追跡不能**。

**修正案**:
- `beatGapMs` を 0 のまま運用するならフィールド導入が無意味なのでロールバック
- 1〜5 ms に変更するなら、推奨値の決定根拠（再現テスト結果・ロス率）を `architecture.md`
  または `progress.md` に残す
- どちらにせよ実機で確かめないと判断不能 → user 作業
