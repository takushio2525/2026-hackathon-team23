### 2.2 `millis()` 24.8 日 wraparound に対する int32_t キャストの落とし穴（Low / コーナーケース）

`firmware/test_v2/node_02/src/applyPattern.cpp:140-142`:

```cpp
const int32_t targetLocalMs =
    (int32_t)data.receiver.pending.playAtMasterMs - data.sync.offsetMs;
const int32_t waitMs = targetLocalMs - (int32_t)now;
if (waitMs <= 0) { ... }
```

`millis()` が wrap する直前（`now ≈ 2^32 − 1`）に `(int32_t)now` は **負の小さい値**
として解釈される。`targetLocalMs` 側も `playAtMasterMs` が同様に wrap 近傍だと
両方とも負になり、引き算結果が誤った wait 値を返す可能性がある。
同様のキャストが `applyPattern.cpp:70` の `if ((int32_t)(now - data.score.pendingSubAtMs) < 0) return;`
にもある（こちらはまだ `uint32_t` 引き算してから cast しているのでより安全）。

ハッカソンの稼働時間（数分〜数十分）では実害は出ない。本番想定でも 24.8 日連続稼働は
非現実的なので **Low**。

**修正案**:
```cpp
// uint32_t で引いてから cast するパターンに統一
const int32_t waitMs = (int32_t)(targetLocalMs - now);
```
ただし `targetLocalMs` は既に `int32_t`。安全に揃えるなら全体を
`uint32_t` 演算 → 最後に `(int32_t)` cast に書き換える。実機テスト要なので
Claude 起点では触らない。

### 2.3 `score_data` の `subOffsetQ8` 範囲制限がコード/ドキュメントどちらにも無い（Low）
