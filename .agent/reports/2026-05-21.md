### 2.2 magic のバイト列表記が実装の wire 順と逆 — Med

報告書 `23_計画書・設計書.tex:906`（共通ヘッダ表）と `:894-895`（本文）は magic を
**「固定のバイト列 `0x4F 0x52`（ASCII "OR"）。バイト列のためエンディアンの対象外」** と記す。
しかし実装は:
```c
// OrcProtocol.h:18
constexpr uint16_t MAGIC = 0x4F52;
// OrcProtocol.h:17 のコメント自身が明記:
// ASCII "OR" (リトルエンディアンで 0x52 0x4F)
```
`MAGIC` は `uint16_t` であり、`#pragma pack` 構造体を memcpy で送出する実装
（`OrcProtocol.cpp:13` の `memcpy`）では、リトルエンディアン MCU（ESP32-S3／RA4M1 とも LE）
上で **wire 上のバイト順は `buf[0]=0x52, buf[1]=0x4F`**。つまり報告書の「`0x4F 0x52`」は
数値 `0x4F52` を 16 進そのまま並べた表記であって、実際に線上を流れるバイト列
（`0x52 0x4F`）とは逆。

Phase 4D の §9 整合チェック⑤で「magic は数値ではなくバイト列、エンディアン対象外」と
わざわざ書き換えた節だが、その結論を採るならバイト列は **`0x52 0x4F`** と書くべきで、
`0x4F 0x52` と書くと「バイト列扱い」という主張と自己矛盾する。報告書 §3.3.6 の
PC 側フレーム同期（`:1010`「magic（0x4F52）を走査」）も、表 `:906` の `0x4F 0x52` を
鵜呑みにすると走査するバイト順を取り違える。
- 修正案: ヘッダ表 `:906` を「`magic` は 16 ビット値 `0x4F52`。多バイト数値と同じく
  リトルエンディアンで配置し、wire 上は `0x52 0x4F` の順に並ぶ」と書き換える。あるいは
  「バイト列」フレームを保つなら列挙を `0x52 0x4F` に直す。ASCII 表記も、wire を素直に
  読むと "RO" になる点に触れると誤解が減る（定数名 `"OR"` は uint16 を概念的に綴った名称）。
- 影響度: 設計書上の事実誤りで、PC 実装者がフレーム同期マークを誤読しうる。実ファーム同士は
  両端が同じ memcpy 方式なので動作はする。Med。

### 2.3 対象範囲リストの実質重複と用語揺れ — Low

`23_計画書・設計書.tex:161-162`（§対象範囲）:

低⑩で F1 の「ピーク検出」→「振り下ろしの検出」を修正した際の取り残しに見える。
- 修正案: 2 項目を 1 つ（例「振り下ろしの検出による楽曲テンポの制御」）に統合、または
  後者を「振りの大きさに応じた…」のような別観点へ書き分ける。用語は「振り下ろしの検出」に統一。
