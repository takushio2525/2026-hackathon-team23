### 2.2 [Med] beatGapMs フィールド導入で旧 ProjectConfig.h を持つブランチがコンパイル不能

- 該当: `firmware/test_v2/common/lib/OrcNetModule/OrcNetModule.h:38` に
  `uint8_t beatGapMs;` を追加したが、構造体リテラルの位置引数なので、
  この 4 フィールド目を埋めていない `OrcNetConfig` リテラルは全てコンパイル
  失敗する（C++ struct designator は `/*foo=*/` 形式のコメント＋順序依存）。
- 影響: 既に node_01/02/03/04 の `ProjectConfig.h` は更新済なので test_v2
  内は問題ない。ただし `firmware/test_v1/` や別ブランチに `OrcNetConfig`
  リテラルがある場合は折れる。test_v1 を grep したところ test_v1 側は
  OrcNetModule を共有していないので実害なし。**ただし将来 production が
  test_v2 の common/lib を流用する想定なら、リテラル形式の脆さは継続課題**。
- **修正案**: 短期は現状維持。中期はメンバ初期化リスト `{}` に変えて
  `beatGapMs{0}` を構造体定義側のデフォルトにする（追加に強くなる）。
- 重要度 Med（test_v1 / 別実験ブランチで壊れる潜在リスク）。

### 4.3 ファームウェア BEAT 連送パケロス切り分け WIP（2026-05-25、1 コミット）

**やったこと**:
- `OrcNetConfig` に `beatGapMs`（連送間 delay [ms]）を追加し、`OrcNetModule.cpp`
  に「`beatGapMs>0` かつ最終回でなければ delay」処理を実装（`ac8c5ff`）
- node_01 で `beatRedundancy` を 2 → 4 に引き上げ、`beatGapMs=0` で

- node_04 は変更せず → 3 ノードで実験対照を作りたかった意図と推測

**完了度**: 明確に WIP。コミットメッセージ「デバック」、コメントに「一時的に
切り分け用」、`beatGapMs=0` のまま（追加した機能を実際には ON にしていない）、
3 ノード中 2 ノードだけ SERIAL_DEBUG=1 にする非対称性、すべて未完成の証拠。
コミットメッセージ規約違反。詳細は 2.1 / 2.4。

**設計判断**: 「連送間の delay で radio 状態の変化余地を作る」アイデア自体は
RF パケロスに対して理にかなっている（ESP32-S3 の Wi-Fi スタックは送信
バッファに連続投入されたパケットを背中合わせで送出するため、瞬間的な干渉で
両方落ちうる）。**ただし、本当に効くかは実機測定が要る**。設定上の対案として
は ①CSMA バックオフを ARM4 連送の代わりに使う、②BEAT を UDP ではなく
ESP-NOW に乗せる（衝突回避制御を OS から取り戻す）、③PC 側から ACK を返す
ハイブリッドにする、などがある。今夜の選択は最小コストで「現実装の中で」
対策する妥当な第一手だが、効果測定の枠組み（パケロス計数の Serial 出力）が
無いと正解にたどり着けない。

**品質（コード系）**: `beatGapMs` フィールド追加で位置引数リテラルが脆く
なる懸念は 2.2 で記述。OrcNetModule.cpp:138 の `if (cfg_.beatGapMs > 0 &&
(i + 1) < reps)` は「最終回後の delay 不要」を正しく排除しており、ループの
排他制御は問題なし。ヘッダ `OrcSenderModule.h:18` の「(1-3)」コメントだけが
言葉として古い（2.3）。

**コンテンツ品質**: 該当なし（コード変更のみ、レポート文章は触っていない）。
