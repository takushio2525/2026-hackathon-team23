# コントリビューションガイド

このリポジトリで開発するときのルール集。**Git も共同開発もほぼ初めて**という
メンバーを想定して書いている。慣れている人は「ブランチ戦略」以降を斜め読みでOK。

> **より広い開発フロー** （PlatformIO で Arduino を書く・Processing アプリを動かす・
> LaTeX 報告書をコンパイルする・コードの中身を読む）は、`docs/` 配下の Astro Starlight

> http://localhost:4321 に立ち上がる。本ガイドは Git に特化した最小ガイド。

> このガイドでは **VS Code（GitLens）** と **ターミナル** の2通りで手順を説明している。
> 好きな方を使えばよいが、慣れないうちは **VS Code の GUI** が分かりやすい。

## 🚨 一番大事なルール：作業前に必ずリポジトリを最新化する

コード編集に取りかかる**前に**、ローカルのリポジトリを最新の状態に揃える。

- **VS Code**: 左下のブランチ名クリック → `main` 選択 → ステータスバーの **同期アイコン（🔄）** をクリック
- **ターミナル**: `git switch main && git pull`

**忘れるとどうなるか**：

- Bさんが昨日 main に PR をマージした
- あなたはそれに気づかずに、古い main からブランチを切って作業
- PR を出すときに「謎のコンフリクト」が大量発生して解消に1時間…

チームメンバーが push した変更を取り込まないまま作業を始めると、
PR を出すときにコンフリクトが大量発生して時間を浪費する。
「作業の最初に必ず最新化」を徹底すること。

## 1. 最初のセットアップ（Git を初めて使う人向け）

### 1.1 Git のインストール

- macOS: `xcode-select --install` か Homebrew で `brew install git`
- Windows: [Git for Windows](https://gitforwindows.org/) をインストール
- Linux: `sudo apt install git` など

インストールできたか確認：

```bash
git --version
```

### 1.2 VS Code と推奨拡張機能のインストール

このリポジトリでは **VS Code** をエディタとして推奨している。

1. [VS Code 公式サイト](https://code.visualstudio.com/) からインストール
2. VS Code でこのリポジトリのフォルダを開く
3. 右下に「**推奨拡張機能をインストールしますか？**」と通知が出るので、**すべてインストール** を選択

手動で入れる場合は、サイドバーの「拡張機能」アイコン（四角が4つ）を開き、以下を検索してインストール：

| 拡張機能 | 用途 |
|---|---|
| **GitLens** (`eamodio.gitlens`) | Git の履歴・差分・ブランチ管理を GUI で操作 |
| **EditorConfig** | コーディングスタイルの自動統一 |
| **PlatformIO IDE** | Arduino 開発する班向け |
| **LaTeX Workshop** | 報告書（LaTeX）を書く人向け |

### 1.3 名前とメールアドレスを登録（初回のみ）

commit の作者情報として使われる。GitHub に登録したアドレスを使うと、
GitHub 上であなたのアイコンが表示されて分かりやすい。

VS Code 内のターミナル（`` Ctrl+` `` で開く）か、普通のターミナルで以下を実行：

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
```

### 1.4 GitHub との認証

HTTPS で clone するなら、push のときに GitHub の**個人アクセストークン（PAT）**を
求められる。パスワードは使えないので注意。

- 参考: [GitHub: 個人アクセストークンの作成](https://docs.github.com/ja/authentication/keeping-your-account-and-data-secure/managing-your-personal-access-tokens)

より楽なのは **SSH 鍵** を登録する方法。一度設定すればパスワード不要になる。

- 参考: [GitHub: SSHでの接続](https://docs.github.com/ja/authentication/connecting-to-github-with-ssh)

## 2. Git の基本操作（push・pull・merge）

Git のコマンドは最初ややこしく見えるが、**ローカル（自分の PC）** と
**クラウド（GitHub）** の間で「**何が・どっち向きに動くか**」さえ掴めば、
日常作業の 8 割はカバーできる。

### 2.1 全体像：ローカルとクラウドの関係

```
              [ GitHub（クラウド） ]
                 ▲      │
          push ──┤      ├── pull / fetch
                 │      ▼
              [ 自分の PC（ローカル） ]
```

ブランチはローカルにもクラウドにも**別々に存在する**。`push` / `pull` は
この 2 つを同期する操作で、`merge` は**ローカル内**でブランチ同士を合体させる操作。

### 2.2 push（ローカル → クラウド）

自分が作ったコミットを GitHub にアップロードする。

- **いつ使う？** コミットが溜まって、チームに共有したいとき
- **VS Code**: ソース管理パネルの **↑ アイコン（雲マーク）**、または「…」→「プッシュ」
- **ターミナル**: `git push`

**注意点**：
- push する前に **コミット済み** でないと送れない（ステージだけでは不十分）
- 初回のブランチは `git push -u origin <ブランチ名>` で追跡設定が必要

### 2.3 pull（クラウド → ローカル、同名ブランチのみ）

**今いるブランチと同じ名前のリモートブランチ** の最新をローカルに取り込む。

- **いつ使う？** 他メンバーが同じブランチに push したとき、自分のブランチを最新化したいとき
- **VS Code**: ソース管理パネルの **↓ アイコン**、または「…」→「プル」
- **ターミナル**: `git pull`

> **よくある誤解**：「pull すれば main の変更も取り込める」→ **違う**。
> pull は同名ブランチしか同期しない。`feature/foo` で pull したら
> `origin/feature/foo` しか来ない。main の変更を自ブランチに入れたいなら、
> 後述の **merge** を使う。

### 2.4 fetch（情報を取ってくるだけ）

`pull` は内部的に **`fetch` + `merge`** の 2 段階。
`fetch` はその前半だけ：リモートの最新状態を取り込むが、**作業中のブランチは変えない**。

- **ターミナル**: `git fetch origin`
- **VS Code**: 「…」→「フェッチ」
- **使いどころ**：merge する前にリモートを最新化したいとき。
  「main がどれだけ進んだか」を確認したいときもこれ。

### 2.5 merge（別ブランチ → 現在のブランチ）

別ブランチ（典型的には `main`）の変更を、**今いるブランチに取り込む** 操作。

- **いつ使う？** main が進んだので、自分のブランチに反映したい
- **VS Code**: コマンドパレット（`Cmd+Shift+P`）→ `Git: Merge Branch...` → `origin/main` を選択
- **ターミナル**:
  ```bash
  git fetch origin          # まずリモート最新化
  git merge origin/main     # main を現在のブランチに取り込む
  ```

**結果**：2 つのブランチが合流した **マージコミット** が 1 つ増える。
履歴はこうなる：

```
A---B---C---D---E          main
         \       \
          F---G---M         自分のブランチ（M がマージコミット）
```

### 2.6 まとめ表

| 操作 | どこから | どこへ | VS Code | ターミナル |
|---|---|---|---|---|
| **push** | ローカル | クラウド（同名ブランチ） | ↑ アイコン | `git push` |
| **pull** | クラウド（同名ブランチ） | ローカル | ↓ アイコン | `git pull` |
| **fetch** | クラウド（全情報） | ローカル（閲覧用） | 「…」→「フェッチ」 | `git fetch origin` |
| **merge** | ローカルの別ブランチ | 現在のブランチ | コマンドパレット | `git merge <ブランチ名>` |

### 2.7 rebase は使わない（このチームのルール）

`rebase` は merge と同じく「main の変更を取り込む」用途で使われるが、
**履歴を書き換える操作**なので扱いが難しい。

- コミットハッシュが書き換わるので **`force push` が必要**
- 共有ブランチで使うと **他人のコミットを壊す事故**が起きやすい
- VS Code の標準 UI で使うには追加設定（`git.allowForcePush`）が要る

**このチームでは main の取り込みは常に `merge`** を使う。
rebase は使わない。どうしても使いたい場面があれば、リーダーに相談してから。

## 3. ブランチ戦略

### 基本ルール

- `main` ブランチへの直接 push は **原則禁止**（軽微な共有系ドキュメント更新などは例外）
- 機能追加・バグ修正・他人と衝突しそうな変更は**必ずブランチを切って作業する**
- 作業が終わったら Pull Request（PR）を作って、レビューしてからマージ

### ブランチ名の付け方

種類 / 短い説明 の形式：

| 種類 | 例 |
|---|---|
| `feature/` | `feature/add-sensor-reading` |
| `fix/` | `fix/null-pointer-on-startup` |
| `docs/` | `docs/update-readme` |
| `refactor/` | `refactor/split-comm-module` |

- 英語でも日本語でもよいが、短く・内容が分かる名前にする
- スペースは使わない（`-` か `_` でつなぐ）

### ブランチ切りのコツ

1. **必ず最新の main から切る**（pull してからブランチを作る）
2. **1ブランチ = 1目的**（レビュー負担を減らす）
3. **長期間持たない**：1週間以上持つとコンフリクト地獄になる。大きい作業は
   途中で PR を分ける
4. **作業中に main が進んだら取り込む**（下記「main の取り込みルール」参照）
5. **マージされたブランチは消す**

### main の取り込みルール（重要）

自分の作業ブランチに main の変更を取り込むときは、**必ず `merge` を使う**。
`rebase` は使わない（理由は [§2.7](#27-rebase-は使わないこのチームのルール) 参照）。

**頻度の目安**：

- **作業再開時**（毎朝 / 毎セッションの頭）に 1 回
- **PR を出す直前** に 1 回
- main に他メンバーがマージした通知を見たら、その日のうちに 1 回

放置すると解消コストが跳ね上がる。**「1 日 1 回は main を取り込む」** を習慣化すると事故が激減する。

**手順（ターミナル）**：

```bash
git fetch origin
git merge origin/main
```

**手順（VS Code）**：

1. コマンドパレット（`Cmd+Shift+P`）→ `Git: Fetch`
2. コマンドパレット → `Git: Merge Branch...` → `origin/main` を選択
3. コンフリクトが出たら [§4.12 コンフリクトの解消（VS Code）](#412-コンフリクトの解消vs-code) 参照

## 4. VS Code で操作する場合の手順

VS Code + GitLens を使った、clone から PR までの一連の流れ。

### 4.1 リポジトリを clone する

1. コマンドパレット（`Ctrl+Shift+P` / macOS: `Cmd+Shift+P`）を開く
2. `Git: Clone` と入力して選択
3. リポジトリの URL（`https://github.com/<組織名>/<リポジトリ名>.git`）を貼り付けて Enter
4. 保存先フォルダを選ぶ → clone が始まる
5. 「クローンしたリポジトリを開きますか？」と聞かれるので **「開く」** を選択

### 4.2 最新化してブランチを作る

1. 左下のブランチ名（例: `main`）をクリック → `main` に切り替え
2. ステータスバーの **同期アイコン（🔄 循環矢印）** をクリックして pull
3. 再び左下のブランチ名をクリック → 「**+ 新しいブランチの作成...**」を選択
4. ブランチ名（例: `feature/my-task`）を入力して Enter

> **注意**: 新しいブランチを作る前に**必ず** main に切り替えて同期（pull）すること。
> これを忘れると、古い状態からブランチが生えてしまい、後でコンフリクトの原因になる。

### 4.3 ファイルを編集する

VS Code でファイルを開いて編集する。

### 4.4 変更を確認する

1. サイドバーの「**ソース管理**」アイコン（`Ctrl+Shift+G` / macOS: `Cmd+Shift+G`）をクリック
2. 「変更」セクションに変更されたファイルの一覧が表示される
3. ファイル名をクリックすると**差分（diff）** が表示される（左が変更前、右が変更後）

> GitLens を入れていると、エディタ上で各行の最終変更者・日時がインライン表示される。

### 4.5 ステージングする（コミットするファイルを選ぶ）

1. 「ソース管理」パネルで、ステージしたいファイルの右側にある **「+」ボタン** をクリック
2. クリックしたファイルが「**ステージされている変更**」セクションに移動する
3. 間違えたら「ステージされている変更」のファイル右側の **「−」ボタン** で戻せる

> 全ファイルを一括ステージしたい場合は「変更」セクションのタイトル右側の「+」を押す。
> ただし `.env` 等を含めないよう注意。

### 4.6 コミットする

1. 「ソース管理」パネル上部の**メッセージ入力欄**にコミットメッセージを入力
   - 例: `[機能追加] センサ読み取り処理を追加`
   - 書式は [コミットメッセージ規約](#6-コミットメッセージ規約) を参照
2. **✓（チェックマーク）ボタン** をクリック、または `Ctrl+Enter`（macOS: `Cmd+Enter`）

### 4.7 push する（GitHub にアップロード）

1. 「ソース管理」パネルの **「…」メニュー** → **「Push」** を選択
2. 初回は「リモートブランチがありません。公開しますか？」と聞かれるので **「OK」** を選択
3. または、ステータスバーの **↑ アイコン（雲マーク）** をクリックしても push できる

### 4.8 Pull Request を作る

1. GitHub のリポジトリページをブラウザで開く
2. 黄色いバナー「Compare & pull request」が出ているのでクリック
3. PR テンプレートが自動で入るので、埋めて「Create pull request」
4. レビュアーを指定する

> VS Code で **GitHub Pull Requests** 拡張（`GitHub.vscode-pull-request-github`）を入れると、
> VS Code 内から PR の作成・レビューもできる。

### 4.9 レビュー指摘に対応する

1. 同じブランチのまま、ファイルを編集 → ステージ（+ボタン） → コミット（✓ボタン） → push
2. PR のページが自動で更新される（GitHub 上でコメントに「Resolve conversation」）

### 4.10 マージ後の片付け

1. GitHub 上でマージボタンを押す
2. 左下のブランチ名クリック → `main` に切り替え
3. 同期アイコン（🔄）をクリックして pull
4. 不要なブランチを消す：コマンドパレット → `Git: Delete Branch...` → 削除するブランチを選択

### 4.11 作業中に main が進んだら

1. コマンドパレット（`Ctrl+Shift+P`）→ `Git: Fetch` で最新情報を取得
2. コマンドパレット → `Git: Merge Branch...` → `origin/main` を選択
3. コンフリクトが起きたら → [コンフリクトの解消（VS Code）](#412-コンフリクトの解消vs-code)

### 4.12 コンフリクトの解消（VS Code）

merge したときに「conflict（衝突）」が起きることがある。2人が同じ行を別々に編集したときに発生する。

1. コンフリクトが発生すると、「**ソース管理**」パネルの「**マージの変更**」セクションに対象ファイルが表示される
2. ファイルを開くと、該当箇所に色付きのハイライトとボタンが表示される：
   - **「現在の変更を取り込む（Accept Current Change）」** — 自分の変更を採用
   - **「入力側の変更を取り込む（Accept Incoming Change）」** — 相手（main）の変更を採用
   - **「両方の変更を取り込む（Accept Both Changes）」** — 両方残す
   - **「変更の比較（Compare Changes）」** — 差分を横並びで確認
3. 正しい内容に編集できたら、ファイルを保存
4. 「ソース管理」パネルで解消済みファイルの **「+」ボタン** でステージ
5. コミットメッセージを入力して ✓ ボタンでコミット

> **ヒント**: どちらの変更が正しいか分からないときは、GitLens の「File History」で
> そのファイルの変更履歴を確認すると判断しやすい。

**一人で悩まず、相手に聞く**のが一番早い。

## 5. ターミナルで操作する場合の手順

ターミナル（コマンドライン）を使った、clone から PR までの一連の流れ。
VS Code 内のターミナル（`` Ctrl+` ``）でも、OS のターミナルアプリでもどちらでもよい。

### 5.1 リポジトリを clone する

```bash
# 好きな場所に移動して
cd ~/Documents

# clone
git clone https://github.com/<組織名>/<リポジトリ名>.git
cd <リポジトリ名>
```

### 5.2 最新化してブランチを作る

```bash
git switch main              # main ブランチに移動
git pull                     # リモートから最新を取り込む
git switch -c feature/my-task   # 新しいブランチを作成して切り替え
```

### 5.3 ファイルを編集する

好きなエディタで編集。

### 5.4 変更を確認する

```bash
git status       # どのファイルを変更したか
git diff         # 具体的にどう変わったか
```

### 5.5 ステージングする（コミットするファイルを選ぶ）

```bash
git add path/to/file.c            # 特定のファイル
git add path/to/dir/              # ディレクトリ丸ごと
# git add -A や git add . は便利だが、意図しないファイル（.env 等）まで
# 含めがちなので、慣れるまではファイル名を直接指定するのが安全
```

### 5.6 コミットする

```bash
git commit -m "[機能追加] センサ読み取り処理を追加"
```

コミットメッセージの書式は [コミットメッセージ規約](#6-コミットメッセージ規約) を参照。

### 5.7 push する（GitHub にアップロード）

```bash
# 初回（ブランチを GitHub に作るとき）
git push -u origin feature/my-task

# 2回目以降
git push
```

### 5.8 Pull Request を作る

1. GitHub のリポジトリページをブラウザで開く
2. 黄色いバナー「Compare & pull request」が出ているのでクリック
3. PR テンプレートが自動で入るので、埋めて「Create pull request」
4. レビュアーを指定する

### 5.9 レビュー指摘に対応する

1. 同じブランチで編集 → `git add` → `git commit` → `git push`
2. PR のページが自動で更新される（GitHub 上でコメントに「Resolve conversation」）

### 5.10 マージ後の片付け

```bash
git switch main
git pull
git branch -d feature/my-task   # ローカルのブランチを削除
```

### 5.11 作業中に main が進んだら

```bash
git fetch origin          # 最新情報を取得（ローカルは変わらない）
git merge origin/main     # 自分のブランチに main を取り込む
```

### 5.12 コンフリクトの解消（ターミナル）

merge したときに「conflict（衝突）」が起きることがある。2人が同じ行を別々に編集したときに発生する。

```bash
# merge 後に CONFLICT と出たら

# 1. 対象ファイルを開くと、以下のようなマーカーが入っている
#    <<<<<<< HEAD
#    あなたの変更
#    =======
#    相手の変更
#    >>>>>>> origin/main

# 2. 正しい内容になるよう編集（マーカーは消す）

# 3. 直したファイルを add して commit
git add path/to/conflicted-file
git commit
```

**一人で悩まず、相手に聞く**のが一番早い。

## 6. コミットメッセージ規約

`[種別] 変更内容の概要` の形式で書く。

| 種別 | 用途 |
|------|------|
| `[機能追加]` | 新機能の実装 |
| `[修正]` | バグ修正 |
| `[改善]` | 既存機能の改善・パラメータ調整など |
| `[リファクタ]` | コード整理（機能変更なし） |
| `[ドキュメント]` | ドキュメントのみの変更 |
| `[スタイル]` | 見た目・UI の変更 |

### コツ

- 機能単位で**細かく分割**する（無関係な変更を1コミットにまとめない）
- コードの変更と、そのコードに対応する README 更新は**同じコミット**に入れる
- 「なぜ」を意識する（「追加した」ではなく「〇〇のために追加した」の視点）

---

## 7. レビュー観点

PR のレビュー時は以下をチェックする。

### コード全般
- 命名規則に従っているか
- 不要なコメントアウト・`print` / `Serial.println` のデバッグ出力が残っていないか
- エラーケースが考慮されているか

### ドキュメント
- README に書いてある挙動と実装が一致しているか
- 設計判断が大きい変更の場合、`docs/src/content/docs/decisions/` に ADR を追加したか（ドキュメントサイトの「意思決定の記録」に表示される）

### CI
- GitHub Actions が全部成功しているか（❌ のままマージしない）

## 8. よくあるミスと対処

### ❌ main に直接 commit してしまった

まだ push していないなら：

```bash
git switch -c feature/my-task   # 今のコミットをブランチに移す
git switch main
git reset --hard origin/main    # main をリモート状態に戻す
git switch feature/my-task
```

push してしまった場合は、リーダーに相談。

### ❌ ステージに意図しないファイルを混ぜた

- **VS Code**: 「ソース管理」パネルの「ステージされている変更」で、外したいファイルの **「−」ボタン** をクリック
- **ターミナル**: `git reset path/to/file`（ステージから外す。中身は消えない）

### ❌ コミットメッセージをタイポした

まだ push していない直前のコミットなら：

```bash
git commit --amend -m "[機能追加] 正しいメッセージ"
```

push した後は、PR の説明欄でフォローするのが安全。

### ⚠️ `git push --force` はやらない

他人のコミットを消してしまう事故のもと。**必要になったら必ずリーダーに相談**。

## 9. Issue で作業を管理する

- 分担タスクは Issue を立てて管理する
- [`.github/ISSUE_TEMPLATE/task.md`](.github/ISSUE_TEMPLATE/task.md) を使うと書式が揃う
- PR の説明文に `Closes #NN` と書くと、マージ時に自動で Issue が閉じる

## 10. 困ったら

- まずは PR / Issue に**質問を書く**（誰かが答えてくれる）
- Slack / Discord などチームのコミュニケーションツールで聞く
- `git` は取り返しのつくこと・つかないことがある。破壊的操作（`reset --hard` /
  `push --force` / `rebase`）は不安なら相談してから実行する
