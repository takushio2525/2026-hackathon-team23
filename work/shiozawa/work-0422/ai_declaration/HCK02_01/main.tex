\section*{1. 何に使ったか}
授業（ハッカソン1 第2回）の例題2 で作成した「マイク入力を Serial Plotter に
表示する」プログラムを動かしたところ、最後のスライドのとおり波形が三角形のように
カクついて、正弦波に見えなかった。提出課題1 はこの例題2 のプログラムを
「シリアルモニタ（プロッタ）に正しく波形が表示されるよう」修正する課題なので、
まず「配布された例題2 のプログラムだと、なぜうまく波形が表示されないのか」という
原因の切り分けを生成AIに相談した。

\section*{2. AIへの指示（プロンプト）}
\begin{lstlisting}
授業の例題2 で配布されたコードをそのまま動かすと、Serial Plotter での
波形がギザギザの三角形のようになって正弦波に見えません。コードは
こんな感じです:

  Serial.print("実測値:");

  Serial.print(",振幅:");
  Serial.print(a);

ボーレートは 921600 です。波形が崩れて見える原因はどこにあると考えられますか?
\end{lstlisting}

\section*{3. AIの出力の使い方}
\begin{itemize}
  \item \cboxoff そのまま使った
  \item \cboxon 一部修正して使った
  \item \cboxoff 参考にしただけ
\end{itemize}
$\rightarrow$ 上で選んだ理由：

AI からは「一番のボトルネックは出力部分そのもの」と指摘された。要点は、
\begin{itemize}
  \item \texttt{"実測値:"}, \texttt{",振幅:"}, \texttt{",最大振幅:"},
        \texttt{",最小振幅:"} という日本語ラベルを毎ループ \texttt{Serial.print} で
        送っており、これだけで 1 サンプルあたり数十バイト送ることになる
  \item 921600\,bps でも 1 バイト送るのに約 11\,$\mu$s かかるので、
        ラベルと数値で 50 バイト前後送ると 1 ループの送信時間だけで
        500\,$\mu$s 以上かかってしまう
  \item さらに \texttt{delay(1)} (1\,ms 待機) と \texttt{analogRead} (約 100\,$\mu$s)
        が加わるので、ループ周期はおおむね 1.5--2\,ms（約 500--700\,Hz）まで
        落ちてしまう。これでは可聴域の音をサンプリングしきれず、
        サンプル点の少なさで波形が三角形に見える
  \item 解決策は「シリアルに送る文字数を減らす」のが最優先。整数 1 つに絞れば
        1 サンプルあたり 5--6 バイトになり、ループの大半を占めていた送信時間が
        大きく減ってサンプリングが速くなる
\end{itemize}
という説明だった。これに加えて Serial Plotter の縦軸が自動で動く件についても
触れられたので、「ラベルと余分な値を全部消す」「縦軸固定用に \texttt{0} と
\texttt{1023} だけ一緒に送る」という方針をそのまま採用した。AI が最初に書いてきた
コード自体にはまだ \texttt{"v:"} のような短いラベルが残っていたので、
最終的にはそれも削って整数だけのカンマ区切りに直した。

\section*{4. 正しいと確認した方法}
\begin{itemize}
  \item Arduino IDE で書き込み後、シリアルモニタを 921600\,bps で開き、
        \texttt{<生値>,0,1023} の 1 行だけが流れている（\texttt{"実測値:"} などの
        文字列が一切出ていない）ことを確認した
  \item Serial Plotter に切り替えると、上下に \texttt{0} と \texttt{1023} の
        ガイド線が常に表示され、縦軸が動かなくなったことを確認した
  \item マイクに 440\,Hz の正弦波（スマホのチューナーアプリで再生）を聞かせ、
        ガタガタの三角形ではなく丸みのある波形として見えることを確認した
  \item 出力をいったん例題2 のラベル付き形式に戻して \texttt{delay(1)} のまま
        動かしたら、AI の説明どおりに波形が再びギザギザになった。ラベルと
        \texttt{delay(1)} を順番に外していくと波形が滑らかになっていったので、
        「シリアル出力の文字数がループ周期を支配していた」ことを再現実験で
        確かめた
\end{itemize}

\section*{5. 問題や間違い}
\begin{itemize}
  \item \cboxon あった
  \item \cboxoff なかった
\end{itemize}

AI の最初の説明では「\texttt{delay(1)} を消すだけでも改善する」とまず示されたが、
これだけだとラベル送信の時間が残り、まだ 1\,kHz には届かない計算になる。
そこをさらに突っ込んで聞いたところ、上に書いた「ラベルがボトルネックで、
そこを削ると一番効く」という回答に至った。最初の浅い回答だけで満足せず、
原因を数値で詰めて聞き直したのがポイントだった。
また、出力例として AI が \texttt{"v:"} のような短いラベル付き形式を出してきたが、
Plotter が複数系列扱いにする可能性が残るため、最終的にはラベル無しの数値のみに
直した。

\section*{6. 自分で工夫した点}
\begin{itemize}
  \item 例題2 にあった日本語ラベル 4 つと、振幅・最大振幅・最小振幅の \texttt{float}
        計算（電圧換算 \texttt{(float) d / (pow(2,RESOLUTION)-1) * 5.0} と中心化）を
        まとめて削除し、ループ内を「\texttt{analogRead} → 整数 1 つ送信」だけに
        絞った
  \item Serial Plotter の縦軸固定のために、毎ループで \texttt{0} と \texttt{1023} を
        一緒に送る形にした（送信文字は数バイトしか増えないので、サンプリング
        周期にはほぼ影響しない）
  \item \texttt{delay(1)} は外し、代わりに \texttt{delayMicroseconds(100)} を入れた。
        Arduino UNO R3 の \texttt{analogRead} 自体に約 100\,$\mu$s かかるため、
        ループ周期は約 200\,$\mu$s（5\,kHz）に落ち着く。これで 440\,Hz の正弦波が
        1 周期あたり 10 サンプル前後で取れるようになり、画面上でも丸い波形に
        見えることを実機で確認したうえで採用した
\end{itemize}
