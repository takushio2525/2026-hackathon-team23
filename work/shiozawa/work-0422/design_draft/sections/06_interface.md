# 6. 共通インターフェース方針

本章は EMA の主要インターフェースを「方針レベル」で示す。各 API の具体的シグネチャと
実装例は第 10 章で詳述する。

## 6.1 3 フェーズループ

すべてのノードで `loop()` は以下の順序で実行する。

```text
loop() {
    InputPhase();   // センサ取得・UDP 受信（世界からの入力）
    LogicPhase();   // 判定・状態更新（SystemData 更新）
    OutputPhase();  // LED・UDP 送信（世界への出力）
}
```

- **入力フェーズ**: I/O 読み取りは基本ここでまとめる。取得結果は `SystemData` に書き出す
- **ロジックフェーズ**: 入力を見て状態を決める。外界への副作用は発生させない（純ロジック）
- **出力フェーズ**: `SystemData` を元に LED / UDP などの出力を行う

これにより「入力 → ロジックだけ差し替えてテスト」「出力だけモック化」といった部分検証が
成立する（単体テスト方針は第 13.1 章）。

## 6.2 IModule / ModuleTimer / SystemData / ProjectConfig

| 抽象 | 役割 | 典型的な粒度 |
|---|---|---|
| `IModule` | 機能を表す最小単位。`setup()` で初期化、`update()` で 1 ステップ進める | 1 クラス = 1 責務（IMU 読み取り、拍検出、UDP 送信 等） |
| `ModuleTimer` | 非ブロッキング周期実行。`delay()` 禁止の代替 | サンプリング周期・送信周期・ハートビート |
| `SystemData` | ノード内でモジュール間共有する状態を 1 構造体に集約 | `acc[3]`, `bpm`, `phase`, `last_beat_ms` 等 |
| `ProjectConfig` | ピン番号・定数・閾値などノード固有設定を 1 ヶ所に集約 | サンプリング周波数・フィルタカットオフ・UDP ポート等 |

モジュール間の結合は **`SystemData` を介した疎結合** とする。モジュールが他モジュールを
直接呼ぶことは避け、「書き込み責務」と「読み取り責務」を役割ごとに明確に分ける
（例: IMU ドライバが `SystemData.acc` に書き、拍検出モジュールはそれを読んで `SystemData.beat_event` を書く）。

## 6.3 通信プロトコル概要（UDP、CTRL / BEAT / NOTE）

指揮者 → 楽器 → PC の 3 系統で、3 種類のメッセージを流す。詳細なパケット構造は
第 10.3 章で定義する。

| メッセージ | 送信元 → 宛先 | 送信契機 | 性質 | 役割 |
|---|---|---|---|---|
| **CTRL** | node_01 → node_02〜05（ブロードキャスト） | 定期（例: 20 Hz） | 冪等・状態送信 | BPM・velocity・演奏状態を通知 |
| **BEAT** | node_01 → node_02〜05（ブロードキャスト） | 拍検出時（不定期） | イベント・即時性重視 | 楽譜ポインタを前進させるトリガ |
| **NOTE** | node_02〜05 → PC(ユニキャスト) | 音符発火時 | イベント | Processing に発音を依頼 |

**設計原則**:

- CTRL は冪等性を持ち、取りこぼしても次回送信で整合する（パケロス耐性を通信方針側で確保）
- BEAT は即時性が必要なので、送信失敗時のリカバリは **楽譜側で内挿**する（直前の BPM から
  次拍の予測時刻を計算し、その時刻を過ぎても BEAT が来なければ自走する）
- NOTE は楽器 → PC で 1 経路。楽譜に従って必ず送出する
- すべてに共通ヘッダ（プロトコルバージョン・メッセージ種別・シーケンス番号）を付ける
