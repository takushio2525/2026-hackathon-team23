# 2. 背景・課題

## 2.1 授業課題の要求

本授業（ハッカソン1）では、以下が制約・評価軸として与えられている（根拠:
[`references/lectures/HCK1_01_Kickoff_A.pdf`](../../../../../references/lectures/HCK1_01_Kickoff_A.pdf)）。

- **ハード台数**: 人数分 − 1 台 の Arduino を使用（6 名 → 5 台）
- **演奏要件**: 各楽器が異なるパートを演奏、最低で主旋律 3 + リズム 1 の輪唱曲
- **評価項目**: 同期信号の受信精度 / タイミング誤差 / エンターテインメント性 / 楽器音らしさ

本チームはこれを踏まえ、**指揮者 1 + 楽器 4 の金管四重奏（輪唱）** を採用することまで
第 1 回ミーティングで合意済み（[ADR-0004](../../../../../docs/decisions/0004-ensemble-structure.md)）。

## 2.2 技術的な課題

本システムが直面する主要な課題は以下のとおり。

1. **多台同期**: 5 台の Arduino が共通の拍・テンポで演奏を進める必要がある。単純な
   「各ノードで独立にタイマ刻み」では時計ずれで数秒～数分で位相が崩れる。
2. **可変テンポ**: 指揮の速度で BPM が変わるため、固定 MIDI 再生は使えない。
   指揮者発の「拍」イベントを楽器側がリアルタイムに受けて進行させる方式が必要。
3. **ジェスチャ認識**: 内蔵 IMU から取得する加速度・角速度の時系列から「振り下ろし」を
   検出し、そこから BPM・強弱を抽出する信号処理が必要(実装未着手)。
4. **通信の信頼性**: UDP はパケロス対策がプロトコル設計側の責務。最低限の冗長化・
   整合性チェック（シーケンス番号など）を自前で設計する必要がある
   （[ADR-0002](../../../../../docs/decisions/0002-udp-original-protocol.md)）。
5. **多人数開発の衝突回避**: 複数メンバーが同じリポジトリを編集するため、責務分離が
   弱いと実装終盤に結合エラーで止まる。設計段階でモジュール境界を明確に切っておく
   必要がある（[ADR-0005](../../../../../docs/decisions/0005-firmware-embedded-module-architecture.md)）。

## 2.3 担当拡大の経緯（本設計書のスコープが広がった理由）

第 1 回ミーティング時点では、塩澤担当は「指揮者マイコン（node_01）」のみだった。
第 2 回ミーティング（2026-04-22）で以下の状況が確認され、**Arduino 系プログラム
全般（node_01〜05）** を塩澤が一括設計することになった。

- 楽器ノードの「コーディング担当」が議事録ベースでは曖昧のまま残っていた
- 齋藤（音階生成）・梅澤（Processing）・他メンバーはそれぞれ別軸の作業（楽譜データ・
  音色合成・実機検証など）に集中するほうが全体効率が良い
- Embedded-Module-Architecture の共通層（`IModule` 等）は 5 台全てで共有するため、
  一人が見通して設計したほうがモジュール境界を揃えやすい

この判断により、本設計書は **指揮者 + 楽器を通貫で設計する** 文書になる。
