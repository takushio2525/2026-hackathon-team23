# 3. 目的

## 3.1 システムとしてのゴール

本設計書が対象とするファームウェアが満たすべきゴールを、必達とストレッチに分けて示す
（出典: [`docs/overview.md`](../../../../../docs/overview.md)）。

**必達**

1. 指揮者 1 台 + 楽器 4 台の **5 台同期演奏** が成立する
2. 輪唱曲（主旋律 3 + リズム 1 以上）を演奏できる
3. **可変テンポ**に対応する（指揮の速度に追従）

**ストレッチ**

4. 指揮ジェスチャで **強弱（ベロシティ）** を制御できる
5. 見た目のエンタメ要素（LED・装飾）を付加

## 3.2 本設計書（原案）が満たすべき条件

設計書として、以下を満たすことを目的とする。

- **設計意図の明文化**: 採用したアーキテクチャ（Embedded-Module-Architecture）の理由と
  モジュール境界を、他メンバーが読んで理解できる水準で記述する
- **実装の出発点**: `firmware/` 配下の実装者（塩澤本人および将来のレビュアー）が、
  本設計書を見ればそのままコーディングに入れる詳細度を持つ
- **検証可能性**: 各モジュールが「何をもって正しく動いたとみなすか」を V&V / TPM の
  形で明記する（第 13 章）
- **柔軟性の確保**: アルゴリズム（拍検出・テンポ推定）は **候補方式＋推奨** で記述し、
  実装フェーズで実測に基づき確定できる余地を残す

## 3.3 非目的（本設計書が扱わないこと）

- 楽器音色の合成アルゴリズム（Processing 側・梅澤担当）
- 具体的な楽譜の選定と編曲（チーム共同で後日決定）
- ハードウェア追加（外付け IMU、スピーカ等）— 現時点では UNO R4 WiFi 内蔵機能のみで設計
- 実装コードそのもの（本原案は設計段階、実機検証は実装フェーズ以降）
