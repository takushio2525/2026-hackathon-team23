## 1. 機能と性能指標（MOP）の対応

| ID | 指標 | 目標値 |
|----|------|--------|
| MOP-1 | 楽器間 同期誤差（4 ノードの発音タイミング差） | ≤ 20 ms |
| MOP-2 | 指揮 → 楽器 通信遅延（送信〜受信） | ≤ 10 ms |
| MOP-3 | 拍検出 正解率（定速指揮時） | ≥ 90 % |
| MOP-4 | テンポ追従 遅延（BPM 階段変化への追従） | ≤ 2 拍 |
| MOP-5 | パケロス耐性（聴感破綻なし） | 5 % パケロス |
| MOP-6 | 入力フェーズ CPU 時間（1 周期） | ≤ 2 ms |
| MOP-7 | 起動時間（電源投入 → 演奏可能） | ≤ 5 s |
| MOP-8 | 強弱制御 分解能（ストレッチ） | 3 段階以上（p / mf / f） |

目標値は原案段階のたたき台。実機計測後にテスト設計で再調整する想定。
MOE（有効性指標）の合意値は ADR-0006 で「楽器間同期誤差 20 ms（および音階の正確性）」と定めている。

## 2. 各 MOP 目標値の根拠

各 MOP の数値は、演奏体験として「合っている／追従している」と感じられる水準と、
採用ハードウェア（XIAO ESP32-S3 Sense + MPU6050、SoftAP + UDP マルチキャスト）で
達成可能と見込まれる水準の、両者が成立する範囲で設定した。

### MOP-1（楽器間 同期誤差 ≤ 20 ms）

合奏における演奏者間タイミング調整の心理学的研究をまとめた河瀬 (2014) [1] は、
Rasch (1979) や Keller & Appel (2010) などの先行研究を引いて、
「演奏者どうしのズレが数 10 ms 程度ならば、演奏者・聴き手ともズレているとは感じない」一方で
100 ms を超えると合っていないと判断されることを整理している。
本プロジェクトでは ADR-0006 で、この「数 10 ms」のなかでも安全側に倒した 20 ms（および音階の正確性）を
MOE（有効性指標）の合意値と定めており、MOP-1 はこれをそのままブレイクダウンしたもの。
同一 SoftAP 配下の LAN 内通信で物理距離由来の遅延は無視できるため、
マスタクロック方式と BEAT 連送（`beatRedundancy = 2`）の組み合わせでこの水準を狙う。

### MOP-2（通信遅延 ≤ 10 ms）

本システムでは BEAT 受信後に楽器ノードが `playAtMasterMs` まで保持して発音する
マスタクロック方式を採るため、通信遅延そのものは MOP-1（同期誤差）にほぼ吸収される。
したがって MOP-2 は MOP-1 の許容範囲（20 ms）の半分程度を割り当てた設計マージンとして 10 ms を採る。
ネットワークを介した遠隔合奏・遠隔合唱の実験報告 [2] でも、通信遅延は演奏の同時性を損なう主要因として
繰り返し議論されており、逆に言えば同一 SoftAP 配下・UDP マルチキャストで生じる十数 ms オーダーの遅延は、
広域ネットワークを介した実験系（数十〜数百 ms 規模になりがち）と比べて十分小さい。

### MOP-3（拍検出 正解率 ≥ 90 %）

加速度ベースの指揮ジェスチャ認識は、HMM とファジィ推論を組み合わせたマルチモーダル指揮シミュレータの
先行研究 [3] で既に実用域の認識精度が示されている。新山王 (2004) [4] による指揮基本動作の分析でも、
熟達者と初心者で打点の特徴差が定量化されており、加速度ノルム閾値による振り下ろし検出は
実装容易性の高い手法として裏づけられる。本ハッカソンでは演奏破綻を起こさない実用水準として、
IR / 分類器評価で「実用域」とされる 90 % を目標にした。

### MOP-4（テンポ追従 ≤ 2 拍）

指揮基本動作の分析 [4] が示すとおり、指揮の打点と拍点の関係から人間の指揮者・演奏者は
1〜2 拍以内でテンポ変化を追従する。本システムはこの「人間と同等の追従感」を最低水準とし、
BPM 推定に EMA（指数移動平均）を用いるため、2 拍以内で目標 BPM の許容誤差に収まる設計とした。

### MOP-5（5 % パケロス下で聴感破綻なし）

ITU-T 勧告 Y.1541 の日本語版である TTC 標準 JT-Y1541 [5] は、IP ベースサービスの品質クラスごとに
パケットロス率の目標値を規定しており、最も厳しいクラス 0（リアルタイム高品質）でロス率 1×10⁻³（0.1 %）、
クラス 1 / 2 / 3 で同値、クラス 4 で 1×10⁻³、クラス 5（規格外・通常通信）では未規定としている。
家庭用 IP 通信の実観測ベースの解説 [6] でもロス率 1 % 以下が良好、3 % 超で不具合が顕在化、
10 % 超でストレスを感じる水準とまとめられており、無線 LAN ではマルチキャストの取りこぼしが有線より有意に多い [7]。
本システムは BEAT 連送（`beatRedundancy = 2`）と「過去 BEAT も即発火する寛容な受信側」でパケロスを吸収する設計であり、
試験条件としては「規格目標を 1 桁以上上回る悪化条件でも演奏破綻しないこと」を確認する意味で 5 % を採用する。
（Version 0.1 で想定していた SelfRun 自走は、テスト実装で「指揮者再起動中に楽器だけ走る」等の副作用が出たため 0.2 で撤廃。）

### MOP-6（入力フェーズ CPU 時間 ≤ 2 ms）

MPU6050 の I2C サンプリングを 200 Hz（周期 5 ms）で行う設計のため、EMA の 3 フェーズ実行モデル
（入力 → `applyPattern()` → 出力）の各フェーズ平均が 5/3 ≈ 1.7 ms 以内に収まる必要がある。
入力フェーズに 2 ms 以下の制約を置けば、残り 3 ms をロジックと出力に割り当てて 5 ms 周期を破綻なく回せる。
数値は文献ではなく、サンプリング周波数とループ周期から導出した実時間制約。

### MOP-7（起動時間 ≤ 5 s）

外部文献ではなく設計上の見積もりとユーザー体験の経験則から定めた。ESP32 系における
SoftAP 起動・STA 接続・初期 DHCP 等で 2〜3 秒、指揮者ノードの Calibrating（停止時ノルムの学習）に 2 秒を要する想定で、
合計 5 秒以内に最初の CTRL を送出する。実演前のセットアップ時間としてユーザーが
「いつ振り始めてよいか」を判断できる範囲はおよそ 5 秒以内であり、体験面の閾値とも整合する。実測で再調整する。

### MOP-8（強弱 3 段階以上）

西洋音楽の強弱記号体系では p（ピアノ＝弱く）／mf（メゾフォルテ＝やや強く）／f（フォルテ＝強く）が
広く用いられる基本単位であり、3 段階あれば多くの初級〜中級曲のダイナミクス指示を最低限再現できる。
ストレッチ機能（F1.5）の初期実装としてこの 3 段階分離を達成水準と置く。

## 3. 参考文献

設計目標値（上記 MOP）の根拠として引用した日本語文献。URL はいずれも 2026 年 5 月時点で

1. 河瀬諭、「合奏における演奏者間コミュニケーション――タイミング調整とその手がかり――」、
   心理学評論、第 57 巻第 4 号、pp. 495–510、2014 年。
   <https://www.jstage.jst.go.jp/article/sjpr/57/4/57_495/_article/-char/ja/>
   （合奏のタイミング知覚に関する心理学的研究のレビュー論文。演奏者間のズレが数 10 ms 程度なら知覚されず、
   100 ms を超えるとズレと判断される旨を、Rasch (1979)、Keller & Appel (2010) などの先行研究を引いて整理。）

2. 和田武、井上洋一、田邉隆、永井明、三原義樹、「研究開発用ネットワークを利活用した遠隔合唱実験」、
   教育情報研究（日本教育情報学会）、第 27 巻第 2 号、pp. 23–29、2011 年。
   <https://www.jstage.jst.go.jp/article/jjsei/27/2/27_KJ00007706909/_article/-char/ja/>
   （研究開発用広帯域ネットワークを用いた遠隔地間の合唱実験の報告。ネットワークを介した同時演奏では
   音質と遅延が中心的な課題になることを論じている。）

3. 宇佐聡史、持田康典、「マルチモーダル指揮シミュレータ」、
   日本ファジィ学会誌、第 10 巻第 4 号、pp. 707–716、1998 年。
   <https://www.jstage.jst.go.jp/article/jfuzzy/10/4/10_KJ00002086732/_article/-char/ja/>
   （加速度センサ等を用いて指揮ジェスチャを認識し、HMM とファジィ推論で拍タイミング・テンポ変動・
   ダイナミクスをリアルタイム認識する研究。）

4. 新山王政和、「指揮基本動作における初心者と熟達者の動作タイミングの違いに関する分析的研究
   －拍点と打点の関係、及び速度変化に基づく指揮基本原則の再考察－」、
   音楽教育学、第 34 巻第 2 号、pp. 1–12、2004 年。
   <https://www.jstage.jst.go.jp/article/jjomer/34/2/34_1/_article/-char/ja/>
   （熟達指揮者の打点が拍点に先行すること、加速度の上下方向比など、拍検出・テンポ追従の設計上の手がかりとなる
   定性的特徴を提示。）

5. 情報通信技術委員会（TTC）、「JT-Y1541 IP ベースサービスにおけるネットワーク性能目標」
   （ITU-T 勧告 Y.1541 日本語標準）。
   <https://www.ttc.or.jp/application/files/1915/5435/3364/JT-Y1541v2.pdf>
   （IP ネットワークの品質クラスごとに、遅延・ジッタ・パケットロス率の目標値を規定する国際勧告の日本語標準版。
   最も厳しいクラス 0 でロス率目標 1×10⁻³。）

6. オプテージ（eo 光）、「パケットロス（パケロス）とは？発生原因や対策を解説」。
   <https://eonet.jp/column/recommend/what-is-packet-loss.html>
   （家庭用 IP 通信の実観測ベースで、パケットロス率の目安として 1 % 以下＝良好、3 % 超＝遅延や不具合、
   10 % 超＝ストレスを感じる水準と整理。）

7. DeNA Engineering Blog、「無線 LAN の通信品質を見える化する話」。
   <https://engineering.dena.com/blog/2018/12/wireless-lan-quality-visualize/>
   （無線 LAN におけるマルチキャストの取りこぼしや有線比でのロス増加が有意に観測されるという
   企業エンジニアリング部門による計測事例。）
