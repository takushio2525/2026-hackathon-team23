## 0. 結論（先に）

**このまま「計画書」として出すと、中身以前に「テンプレートに沿っていない」で減点される可能性が高い。**
最大の問題は技術論ではなく構成で、配布テンプレートが第1章として明確に要求している
**「プロジェクトの計画書」（概要・目的・独自性／スコープマネジメント／成果物／FBS・PBS・対応関係／WBS／アローダイアグラム・クリティカルパス／役割分担／ガントチャート／V&V計画（MOE・MOP・TPM）／資源調達管理／リスク管理）がまるごと欠落**している。提出物は「システムの基本設計」からいきなり始まっている（`\setcounter{page}{1}` の位置からも、計画書章を意図的に外したのが分かる）。サブタイトルは「～計画書と設計書～」、ファイル名は「提出用計画書」のまま。

- → **もし「計画書パートはチームの別メンバーが別ファイルで出す／塩澤は設計書パートだけ担当」という分担なら、以下 §1 は読み飛ばしてよい。** そうでなくこの1冊を「計画書」として出すなら、§1 が最優先。これは本人にしか分からないので、ここを最初に確かめてほしい。
- 技術的な不安要素・不整合は §2 以降。こちらは「致命的ではないが、突っ込まれると弱い」レベル。特に **§2-1（楽譜進行の脆さ）と §2-2（test_v2 との乖離）** は、先生が `firmware/` のコードも見るなら確実に聞かれる。

## 2. 設計の不安要素（手当て後に残った／新たに見えたもの）

### 2-1.【設計の根幹】楽譜進行が「BEAT 1発で index +1」の純カウント駆動 — BEAT を1回落とすとその楽器だけ永久にズレる

§2.4.3「楽譜進行（純粋 index 駆動）」: 「発火イベントが立っていれば `kScore[currentEventIndex]` を発火し `currentEventIndex` を 1 進める」。`beatAt` は「参考値で発火条件には使わない」と**わざわざ**書いてある。

一方で BEAT ペイロードには `beatNo`（uint16_t, 曲頭からの拍番号）を載せている。**送っているのに、楽器側の進行に一切使っていない。** これだと:

- BEAT が 1 個落ちる（`beatRedundancy=2` の連送が両方落ちる）→ その楽器の `currentEventIndex` だけ 1 個遅れる → **以降ずっと 1 拍ズレたまま**。次に BEAT が来ても「index を 1 進める」だけなので自己修復しない。
- しかも「過去 BEAT も即発火・捨てない」方針なので、遅れて届いた BEAT は素直に消化されるが、**完全に欠落した分は永久に欠落**。

`firmware/test_v2/` の実装はここが違っていて、`firedBeatNo`（= 受信 BEAT の拍番号）と `headRestBeats` から「楽譜のどのインデックスを鳴らすべきか」を**毎回計算**している（`applyPattern.cpp`: `effective = firedBeatNo - 1 - headRestBeats`）。これなら 1 個落ちても次の BEAT で正しい位置に復帰する（自己同期）。**計画書の設計は test_v2 より退化している。**

→ どちらかにすべき。(a) BEAT の `beatNo` を進行の正にする（test_v2 方式・推奨）、(b) どうしても純カウント駆動でいくなら「なぜ beatNo を使わないか（ループ再生時の beatNo の扱い等）」を書き、「1 個でも落ちたら復帰しない」リスクを §リスク管理に明記する。今は「index 駆動」と書いてあるだけで判断の理由が無く、しかも実装（test_v2）と食い違っている。

### 2-2. test_v2 で実際に動かした方式が計画書に反映されていない（第1回 B-1 が「言及削除」で回避されただけ）

第1回レビュー B-1 / D-5 で「test_v1 と test_v2 のどちらを正にするか決める／R-2 が輪唱だから test_v2 を反映すべき」と指摘されていた。対応方針は「テスト実装への言及を全削除」。結果、**test_v2 の存在が本文から消えただけで、設計内容は古いまま**:

| 項目 | 計画書(提出版) | test_v2 実装 |
|---|---|---|
| 楽譜進行 | `currentEventIndex` を BEAT で +1（純カウント） | `firedBeatNo` から実インデックスを算出（自己同期） |
| 輪唱の頭ずらし | `startBeatNo`（例: 0 / 4 / 8） | `headRestBeats`（0 / 8 / 16） |
| NOTE ペイロード | `partId`（0x02–0x05、説明は「楽器番号」）のみ | `partId`（ノード識別）＋ **`instrumentId`（PC 側の楽器定義インデックス, 0..N-1）** の 2 本立て |
| 設定キー名 | `startBeatNo` | `headRestBeats` |

「test_v2 はあくまで試作で本番はこの計画書の設計でいく」なら、それを一文書けばいい（「試作 test_v2 では beatNo 駆動を試したが本番は…／本番は instrumentId を分離する／しない」）。今は整理がないので、test_v2 のコードと並べると「計画書が古い」「実装と仕様が乖離」に見える。CLAUDE.md にも「コードと仕様書は常に同期」とあり、`firmware/test_v2/` も見られる前提なら触れられる。

### 2-3. 「同期誤差 ≤ 20ms」（TP-3 合否基準）と「遅延無制限で即発火・厳密同期は保証しない」（§clock-sync）が依然ねじれている

第1回 A-1 への手当てで「best-effort、ある 1 台だけ大きく遅れたら一時的にずれる、厳密な台間同期は保証しない」と正直に書いた（これは良い）。ただ TP-3 の合否基準は「最大差 ≤ 20ms」のまま。**設計が「保証しない」と言っているものを、テストが「20ms 以内であること」と判定する** → テストの位置づけが宙ぶらりん（落ちても「想定内」と言える＝テストとして機能しない）。

直し方は 2 択: (i) TP-3 を「定常状態（同期収束後・全台が `waitMs > 0` で受信できている拍）に限定して最大差 ≤ 20ms」と条件を明示、(ii)「20ms はあくまで目標値、未達でも『どれか 1 発が lookahead 内に届いた拍の割合 ≥ X%』を主合否基準にする」。今のままだと「20ms 狙う」と「保証しない」が同じ文書に並んで読み手が混乱する（第1回 A-1 で「矛盾に見える」と書いた件が、文章は丁寧になったが構造としては未解消）。

### 2-4. マスタクロックの「片道遅延が 4 台でほぼ同じ」仮定 — 崩れたときの代替が「パラメータ調整」止まり

§clock-sync:「この片道遅延が 4 台でほぼ同程度であれば相対同期は保たれる。…崩れるため 4 台の同期誤差は実機で確認し（TP-3）、必要なら `beatRedundancy` や `beatLookaheadMs` を調整する」。

正直に書けてはいるが、**`beatRedundancy` も `beatLookaheadMs` も「片道遅延の系統差」は消せない**（連送は「届くか」を上げるだけ、lookahead は「待つ時間」を増やすだけで、台ごとの offset 推定値そのもののズレは残る）。「崩れていたらこうする」の中身が実質ない。本来の代替は「往復測定で各台の片道遅延を個別に較正して offset に足し込む」あたりだが、それは設計に無い。せめて「系統差が出たら各台に固定補正値を持たせる」程度を §tbd かリスク欄に。

### 2-5. 指揮者ノード 1 台への集中 — TP-6 の測り方では CPU 余裕を見たことにならない

TP-6 が「楽器ノードと指揮者ノード両方で入力フェーズ所要時間を計測」に拡張されたのは改善（第1回 A-4 への手当て）。ただ指揮者は SoftAP（ビーコン送信・STA 管理・DHCP）＋ IMU 200Hz ＋ 拍検出 ＋ CTRL 20Hz/BEAT 送信 を 1 チップで回す。**SoftAP/WiFi のビーコン・割り込み処理は ESP-IDF のタスク/ISR 側で走り、「`applyPattern()` の入力フェーズ所要時間」には現れない**。だから「入力フェーズ ≤ 2ms」が満たせても、IMU の I2C 読みがビーコン処理に割り込まれてサンプル落ちする、という別問題は検出できない。指揮者側は「IMU の実サンプリング間隔のジッタ（200Hz=5ms に対し何 ms ばらつくか）」を見る指標を別に立てるべき。

### 2-6. 「1 ループ 1 NoteOn」が既知バグのまま — 計画書なのに「いつ直すか」が無い

§2.4.3 末:「`noteOut` は 1 スロットしか持たないため、細分音符と 4 分音符が偶然同一ループに重なると後勝ちで上書き…本番までに 2 スロット化 or 繰り越しのいずれかで安全側に倒す予定」。「予定」止まり。輪唱で 8 分音符（細分音符）を使うなら普通に踏む。計画書ならスケジュール（無い…）に「いつ直す」を入れるか、せめて §tbd の表に 1 行。

### 2-7. ドラム node_05 が未実装で「いつ」が読めない

§tbd:「主旋律 3 パートの完成後に着手」。R-2 の必達要件が「主旋律 3 ＋ リズム 1 **以上**」なら、ドラムは必達側。「後で」では計画として弱い。ガントチャートが無いので「主旋律 3 パートの完成」がいつ来るのかも本文から分からない（第1回 A-7 と同じ穴。スケジュールが無い以上ふさがらない）。

### 2-8. TP-2（通信遅延）の試験が「設計に存在しない機能」を前提にしている

TP-2 手順:「指揮者が BEAT を送出 → 楽器が即 ACK を返す → 指揮者で受信、の往復時間を node_01 の `millis()` で計測」。

この設計には**楽器 → 指揮者の経路（ACK）が無い**。メッセージは CTRL/BEAT（指揮→楽器, UDP マルチキャスト）と NOTE（楽器→Mac, USB）の 3 種だけ。「楽器が ACK を返す」には新しいパケット種別と、楽器側→指揮者側への UDP ユニキャスト送信を実装しないといけない。テストのためだけにプロトコルを足すのは構わないが、それなら §test-design か §tbd に「TP-2 用に楽器→指揮者の ACK パケットを試験ビルドにだけ実装する」と明記しないと手順だけ宙に浮く。第1回 A-8 の「ループバックは経路を測れていない」は直ったが、直した先が「無い機能」になっている。

### 2-9. TP-3 は `millis()`（1ms 粒度）＋`loopIntervalMs=5ms` 間引き — TP-6 は `micros()`。粒度がちぐはぐ

TP-3 で「発音時刻は `millis()` で記録、これより細かい分解能は扱わない／方式上は 20ms より小さく収まる見込み」と書いたのは第1回 A-9 への手当てとして正直。ただ TP-6 は `micros()` を使う。同じテスト設計の中で時間分解能の扱いが揃っていない。TP-3 も「同期誤差を語るなら `micros()` で測る」か、せめて「`millis()` 粒度（±1ms）＋5ms 間引きの粒度で『20ms 以内に収まること』を確認する（それ未満の差は議論しない）」と書いて TP-6 と方針を合わせる。「方式上は 20ms より小さく収まる見込み」は願望なので落とすか、根拠（lookahead 50ms に対し連送 2 発のどれかが届けば…）を書く。

### 2-10. パケロス耐性 TP-5 が「聴感破綻なし」だけ — `seq` を「将来用予約」と正直にした副作用

第1回 A-10 への手当てで `seq` を「現設計では送出のみ／将来用の予約」と正直に書いた（良い）。が、結果として TP-5（5% パケロス）の合否が「耳で聴いて破綻なし」だけになり、定量評価する手段が無い（落ちた BEAT 数・楽譜ズレ量を数える仕組みが無い）。試験ビルドで「捨てた BEAT 数」と「楽譜 index のズレ」をログに出すくらいは設計しておくと、TP-5 が「測れるテスト」になる。

## 3. おかしな記述・内部不整合（doc 内）

### 3-1. `beatRedundancy` の既定値が「2」と「1」で二重

- §1.3.3:「同一 `beatNo` を `beatRedundancy` 回（既定 **2**）連送」
- §1.3.4 パケロス対策:「同一 BEAT を `beatRedundancy` 回（既定 **2**）連送」
- §2.4.x `OrcNetData` 送信側の表:「`pendingBeatRedundancy` … 同一 BEAT を何発まで連送するか（既定 **1**）」
- §`ORC_SENDER_CONFIG`:「`beatRedundancy` = **2**」
- §2.4.2 step2:「`pendingBeatRedundancy = beatRedundancy`」（＝実行時は 2 になる）

「`OrcNetData` のフィールド初期値が 1、設定 `beatRedundancy` で 2 に上書き」という意図は読めるが、表に「（既定 1）」と書くと「連送 1 回＝連送しない」に読めて混乱する。`OrcNetData` の表は「（`OrcSenderModule`／`applyPattern()` が `beatRedundancy` を入れる。未設定時 1）」と書くか、初期値も 2 に揃える。

### 3-2. `partId` を 2 か所で別々に持っている

`ORC_RECEIVER_CONFIG.partId`（0x02）と `NOTE_SENDER_CONFIG.partId`（0x02）が別フィールド。同じ値を 2 か所に書くので、ノード追加・差し替え時に片方だけ直し忘れるとズレる。`ProjectConfig` 集約の趣旨（「具体値はこの 1 ファイルでだけ管理」）からすると、`partId` は 1 個の定数にして両 Config がそれを参照する形が自然。設計の小さな綻び。

### 3-3. 「Mac 1 台が 4 ポート受け」フォールバックと「Processing は 1 ポートだけ開く」が矛盾

部品表 P4 とネットワーク構成節に「Mac が 4 台揃わなければ 1 台で 4 ポートを受ける構成にフォールバックできる」と書いたが（第1回 A-5 への手当て）、別の箇所では「Mac 側 Processing は自分に繋がっているシリアル 1 ポートだけを開く」と断言している。フォールバック時は Processing 側を「4 ポート開く」に変える必要があり、それは「梅澤担当・スコープ外」。「フォールバック可能」と書くなら「その場合 Mac 側の改修が要る（スコープ外）」と添えないと、できる風に見えて中身が無い。

### 3-4. `magic = 0x4F52`（"OR"）と「リトルエンディアンでシリアライズ」「Mac 側は magic を探して同期」が、バイト列のどっち向きか曖昧

`uint16_t magic = 0x4F52` をリトルエンディアンで書くと、バイト列は `0x52, 0x4F`（"R","O" の順）。Mac 側 Processing が「バイトストリームから magic を探す」とき `0x4F 0x52` を探すのか `0x52 0x4F` を探すのかは実装次第で、取り違えると延々フレーム同期できない（実装時に踏む典型）。「2 バイト読んでリトルエンディアンで `uint16` にして `== 0x4F52` を確認」と書けば曖昧でなくなる。優先度は中だが、フレーミングの要なので一文足す価値あり。

### 3-5. 役割分担が本文に散らばっていて一覧が無い

「塩澤＝node_01〜05 のファームウェア」「梅澤＝Mac の音色合成（参考）」「齋藤＝楽譜エディタ」が、それぞれ別の段落・表に断片的に出てくるだけ。テンプレ 1.3.3「役割分担」の表を 1 つ作って集約すべき（§1-1 と重複指摘）。「齋藤担当」と個人名が出ているのに、その人がチームの誰で何担当かが本文のどこにも整理されていない。

## 4. 「嘘」寄り・確証の弱い技術記述

第1回 D-1〜D-5（`WiFi.beginAP()`、SPI ブリッジ、AT bridge、80ms、test_v2 反映漏れ）は本文上は手当て済み（`softAP()`／UART／80ms 削除…）。残っているもの:

### 4-1. 「STA 側の WiFi 省電力を無効化して運用（具体的な設定方法は実機で確認）」 — 生命線なのに方法が未確定

§hw-design 末:「楽器ノード(STA)の WiFi 省電力モードは無効化して運用する（具体的な設定方法は実機で確認する）」。マルチキャストの DTIM 配送遅延を避けるための**唯一の対策**がこれなのに、「`WiFiS3` でそれを切る API があるか」自体が確認できていない（第1回 A-3 で「`WiFi.noLowPowerMode()` を明示的に呼ぶ必要」と指摘されたが、コード例も API 名も本文に無い）。本当に切れなかった場合（≒ 〜100ms の配送遅延が常時乗る）、マスタクロック方式の `beatLookaheadMs=50ms` を全部食い潰す。§tbd かリスク欄に「`WiFiS3` で省電力 OFF にできなかった場合の代替（lookahead を 150ms に広げる／指揮者側でユニキャスト連送に切り替える 等）」を 1 行。

### 4-2. 「USB CDC ではボーレートは実質無視されるが両端で値を揃えておく」 — 「揃える」に実益がほぼ無い

事実関係（CDC ACM はボーレート無視）は正しい。ただ「揃えておく」は揃えても揃えなくても動くので、`NoteSenderConfig.baudRate` を「集約する設定値」として持つこと自体が形だけ。害は無いが、「115200 に意味があるかのよう」に複数箇所で繰り返し書いてあるのは冗長。1 か所で「CDC では無視される（形式上 115200 を入れておくだけ）」と書けば足りる。

### 4-3.（軽微）`beatNo` の `uint16_t` とループ再生

`beatNo`（曲頭からの拍番号, uint16_t, 「曲開始時に 0 リセット」）。120BPM で 65535 拍 ≒ 9 時間なのでハッカソンの曲長では溢れない。ただ「曲を末尾→0 でループ再生」する設計なのに「ループ時に beatNo がどうなるか」は書いていない。`currentEventIndex` 駆動なので今は実害が出ないが、§2-1 で beatNo 駆動に直すなら、ループ時の beatNo の扱いを定義する必要が出る。
