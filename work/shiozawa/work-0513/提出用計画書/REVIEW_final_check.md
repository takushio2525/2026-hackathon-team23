## 1. §2.5「テストの設計」と「要件」の置き場所

### 1-1. §2.5 をどうするか

試作テストの話を提出物に書かない方針なら、§2.5 の中身——「PlatformIO の `test/` を使い…単体テストを整備する」「ゴールデンテスト」「結合テスト Tier T1〜T4」「TP-1〜TP-8 を実機で計測」「実機で確定する仮値の表」——は、いずれも「これからテストする」という**計画**で、提出範囲（基本設計・詳細設計）にも方針にも収まりにくい。選択肢:

1. **§2.5 をまるごと削除。** 一番シンプル。ただし §tbd（実機で確定する仮値）が「T1〜T4 で詰める」と §2.5 を参照しているので、その参照を「実装後の調整で詰める」などに書き換える。「実機で確定する仮値」の表自体は設計の前提（仮値の一覧）として残しても良い——その場合は「T3 で」「TP-8 で」のような結合テスト Tier 参照だけ落とす。
2. **§2.5 を「設計が満たすべき要件＋将来の検証の方向性」に縮める。** TP-1〜TP-8 の「目標／合否基準」列だけ抜き出して**性能要件の表**にし、「手順」「段階」「単体テスト表」「結合テスト Tier 表」は落とす。要求 R-1〜R-4 は「要求一覧」として残す（「検証方法」の列は薄めるか落とす）。

どちらに倒すか先に決める。今のまま「テストを書く・実機でこう測る」が残っていると、方針（試作テストは提出物に書かない）と整合しない。

### 1-2. 「要求・数値目標」はどこに書くか（＝「そこら辺の要件も描いておいて」）

§2.5 を縮める／消すなら、その中に埋もれている「このシステムが満たすべきもの」は、設計文書として普通に書いておくべき。具体的には:

- **機能要求**（今 §2.5 の対応表に R-1〜R-4 / R-Internal-1〜3 として出てくるもの）: 「5台同期演奏」「輪唱曲（主旋律3＋リズム1以上）」「可変テンポ追従」「(ストレッチ)強弱制御」「通信の信頼性」「CPUリソース」「起動時間」。基本設計の §1.1（機能一覧）の前か直後に「要求一覧」の小節を立てて置く。今は要求 ID が**定義なしに**対応表で唐突に出てくるので、読み手は R-1 が何だったか遡れない。
- **数値目標（設計目標値）**: 同期誤差 ≤ 20ms（ADR-0006 の MOE）／拍検出 ≥ 90%／テンポ追従 ≤ 2拍／入力フェーズ CPU ≤ 2ms／起動 ≤ 5s／強弱3段階。これは「テスト合否基準」ではなく「設計が狙う値」として書く。出どころ（ADR-0006／`MOPと参考文献メモ.md` の根拠）を各行1〜2行で添えると、なぜその値かが伝わる。§clock-sync が `beatLookaheadMs=50ms` の妥当性を語るとき暗黙にこの「20ms」を前提にしているので、本文のどこかに定義は必ず要る。
- これらは「計画書の MOP/TPM」ではなく「基本設計が前提とする要件」として書けば、今回の提出スコープ（基本設計・詳細設計）の中に自然に収まる。MOP メモの参考文献7本も、数値目標の根拠としてここに添えれば活きる（捨てるのは惜しい）。

## 2. 設計の不安要素

### 2-1.【設計の根幹】楽譜進行が「BEAT 1発で index +1」の純カウント駆動 — BEAT を1回落とすとその楽器だけ永久にズレる

§2.4.3「楽譜進行（純粋 index 駆動）」: 「発火イベントが立っていれば `kScore[currentEventIndex]` を発火し `currentEventIndex` を 1 進める」。`beatAt` は「参考値で発火条件には使わない」と**わざわざ**書いてある。

一方で BEAT ペイロードには `beatNo`（uint16_t, 曲頭からの拍番号）を載せている。**送っているのに、楽器側の進行に一切使っていない。** これだと:

- BEAT が 1 個落ちる（`beatRedundancy=2` の連送が両方落ちる）→ その楽器の `currentEventIndex` だけ 1 個遅れる → **以降ずっと 1 拍ズレたまま**。次に BEAT が来ても「+1」だけなので自己修復しない。
- 「過去 BEAT も即発火・捨てない」方針は**遅れて届く**ぶんには強いが、**完全に欠落した**ぶんには無力。
- 楽器ごとに独立にズレるので、運悪く 1 台だけ落とすと、その 1 台だけ曲がズレた状態で復帰せず流れ続ける。

→ BEAT に `beatNo` を持たせている以上、楽器側は「受信した `beatNo` から鳴らすべきインデックスを算出」する方が頑健（1 個落ちても次の BEAT で正しい位置に復帰する＝自己同期）。純カウント駆動でいくなら、なぜ `beatNo` を使わないか（曲ループ再生時の `beatNo` の扱い、`startBeatNo` オフセット計算の単純化、など）を書き、「1 個落ちたら復帰しない」を設計上の制約として明記すべき。今は「index 駆動」とだけ書いてあって判断の理由が無く、しかも BEAT ペイロードに `beatNo` がある以上「使えるのに使っていない」状態。

### 2-2. 同期方式 — 「同期誤差 ≤ 20ms（設計目標）」と「best-effort で厳密同期は保証しない（§clock-sync）」がどの条件で両立するかを一文で切り分ける

§clock-sync は「過去時刻の BEAT も即発火、ある 1 台だけ大きく遅れたらその台の発音は一時的にずれる、厳密な台間同期は保証しない（best-effort）」と正直に書けている（良い）。ただ設計目標としては「同期誤差 20ms」（ADR-0006）があり、§clock-sync 自身も `beatLookaheadMs=50ms` の妥当性をこの水準感で語っている。「20ms を狙う」と「遅延無制限で即発火だから保証しない」が同じ文書に並ぶと矛盾に見える。

→ 「**定常状態（同期収束後・全台が `waitMs > 0` で受信できている拍）では 20ms 以内に収まる見込み。受信が lookahead を大きく超えた拍は、その台だけ一時的にずれる（鳴らないより優先）**」のように、どの条件で 20ms を狙うのかを一文で切り分ける。これで「目標値」と「best-effort の限界」が衝突しなくなる。

### 2-3. マスタクロックの「片道遅延が 4 台でほぼ同じ」仮定 — 崩れたときの代替が「パラメータ調整」止まり

§clock-sync は「片道遅延が 4 台でほぼ同程度であれば相対同期は保たれる…崩れるため実機で確認し、必要なら `beatRedundancy` や `beatLookaheadMs` を調整する」とする。正直に書けてはいるが、**`beatRedundancy` も `beatLookaheadMs` も「片道遅延の系統差」は消せない**（連送は到達率を上げるだけ、lookahead は待ち時間を増やすだけで、台ごとの offset 推定値そのもののズレは残る）。「崩れていたらこうする」の中身が実質ない。本来の代替は「往復測定で各台の片道遅延を個別に較正して offset に足し込む（各台に固定補正値を持たせる）」あたり。それを §clock-sync か §tbd に 1 行。

### 2-4. 指揮者ノード 1 台への機能集中 — 監視すべき指標が「入力フェーズ所要時間」では足りない

指揮者は SoftAP（ビーコン送信・STA 管理・DHCP）＋ IMU 200Hz ＋ 拍検出 ＋ CTRL 20Hz/BEAT 送信 を 1 チップで回す。SoftAP/WiFi の処理は ESP-IDF のタスク/ISR 側で走るので、`applyPattern()` の所要時間や「入力フェーズ所要時間」には現れない。だから入力フェーズが軽くても、IMU の I2C 読みがビーコン処理に割り込まれてサンプルが落ちる・周期が乱れる、という別問題は見えない。指揮者側で気にすべきは「**IMU の実サンプリング間隔のジッタ**（5ms に対して何 ms ばらつくか）」で、設計文書としては「これを許容範囲内に保つ／監視する」と一言あると、機能集中のリスクに目配りしていることが示せる。

### 2-5. 「1 ループ 1 NoteOn」が既知の制約のまま「本番までに直す予定」止まり

§2.4.3 末:「`noteOut` は 1 スロットしか持たないため、細分音符（8 分音符）と 4 分音符が偶然同一ループに重なると後勝ちで上書き…本番までに 2 スロット化 or 繰り越しのいずれかで安全側に倒す予定」。輪唱で 8 分音符を使うなら普通に踏む条件。設計文書なら「どちらにするか」を決め切るか、§tbd（実機で確定する仮値）の表に 1 行立てて「未確定」と明示する。

### 2-6. ドラム node_05 が「同じ `ScoreEvent` 形式で扱える」ことは示せているが、`noteNumber` の規約が未定

§tbd:「ドラム node_05 は金管同様の `ScoreEvent` で実装（仮）。`noteNumber` は GM ドラムマップ準拠を想定」。設計文書としては「ドラムも同じ楽譜形式・同じ NOTE パケットで流せる（音色の解釈は Mac 側）」ことが示せていれば十分なので、ここは大きな穴ではない。詰めるなら「`noteNumber` を GM ドラムマップ（36=バスドラム, 38=スネア, 42=クローズドハイハット …）に固定する」と書いておくと、Mac 側（梅澤）との受け渡し境界がはっきりする。

## 3. おかしな記述・内部不整合（doc 内）

### 3-1. `beatRedundancy` の既定値が「2」と「1」で二重

- §1.3.3／§1.3.4 パケロス対策:「同一 BEAT を `beatRedundancy` 回（既定 **2**）連送」
- §2.4.x `OrcNetData` 送信側の表:「`pendingBeatRedundancy` … 何発まで連送するか（既定 **1**）」
- §`ORC_SENDER_CONFIG`:「`beatRedundancy` = **2**」／ §2.4.2 step2:「`pendingBeatRedundancy = beatRedundancy`」（＝実行時は 2）

「`OrcNetData` のフィールド初期値が 1、`beatRedundancy` で 2 に上書き」という意図は読めるが、表に「（既定 1）」と書くと「連送 1 回＝連送しない」に読めて混乱する。`OrcNetData` の表は「（送信側が `beatRedundancy` を入れる。未設定時 1）」と書くか、初期値も 2 に揃える。

### 3-2. `partId` を 2 か所で別々に持っている

`ORC_RECEIVER_CONFIG.partId`（0x02）と `NOTE_SENDER_CONFIG.partId`（0x02）が別フィールド。同じ値を 2 か所に書くので、ノード追加・差し替え時に片方だけ直し忘れるとズレる。`ProjectConfig` 集約の趣旨（「具体値はこの 1 ファイルでだけ管理し、モジュール本体にハードコードしない」）からすると、`partId` は 1 個の定数にして両 Config がそれを参照する形が自然。

### 3-3. 「Mac 1 台が 4 ポート受け」フォールバックと「Processing は 1 ポートだけ開く」が矛盾

部品表 P4 とネットワーク構成節に「Mac が 4 台揃わなければ 1 台で 4 ポートを受ける構成にフォールバックできる」と書いたが、別の箇所では「Mac 側 Processing は自分に繋がっているシリアル 1 ポートだけを開く」と断言している。フォールバック時は Processing 側を「4 ポート開く」に変える必要があり、それは「梅澤担当・スコープ外」。「フォールバック可能」と書くなら「その場合 Mac 側の改修が要る（スコープ外）」と添えないと、できる風に見えて中身が無い。

### 3-4. `magic = 0x4F52`（"OR"）と「リトルエンディアンでシリアライズ」「Mac 側は magic を探して同期」が、バイト列のどっち向きか曖昧

`uint16_t magic = 0x4F52` をリトルエンディアンで書くと、バイト列は `0x52, 0x4F`（"R","O" の順）。Mac 側 Processing が「バイトストリームから magic を探す」とき `0x4F 0x52` を探すのか `0x52 0x4F` を探すのかは実装次第で、取り違えると延々フレーム同期できない（実装時に踏む典型）。「2 バイト読んでリトルエンディアンで `uint16` にして `== 0x4F52` を確認」と書けば曖昧でなくなる。優先度は中だが、フレーミングの要なので一文足す価値あり。

### 3-5. 「梅澤（参考）」「齋藤担当」が断片的に出てくるだけで、誰が何担当かの一覧が無い

役割分担表（テンプレ第1章の話）は今回スコープ外でいいが、本文中で「梅澤担当の Mac 音色（スコープ外）」「楽譜エディタ（齋藤担当）」が前置きなく出てくるので、第1章の冒頭あたりに「本書のスコープは塩澤担当のファームウェア（指揮者・楽器の全ノード）。Mac 側の音色合成は梅澤、楽譜エディタは齋藤の担当で本書スコープ外」程度の一文があると、読み手が「これ誰？」と迷わない。

## 4. 「嘘」寄り・確証の弱い技術記述

第1回 D-1〜D-5（`WiFi.beginAP()`、SPI ブリッジ、AT bridge、視聴覚同時性閾「80ms」、test_v2 反映漏れ）は本文上は手当て済み（`softAP()`／UART／80ms 削除…）。残っているもの:

### 4-1.（生命線）「STA 側の WiFi 省電力を無効化して運用（具体的な設定方法は実機で確認）」

§hw-design 末:「楽器ノード(STA)の WiFi 省電力モードは無効化して運用する（具体的な設定方法は実機で確認する）」。マルチキャストの DTIM 配送遅延を避けるための**唯一の対策**がこれなのに、「`WiFiS3` でそれを切る API があるか」自体が確認できていない（第1回 A-3 で「`WiFi.noLowPowerMode()` を明示的に呼ぶ必要」と指摘されたが、コード例も API 名も本文に無い）。本当に切れなかった場合（≒ 〜100ms の配送遅延が常時乗る）、マスタクロック方式の `beatLookaheadMs=50ms` を全部食い潰す。§tbd か §clock-sync に「`WiFiS3` で省電力 OFF にできなかった場合の代替（lookahead を 150ms に広げる／指揮者側でユニキャスト連送に切り替える 等）」を 1 行。

### 4-2. 「USB CDC ではボーレートは実質無視されるが両端で値を揃えておく」 — 「揃える」に実益がほぼ無い

事実関係（CDC ACM はボーレート無視）は正しい。ただ「揃えておく」は揃えても揃えなくても動くので、`NoteSenderConfig.baudRate` を「集約する設定値」として持つこと自体が形だけ。害は無いが、「115200 に意味があるかのよう」に複数箇所で繰り返し書いてあるのは冗長。1 か所で「CDC では無視される（形式上 115200 を入れておくだけ）」と書けば足りる。

### 4-3.（軽微）`beatNo` の `uint16_t` とループ再生

`beatNo`（曲頭からの拍番号, uint16_t, 「曲開始時に 0 リセット」）。120BPM で 65535 拍 ≒ 9 時間なのでハッカソンの曲長では溢れない。ただ「曲を末尾→0 でループ再生」する設計なのに「ループ時に `beatNo` がどうなるか」は書いていない。`currentEventIndex` 駆動なので今は実害が出ないが、§2-1 で `beatNo` 駆動に直すなら、ループ時の `beatNo` の扱いを定義する必要が出る。
