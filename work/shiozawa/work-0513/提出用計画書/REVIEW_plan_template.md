# `plan_template.tex` レビュー（設計の不安要素・不整合・不確定要素・事実誤認）

対象: `work/shiozawa/work-0513/提出用計画書/plan_template.tex`
照合: `firmware/test_v1`・`firmware/test_v2`・`firmware/production`・各 `platformio.ini`
注記: MOP 節は別セッションで削除作業中。本レビューでは MOP 固有の項目は軽く触れる程度にし、それ以外を中心に挙げる。
各項目は別セッションで個別対応していく前提のチェックリスト。

## A. 設計そのものの不安要素

- [ ] **A-1. 「過去 BEAT は遅延量に関わらず即発火」と MOP-1（同期誤差 ≤ 20 ms）が原理的に両立しない場面がある。**
  受信が大きく遅れた 1 台だけ先に鳴る → 台間ズレ。doc 自身が「将来 `expiredGraceMs` を再導入する余地」と書いており、現設計は同期を保証していない（best-effort）。`beatLookaheadMs = 50 ms` を超える受信遅延が 1 台でも出れば「ほぼ揃う」前提が崩れる。「鳴らない」回避を優先した判断自体は妥当だが、「20 ms を狙う」と「遅延無制限で即発火」が同じ文書に並ぶのは矛盾に見える。トレードオフを正面から書く。

- [ ] **A-2. マスタクロックの系統誤差を「4 台とも同じだけずれる」と仮定している（§clock-sync）。**
  片方向放送＋EMA の offset 推定。楽器側 UNO R4 WiFi は RA4M1 ↔ ESP32-S3 の内部ブリッジ越しに受信するため、そのブリッジ遅延ジッタの分布が台ごとに違えば「同じだけずれる」は崩れ、相対同期も崩れる。T3 で測ると書いてはあるが、生命線なので「測ったらズレていたときの代替案」が欲しい。

- [ ] **A-3. SoftAP のマルチキャスト配送遅延への対策の説明が因果逆。**
  AP は省電力 STA 向けにグループ宛フレームを DTIM ビーコンまでバッファする（ビーコン 100 ms → 最大 ~100 ms 遅延）。対策「STA 側の省電力を無効化」は方向は正しいが、(a) `WiFiS3` の既定が省電力 ON なら「`WiFi.lowPowerMode()` を呼ばない」では不十分で `WiFi.noLowPowerMode()` を明示的に呼ぶ必要がある、(b) doc の「SoftAP はマルチキャストをキャッシュしない」は説明が逆向き（キャッシュする＝遅延／しない＝欠落、どちらにせよ省電力 OFF が答え）。

- [ ] **A-4. 指揮者ノード 1 台への機能集中。**
  SoftAP ＋ IMU 200 Hz ＋ 拍検出 ＋ CTRL/BEAT 送信を全部 node_01 が担うのに、CPU 予算（MOP-6）は楽器側しか定義がない。ビーコン処理と IMU 周期が干渉したときの余裕が見えない。

- [ ] **A-5. 本番は独立 PC（Mac）4 台が前提なのに調達計画が無い。**
  「Mac × 4」「USB ケーブル 5 本」と台数だけ書いてあるが、当日 4 台に Processing を立て 4 ポートを割り当てる作業の現実性、調達できなかった場合のフォールバック（1 台で 4 ポート受け）の検討が無い。

- [ ] **A-6. 「1 ループ 1 NoteOn」制約が未解消の既知バグ。**
  細分音符と 4 分音符が同一ループに重なると後勝ち上書き。doc 自身「本番までに 2 スロット化 or 繰り越しで安全側に」＝未対応。輪唱で細分音符を使うなら効いてくる。

- [ ] **A-7. 必達 R-2「輪唱曲（主旋律 3 ＋ リズム 1 以上）」に対しドラム node_05 が未実装。**
  §tbd で「楽譜とパーカッション音色の合意後に実装」止まり。いつ実装するかが本書から読めない（C-1 とも関係）。

- [ ] **A-8. TP-2（通信遅延）の試験が実際の経路を測っていない。**
  「node_01 が BEAT を送って自身でループバック受信」は AP の内部経路で、空中区間と STA 側ブリッジ遅延を含まない。MOP-2 が本来見たい「指揮者 → 楽器」を検証できていない。

- [ ] **A-9. 「実機では 5 ms 以下を目標」が計測手段の分解能を下回る。**
  発音時刻を `millis()`（1 ms 粒度）で記録し、楽器ループは `loopIntervalMs = 5 ms` 間引き。原理的に ~5 ms 未満は分解できない。MOP-6 のように `micros()` を使うか、「5 ms 粒度で揃うことを確認」と表現を落とす。

- [ ] **A-10. ヘッダの `seq`（「パケロス検出用」）が宣言だけで未使用。**
  欠落を検出・記録するロジックの設計が無く、MOP-5 の試験も耳で聴くだけ。使う側も設計するか、「将来用の予約」と正直に書く。

## B. おかしな記述・不整合（doc 内 / コードとの乖離）

- [ ] **B-1.（最優先）`test_v1` と `test_v2` の取り違え／混在。**
  doc は本文・改訂ノートで「テスト実装＝`firmware/test_v1/`」と明言し、楽譜例も test_v1 の「ドレミファミレド／木管 1」。だがリポジトリには後発の **`firmware/test_v2/`（きらきら星 全声部同一楽譜の輪唱 ＋ `NotePayload.instrumentId`＝楽器番号 ＋ `headRestBeats`）** がある（コミット時系列: test_v1 リネーム → test_v2 追加 → doc を test_v1 に整合）。結果として doc は以下がズレている:
  - **NOTE ペイロードに `instrumentId`（楽器番号）が無い。** test_v2 は旧 `reserved[0]` を `instrumentId` に充てている。プロジェクト名が「楽器番号付き NOTE」を含むのに本書のパケット定義は古い形のまま。
  - **輪唱の声部ずれ: doc は 4 拍／8 拍、test_v2 は 8 拍／16 拍**（`headRestBeats` = node_03:8, node_04:16）。
  - **設定キー名: doc は `startBeatNo`、test_v2 は `headRestBeats`。**
  - **「金管 3 ＋ ドラム 1」という楽器構成**が、test_v2 の「声部＝楽器番号で音色だけ切り替える（全員同じ楽譜）」モデルと噛み合わない。
  → 「test_v1 を正とする／test_v2 を正とする」のどちらに寄せるかを最初に決め、寄せた側に統一する。R-2 が「輪唱曲」で、輪唱を実装したのは test_v2 なので、本来は test_v2 を反映すべきに見える。**この決定が B-1〜B-5 の前提になる。**

- [ ] **B-2. AP 起動 API の混在 `WiFi.beginAP()` vs `WiFi.softAP()`。**
  node_01 は Arduino-ESP32（XIAO ESP32-S3）。正しいのは `WiFi.softAP(ssid, pass)`。`WiFi.beginAP()` は WiFiNINA/WiFiS3 の API で ESP32 側には無い（§通信仕様の表と `OrcNetModule.init()` の説明が `beginAP()`、ハードウェア設計節は `softAP()`）。

- [ ] **B-3. UNO R4 WiFi 内部の「SPI ブリッジ」（要検証／おそらく誤り）。**
  RA4M1 ↔ ESP32-S3 間は UART（内部シリアル）連結が定説。ジッタの根拠としてこの方式を持ち出すなら正確に。

- [ ] **B-4. 「`WiFiS3` が AT bridge を経由する」（要検証）。**
  UNO R4 WiFi の ESP32-S3 は古典的 AT ファームではなく Arduino 専用ファーム。さらに「SoftAP/multicast に不安があった」と書きつつ §tested-values では「テスト実装でマルチキャストが安定動作」と書いていて、採用判断の理由づけと結果が時制で衝突（「テスト前の懸念だった」と明記すれば済む）。

- [ ] **B-5. 「Mac 側は magic を探して同期し、`type` でペイロード長を判定して残りを読む」が実態と違う。**
  CTRL/BEAT/NOTE は全部 20 バイト固定で、しかも Serial には NOTE しか流れない（CTRL/BEAT は UDP）。「`type` でペイロード長を判定」は不要で誤解を招く。「magic で同期して固定 20 バイト読む」が正。

- [ ] **B-6. 視聴覚同時知覚閾「約 80 ms（聴覚が遅れる方向）」が無出典。**
  MOP の数値は 7 本も文献を付けたのに、この 80 ms は文献も ADR も無い。同時性窓は非対称（映像先・音後で広い／音先で狭い）で「80 ms」は丸め値。MOP 節を削っても §clock-sync の「設計上の固定遅延（受け入れ判断）」に残るので、出典を足すか「経験則」と明記して他と粒度を揃える。

- [ ] **B-7. MOP-7（起動時間）が本文で「§tbd に従い実測で再調整」と書かれているのに、§tbd の表に起動時間の行が無い。**

- [ ] **B-8. `platformio.ini`「必ず含める 2 行」が実態より少ない。**
  doc は `-I include` と `lib_extra_dirs = ../common/lib` の 2 行と書くが、実際の `test_v1/v2` の `ini` は各モジュールへの明示 `-I ../common/lib/<Module>` 群と `lib_ldf_mode = deep+` も要る（無いとビルドが通らないと `ini` のコメント自身が言っている）。

- [ ] **B-9. `SyncLogicData.sampleCount` の役割記述が矛盾。**
  ある表では「`clockSyncMinSamples` 以上で WaitStart 解除条件成立」と書きつつ、別の箇所では「Playing 遷移条件には使わない（鳴らない症状の温床）」「診断・デバッグ表示のみ」。Version 0.2 の方針（最初の BEAT で Playing）に合わせて統一。

- [ ] **B-10. `ScoreEvent.beatAt` の説明（doc 正）とコード側コメントの乖離。**
  doc は「`beatAt` は参考値、進行は `currentEventIndex` 駆動」と正しいが、`test_v1/node_02/include/score_data.h` 冒頭コメントは「`beatAt` は `startBeatNo` 相対の 1 始まり拍番号。effective=1 で beatAt=1 が発火」と古い実装の説明のまま。doc を正とするならコード側コメントを直す（`/spec-check` 対象）。

- [ ] **B-11. 「USB CDC は実質高速だが値は揃える」「ボーレート 115200」。**
  USB CDC ではボーレート設定は無視される。「値は揃える」と添えてあるので致命ではないが、115200 に意味があるかのように読める表現。

- [ ] **B-12. node_01「Serial は使用しない」と T1 試験「シリアルに beatNo/bpm が出る」の衝突。**
  本番は給電のみ、開発時はデバッグ Serial 接続、と書き分ければ解消。`platformio.ini` も `SERIAL_DEBUG=1` のままなので、本番ビルドで `0` に戻す手順がどこかに要る（Processing 連携時は `0` 必須と §tested-values にあるのに本番ビルド切替の言及が薄い）。

## C. 不確定要素（doc が §tbd に出している以外）

- [ ] **C-1. WBS タスク 121/122/123・マイルストーン M3–M7・週 W6–W11 が本文で参照されるのに、WBS 表もスケジュール表も本書に存在しない。**
  「提出用計画書」を名乗るなら最低限のスケジュール／体制図が要る。参照だけあって実体が外部依存なのは不親切。

- [ ] **C-2. test_v1 で「動作確認済」とする MOP-1（同期誤差）の実測値が本書に無い。**
  「マスタクロック方式により実機では 5 ms 以下を目標」とあるが、実際に何 ms 出たかが書かれていない。書ければ説得力が大きく上がるし、書けないなら「未測定（収束後の定常で再測定予定）」と明示。

- [ ] **C-3. §tbd「楽譜データスキーマ … 齋藤担当との {pitch, durationBeats} 形式との橋渡しは変換層で吸収」。**
  変換層の所在・担当・期限が未定。R-2 のクリティカルパスになりうる。
