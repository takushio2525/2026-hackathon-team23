# MOP検証方法の検討内容

報告者：梅澤颯太

日付：2026年6月16日

プロジェクト名：タクトーン～IMU ジェスチャーで奏でるArduino オーケストラ～

## 1. 何を検討したか

MOPの数値目標はすでに決まっているため，今回は「その数値をどうやって実機で確認するか」を検討した．
特に，ただ演奏を聞いて「できている」と判断するのではなく，ログや録音を使って後から確認できる形にすることを重視した．

検討した内容は大きく分けて次の3つである．

- どのMOPをどの方法で測るか
- ArduinoやProcessingからどのログを残すか
- 自動で判定できるものと，人が確認する必要があるものを分けること

## 2. 検討の方針

MOPには，同期誤差，通信遅延，拍検出精度，楽譜との一致，起動時間，パケットロス耐性などがある．
これらはすべて同じ方法では測れないため，MOPごとに検証方法を分ける必要があると考えた．

検討では，以下のように分類した．

| 分類 | 内容 | 検証の考え方 |
|---|---|---|
| ログで確認できるもの | 拍検出精度，通信遅延，楽譜との相違，パケットロス | ArduinoやProcessingのログを保存し，数値として集計する |
| 録音が必要なもの | 楽器間同期誤差，音階誤差 | 実際に出た音を録音し，波形や周波数を確認する |
| 測定条件を決める必要があるもの | 起動時間，テンポ追従 | どの状態を「起動完了」「追従完了」とみなすかを決めて測る |

## 3. MOPごとに検討した検証方法

### 3.1 拍検出精度

拍検出精度は，指揮棒を振った拍をシステムがどれだけ正しく検出できたかで確認する．
メトロノームを使い，80 BPM，120 BPM，160 BPMなどのテンポで一定時間振る方法を考えた．

例えば120 BPMで30秒振ると，本来の拍数は60拍になる．
指揮者ノードのログに出る拍検出回数を数え，54拍以上検出できれば90%以上となる．
ただし，同じ拍で2回検出された場合は誤検出として扱う必要がある．

この検証では，指揮者ノードのSerialログを使う．

### 3.2 通信遅延

通信遅延は，指揮者ノードから出た拍情報が楽器ノードに届くまでの遅れで確認する．
指揮者ノードの送信ログと，楽器ノードの受信ログを同じBEAT番号で対応付ける方法を考えた．

楽器側ログに`ahead`や`playAt`のような値が出ていれば，そこから通信遅延を間接的に計算できる．
1回だけではばらつきが大きいため，100拍程度のログを取り，平均値と最大値を見るのがよいと考えた．

この検証では，指揮者ログと楽器ログの両方が必要である．

### 3.3 楽器間同期誤差

楽器間同期誤差は，複数の楽器が同じタイミングで鳴っているかを確認するMOPである．
ログだけでは実際にスピーカから出た音のタイミングまでは分からないため，録音して波形を確認する方法を考えた．

複数の楽器ノードで同じ拍に音を鳴らし，PCやスマートフォンで録音する．
その録音をAudacityなどで開き，各音の立ち上がり位置を比較する．
最も早い音と遅い音の差が20 ms以内であれば，同期できていると判断できる．

この検証は自動化も可能だが，録音環境の影響を受けるため，最終的には波形を目視確認するのがよいと考えた．

### 3.4 楽譜との相違

楽譜との相違は，予定した音が予定した順番で鳴っているかを確認する．
Processing側のCSVログには，`partId`，`noteNumber`，`durationMs`，`seq`などが残るため，これを期待する楽譜データと比較する方法を考えた．

輪唱の場合は，各声部が同じタイミングで同じ音を鳴らすわけではない．
そのため，`partId`ごとに入り拍のずれを考慮して確認する必要がある．

この検証はプログラムで自動化しやすいと考えた．

### 3.5 音階誤差

音階誤差は，出力された音の高さが理論値に近いかを確認する．
単音を鳴らして録音し，チューナーやFFT解析で周波数を測る方法を考えた．

測定した周波数と理論周波数を比較し，cent単位の誤差を出せば数値として判断できる．
ただし，音色によっては倍音やノイズの影響があるため，単純なFFTだけでは正しく測れない場合がある．

そのため，音階誤差は録音解析と人の確認を組み合わせる必要があると考えた．

### 3.6 テンポ追従

テンポ追従は，指揮者が振るテンポを変えたときに，楽器側が何拍で追従できるかを見る．
80 BPMから120 BPMへ変えるなど，はっきりしたテンポ変化を与えてログを確認する方法を考えた．

ログに出るBPMの推移を見て，2拍以内に目標テンポ付近へ入るかを確認する．
目標テンポ付近の判断は，まずは±5 BPM程度を目安にするのがよいと考えた．

### 3.7 起動時間

起動時間は，電源投入から演奏できる状態になるまでの時間で確認する．
ただし，「どの時点を演奏可能とみなすか」を先に決める必要がある．

今回の検討では，次の状態がそろった時点を演奏可能状態と考えた．

- 指揮者ノードが起動している
- 楽器ノードがWi-Fiに接続している
- 楽器ノードの時刻同期が完了している
- Processing側でNOTEを受け取れる

この時間を5回測定し，平均値を使って判定する方法を考えた．

### 3.8 パケットロス耐性

パケットロス耐性は，一部の通信が欠落しても演奏が破綻しないかを確認する．
Processing側では`seq`の欠落を数えられるため，まずは実際の演奏中にどれだけ欠落するかを記録する方法を考えた．

ただし，通常の環境で自然にパケットロスが起きない場合，耐性の確認としては不十分である．
そのため，必要に応じて測定用の設定を入れ，疑似的に5%程度パケットを欠落させる方法も検討した．

このMOPは，数値ログだけでなく，聴感上演奏が止まらないか，大きく崩れないかも確認する必要がある．

## 4. プログラム化できそうな部分

検討した結果，MOP検証の一部はプログラムで自動化できると分かった．

自動化しやすいものは以下である．

- Processing CSVの`seq`欠落数を数える
- `partId`と`noteNumber`を期待楽譜と比較する
- 指揮者ログから拍検出回数を数える
- 楽器ログから通信遅延の平均や最大値を計算する

一方で，録音を使う同期誤差や音階誤差は，自動化できても録音環境の影響を受けるため，人の確認も必要である．
CPU負荷のように，現状のログだけでは測れないものは，ファーム側に測定用ログを追加する必要がある．

## 5. 足りないもの

検証方法を実際に使える形にするには，次のものが足りない．

- 指揮者ログ，楽器ログ，Processing CSVの実例
- どのMOP目標値を正式な合否基準にするかの最終表
- 楽譜一致チェック用の期待楽譜データ
- 同期誤差や音階誤差を確認するための録音ファイル
- CPU負荷を測るための測定用ログ
- パケットロスを自然発生で見るか，疑似的に落とすかの方針

## 6. まとめ

今回の検討では，MOPの数値目標を変更するのではなく，その数値をどう実機で確認するかを整理した．
結論として，ログで確認できるMOPと，録音や人の確認が必要なMOPを分けて考える必要があると分かった．

特に，拍検出精度，楽譜との相違，パケットロスはログから比較的確認しやすい．
一方で，楽器間同期誤差や音階誤差は，実際に出た音を録音して確認する必要がある．

今後は，実際のログを集めて，MOPごとに測定値と合否を表にまとめられるようにする．
