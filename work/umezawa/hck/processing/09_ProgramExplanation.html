orchestra_processing_final プログラム解説

Processing を触り始めた人でも追えるように、 「前提知識」「データの流れ」「各ファイルの役割」「音が鳴る理由」を順番に説明します。 まず全体をつかみ、そのあとコードの細部に入る構成です。

Processing / Java モード Serial 通信 20 byte NOTE フレーム Minim 音源 初学者向け
一言でいうと
このプログラムは、Arduino などの外部機器から届く「20バイトの NOTE データ」を Processing で受け取り、 Minim という音声ライブラリを使って、金管パートやリズム音として鳴らすための受信・音源プログラムです。
目的

外部機器から届いた音符データを、Processing 側で安全に読み取り、音に変換します。

中心になる考え方

通信で届くただの数字の列を、NotePacket という意味のあるデータに直してから使います。

読むときのコツ

最初から全部の文法を理解しようとせず、「入力、変換、確認、音」の流れで見ると楽です。

0. 先に知っておく基礎知識

このプログラムは、Processing、シリアル通信、音声合成が一緒に出てくるので、 いきなりコードから読むと難しく感じやすいです。 先に下の言葉をざっくり押さえておくと、後半の説明がかなり読みやすくなります。

Processing
絵、音、センサー連携などを作りやすくしたプログラミング環境です。Java に近い書き方をします。
スケッチ
Processing のプロジェクト単位です。このフォルダ全体が1つのスケッチとして扱われます。
.pde ファイル
Processing のソースコードです。同じフォルダ内の複数の .pde はまとめて1つのプログラムになります。

プログラム開始時に1回だけ動く関数です。画面、音声、通信などの準備を書きます。

Arduino と PC のように、機器同士でデータを1バイトずつ送る通信方法です。

小さな数字の入れ物です。1バイトは通常 0 から 255 までの値を表せます。

通信で送るデータの1セットです。このプログラムでは「20バイトで1つの音符情報」です。

この資料で一番大事な見方

送られてくるデータは、最初はただの数字の列です。 プログラムはそれを「これはパート番号」「これは音の高さ」「これは音量」と読み替え、 最後に音源へ渡しています。 つまりこのプログラムの本質は、数字の列を音楽の命令に翻訳することです。

1. 全体像

このスケッチは、外部機器からシリアル通信で届いた音符情報を読み取り、画面に状態を表示しながら音を鳴らします。 音符情報には、パート番号、音の高さ、強さ、鳴らす時間などが入っています。

1. 外部機器 Arduino などが NOTE フレームを送る
2. Serial Processing が1バイトずつ受信する
3. Decode 20バイトを NotePacket に変換する
4. Validate version、type、partId、seq を確認する
5. Sound PartManager と Voice が音を鳴らす
Arduino 側は「音符の命令」を数字にして送る

たとえば「Brass 1で、ドを、少し強めに、500ms鳴らす」という情報を、20個のバイトに分けて送ります。

Processing 側は、まず正しいまとまりを探す

通信は1バイトずつ届くので、先頭の目印 magic を探して「ここから20バイトが1命令」と判断します。

20バイトを、名前つきのデータへ直す

ただの数字列のままだと扱いにくいので、partId、noteNumber、velocity などに分解します。

変な命令を捨てて、正しい命令だけ音源へ渡す

バージョン違い、知らないパート、重複した番号などはここで止めます。音を鳴らす前の安全確認です。

Processing では、同じフォルダ内にある複数の .pde ファイルは、1つのスケッチとしてまとめて扱われます。 つまり、02_ConstantsAndParts.pde や 06_SoundVoices.pde に書かれた関数やクラスも、 メインファイル processing.pde からそのまま使えます。

読みやすい見方

「このファイルは何係か」「この関数は入力を何に変えるか」を追います。細かい文法はあとからで大丈夫です。

つまずきやすい見方

最初から byte、ConcurrentLinkedQueue、Waveform を全部暗記しようとすると、全体像を見失いやすいです。

Processing の複数ファイルは「分担表」として見る

このフォルダには .pde がたくさんありますが、別々のアプリではありません。 1つの大きなプログラムを、役割ごとにファイル分けしているだけです。 初めて読むときは「どのファイルが何係か」を先に押さえると、コードの場所を探しやすくなります。

3. メイン処理の流れ

Processing のプログラムは、基本的に setup() と draw() を中心に動きます。 このスケッチでは、そこにシリアル受信用の serialEvent() が加わります。 まずは次の3つを「いつ動くか」で分けて考えると理解しやすいです。

setup()

起動時に1回だけ。道具を机に並べる準備のような役割です。

draw()

実行中ずっと繰り返し。状態更新と画面描画を続けます。

serialEvent()

シリアルデータが届いたとき。届いたバイトを読み取り係へ渡します。

setup(): 最初に1回だけ実行される

setup() では、画面サイズを決め、音声出力を準備し、必要な管理クラスを作ります。 また、接続可能なシリアルポート一覧を取得します。 

void setup() {
  size(1000, 560);
  minim = new Minim(this);
  out = minim.getLineOut(Minim.STEREO, 1024);

  frameReader = new SerialFrameReader();
  partManager = new PartManager();
  ui = new UiView();
  logger = new Logger();

  refreshSerialPorts();
}

Minim は Processing で音を鳴らすためのライブラリです。 out はスピーカーにつながる出口のようなもので、あとで作った音をここに流します。

draw(): 毎フレーム繰り返し実行される

Processing では、draw() が1秒間に何十回も呼ばれます。 このプログラムでは、受信済みのデータを処理し、音の状態を更新し、画面を描画しています。 

void draw() {
  background(18);
  drainPackets();
  partManager.update();
  updateAppState();
  ui.draw();
}

ここで大切なのは、シリアル受信そのものは serialEvent() で行い、 実際の処理は draw() 側の drainPackets() で行っている点です。 受信と処理を分けることで、データが連続して届いても扱いやすくなっています。

4. シリアル通信の受け取り方

serialEvent(): バイト列を受け取る

serialEvent(Serial p) は、シリアルポートにデータが来たときに呼ばれます。 ここでは届いたデータを1バイトずつ SerialFrameReader に渡しています。 

void serialEvent(Serial p) {
  while (p.available() > 0) {
    frameReader.pushByte(p.read());
    NotePacket packet = frameReader.pollPacket();
    while (packet != null) {
      packetQueue.offer(packet);
      packet = frameReader.pollPacket();
    }
  }
}

packetQueue は、受信できた NotePacket を一時的に置いておくキューです。 キューとは「順番待ちの列」のようなものです。先に入れたものから順番に取り出されます。

なぜすぐ音を鳴らさず、キューに入れるのか
serialEvent() は通信データが届いた瞬間に呼ばれる処理です。 ここで音源処理や画面更新まで全部やると、処理が混み合ったときに受信が不安定になることがあります。 そこで、受信側では「読めたパケットをキューに入れる」だけにして、 メインの draw() 側で落ち着いて処理しています。

SerialFrameReader: 20バイトのまとまりを探す

シリアル通信では、データがいつもきれいに20バイトずつ届くとは限りません。 そのため SerialFrameReader は、受け取ったバイトを一度 buffer にためます。

NOTE フレームの先頭には、magic = 0x4F52 が入っています。 little-endian で送られるため、実際の先頭2バイトは 0x52、0x4F です。 pollPacket() はこの2バイトを探し、そこから20バイトを取り出してデコードします。

初学者向けポイント
magic は「ここから正しいデータが始まる」という目印です。 通信途中で余計なバイトが混ざっても、この目印を探すことで正しい位置に戻れます。

上の図は「何バイト目に何が入っているか」を表しています。 seq や timestampMs のように複数バイトを使う値は、あとで1つの数値に組み立てます。

位置 名前 意味
0-1 magic フレーム開始の目印です。値は 0x4F52 です。
2 version 通信仕様のバージョンです。このプログラムでは 1 を期待します。
3 type データの種類です。NOTE は 3 です。
4-7 seq 通し番号です。重複や抜けを検出するために使います。
8-11 timestampMs 送信側の時刻情報です。ログに記録されます。
12 partId どのパートの音かを表します。0x02 から 0x05 までを使います。
13 noteNumber MIDI ノート番号です。例: 60 は中央のドです。
14 velocity 音の強さです。通常は 0 から 127 です。
15 gate 1 なら音を鳴らす、0 なら音を止めるという意味です。
16-17 durationMs 何ミリ秒鳴らすかを表します。
18-19 予約領域 今は使っていません。将来の拡張用です。

little-endian とは

seq や durationMs のように複数バイトで表す値は、バイトの並び方を決める必要があります。 このプログラムでは little-endian、つまり「小さい桁のバイトから先に並べる」方式を使っています。 

int u16le(byte[] b, int off) {
  return (b[off] & 0xff) | ((b[off + 1] & 0xff) << 8);
}

& 0xff は、Java の byte を 0 から 255 の値として扱うための書き方です。

decodeNote() を日本語にすると

decodeNote() は難しそうに見えますが、やっていることはかなり素直です。 20バイトの箱から、決まった位置の値を取り出して NotePacket に入れています。

長さを確認する

frame.length != NOTE_FRAME_SIZE なら、20バイトではないので失敗として扱います。

先頭の目印を確認する

u16le(frame, 0) != MAGIC なら、正しい NOTE フレームの始まりではありません。

各位置から値を取り出す

frame[12] は partId、frame[13] は noteNumber のように、表の通りに読んでいきます。

扱いやすい形で返す

最後に NotePacket を返します。ここから先の処理は、バイト位置ではなく名前で値を扱えます。

6. 受信パケットのチェック

handlePacket() は、届いた NotePacket が正しいか確認してから音源へ渡します。 主に次のようなチェックをしています。

  • version が 1 か
  • type が NOTE を表す 3 か
  • gate が 0 または 1 か
  • partId が知っているパートか
  • 現在受け付ける設定のパートか
  • seq が重複していないか、抜けていないか

問題があるパケットは dropPacket() で破棄され、理由が画面とログに残ります。 これにより、通信ミスや設定違いに気づきやすくなっています。

チェック なぜ必要か 失敗するとどうなるか
version 送信側と受信側が同じ通信仕様を使っているか確かめるためです。 古い仕様や別仕様のデータを、間違って音符として読んでしまう可能性があります。
type NOTE 以外のデータが混ざったときに区別するためです。 音符ではないデータで音を鳴らそうとしてしまいます。
gate 音を出すか止めるかの値が正しい範囲か確認するためです。 2 や 255 のような意味不明な値を処理してしまいます。
partId このプログラムが知っているパートか確認するためです。 存在しないパートの音を鳴らそうとして、管理が分かりにくくなります。
seq 同じデータの重複や、途中で抜けたデータを見つけるためです。 通信の不安定さに気づけません。

7. パートID

02_ConstantsAndParts.pde では、使うパートIDが次のように定義されています。

定数 値 表示名
PART_BRASS_1 0x02 Brass 1
PART_BRASS_2 0x03 Brass 2
PART_BRASS_3 0x04 Brass 3
PART_RHYTHM 0x05 Rhythm

通常は、キー 1 から 4 で受け取りたいパートを選びます。 a を押すと、すべてのパートを受け付ける検証モードになります。

8. 音が鳴る仕組み

音を鳴らす部分は、初学者には少し難しく見えます。 でも大きく見ると、やっていることは「音の高さを周波数に変える」「音量を決める」「音の形を決める」「時間が来たら消す」の4つです。

高さ

noteNumber を midiToHz() で Hz に変換します。Hz は1秒間に何回振動するかです。

強さ

velocity を 0.0 から 1.0 の音量に変換し、MASTER_GAIN で全体音量を調整します。

長さ

durationMs の時間が過ぎたら release() を呼び、音を止める方向へ動かします。

PartManager: 音を作る係を選ぶ

PartManager.handleNote() は、パケットの gate、velocity、 noteNumber を見て、音を鳴らすか止めるかを判断します。 

if (packet.gate == 0 || packet.velocity == 0 || packet.noteNumber == 0) {
  noteOff(packet.partId, packet.noteNumber);
  return;
}
noteOn(packet.partId, packet.noteNumber, packet.velocity, packet.durationMs);

gate が 0、または velocity が 0 のときは「音を止める」と解釈しています。 それ以外なら noteOn() で新しい音を作ります。

金管パートとリズムパート

partId が PART_RHYTHM なら RhythmVoice、 それ以外なら BrassVoice が作られます。 

if (partId == PART_RHYTHM) {
  voice = new RhythmVoice(partId, noteNumber, amp, dur);
} else {
  voice = new BrassVoice(partId, noteNumber, amp, dur, brassWaveform);
}

velocity は 0 から 127 の値なので、 velocityToAmp() で 0.0 から 1.0 の音量に変換されます。 さらに MASTER_GAIN を掛けて、全体の音量を少し下げています。

Voice: 1つの音を表す基本クラス

Voice は「いま鳴っている1音」を表すクラスです。 パート、ノート番号、音量、鳴らす時間、開始時刻などを持っています。

update() では、指定された durationMs を過ぎたら release() して音を止め始めます。 finished() が true になると、PartManager によってリストから削除されます。

BrassVoice: 金管らしい音

BrassVoice は、midiToHz() で MIDI ノート番号を周波数に変換し、 Oscil で波形を鳴らします。 金管音らしくするため、倍音を含む波形を PartManager のコンストラクタで作っています。

MIDI ノート番号から周波数への変換

midiToHz(69) は 440Hz になります。 これは音楽でよく基準にされる「ラ」の音です。 ノート番号が1増えると半音上がり、12増えると1オクターブ上がります。 コード中の pow(2.0f, (noteNumber - 69) / 12.0f) は、この半音とオクターブの関係を計算しています。 

brassWaveform = WavetableGenerator.gen10(
  4096,
  new float[] { 1.00, 0.70, 0.50, 0.30, 0.20, 0.10, 0.05 }
);

また、音量は急に最大になって急に消えるのではなく、 attack、decay、sustain、release の段階を持っています。 これにより、少し自然な鳴り方になります。

段階 このコードでの意味
attackMs 音が0から指定音量まで立ち上がる時間です。このコードでは20msです。
decayMs 最大音量から少し落ち着いた音量へ下がる時間です。このコードでは40msです。
sustain 鳴っている間に保つ音量の割合です。ここでは最大音量の75%です。
releaseMs 音を止めるとき、急に切らずに0へ下げる時間です。このコードでは60msです。

RhythmVoice: 短く減衰するリズム音

RhythmVoice は、ノート番号によって減衰時間、周波数、波形を変えます。 低いノート番号は低い周波数で長めに鳴り、高いノート番号は短く鋭い音になります。

リズム音は鳴り始めた直後から音量が0へ向かうため、start() の中で releasing = true にしています。

9. 同時発音数の制限

MAX_VOICES_PER_PART は 4 です。 つまり、1つのパートで同時に鳴らせる音は最大4つまでです。

新しい音を鳴らす前に enforceVoiceLimit() が呼ばれます。 すでに4音以上鳴っている場合は、そのパートで一番古い音を release() してから新しい音を追加します。 これにより、音が増えすぎて重くなったり、音量が大きくなりすぎたりするのを防いでいます。

10. 画面表示

07_ScreenView.pde は、画面に見える部分をまとめて担当しています。 draw() の中で、次の5つを描いています。

  • drawHeader(): タイトルと簡単な説明
  • drawPorts(): シリアルポート一覧
  • drawStatus(): 現在の状態、受信数、破棄数など
  • drawWaveform(): 実際に出ている音の波形
  • drawHelp(): キー操作の案内

drawWaveform() では、out.left の音声バッファを線で描いています。 これは「いまスピーカーへ送られている音の形」を簡易的に表示しているものです。

11. ログ出力

Logger は起動時に CSV ファイルを作ります。 ファイル名は orchestra_processing_final_log_年月日_時分秒.csv のような形式です。

ログには、受信したパケット、破棄された理由、シリアル接続、音の開始・停止などが記録されます。 通信がうまくいかないときは、画面の lastWarning とあわせて CSV を確認すると原因を探しやすくなります。

12. キー操作

キー 動作
1 から 4 受け取りたいパートを Brass 1、Brass 2、Brass 3、Rhythm に切り替えます。
a すべてのパートを受け付けるモードと、選択中パートだけ受け付けるモードを切り替えます。
m ミュートを切り替えます。ミュート時は現在の音も止めます。
t 選択中パートのテスト音を鳴らします。シリアル接続なしでも音の確認ができます。
g 疑似 NOTE フレームを内部で作って注入します。受信処理のテストに使えます。
r シリアルポート一覧を再取得します。
d 現在のシリアル接続を切断します。

13. 実行手順

  1. Processing 4 を開きます。
  2. processing.pde を開きます。
  3. Minim ライブラリが入っていない場合は、Processing の Library Manager から Minim を追加します。
  4. 実行ボタンを押します。
  5. 画面左の Serial 一覧から Arduino などのポートをクリックします。
  6. 接続後、正しい NOTE フレームが届くと音が鳴ります。

14. よく見るべき変数

変数 意味
appState 画面上の状態です。PortSelect、Ready、Playing、Error などになります。
expectedPartId 現在受け付けるパートIDです。
acceptAllParts true のとき、どのパートIDも受け付けます。
muted true のとき、音を鳴らしません。
lastSeqByPart パートごとに最後に受け取った seq を覚えます。
receivedPackets 正常に受け取ったパケット数です。
droppedPackets 不正などの理由で破棄したパケット数です。
missingPackets seq の番号飛びから推定した、抜けたパケット数です。
lastWarning 最後に起きた注意メッセージです。画面にも表示されます。

15. 改造するときの考え方

受け付けるパートを増やしたい

02_ConstantsAndParts.pde に新しい PART_... 定数を追加し、 partName() と isKnownPart() にも追加します。 そのうえで PartManager.noteOn() で、どの Voice を使うか決めます。

音色を変えたい

金管音は PartManager の brassWaveform と、 BrassVoice の attackMs、decayMs、sustain、releaseMs を調整すると変わります。 リズム音は RhythmVoice の rhythmDecay()、rhythmFreq()、 rhythmWave() を変更します。

通信仕様を変えたい

20バイトの中身を変える場合は、送信側と受信側の両方を同じ仕様にする必要があります。 受信側では NOTE_FRAME_SIZE、decodeNote()、 必要なら makeNoteFrame() を変更します。

注意
送信側だけ、または Processing 側だけを変更すると、バイトの位置がずれて正しく読めなくなります。 通信フォーマットを変えるときは、必ず表を書き直してから両方のコードを合わせるのがおすすめです。

16. デバッグの見方

デバッグでは、いきなり全部を疑うより「音源」「受信」「データの中身」の順番で切り分けると楽です。 まず t で音源が生きているか確認し、次に g で受信処理を確認し、 最後に本物のシリアル通信を見ます。

症状 最初に見る場所 考えられる原因
t でも音が鳴らない Minim、PCの音量、ミュート状態 音源準備、ライブラリ、出力先の問題かもしれません。
t は鳴るが、実機では鳴らない シリアル接続、送信側のフレーム ポート違い、通信速度違い、フレーム形式違いが疑われます。
wrong part が増える expectedPartId と送信側 partId Processing 側が待っているパートと、送信側のパートが違います。
missing が増える seq の増え方 通信抜け、送信側の番号飛び、受信処理の遅れが考えられます。
  • 音が鳴らないときは、まず t でテスト音が鳴るか確認します。
  • t で鳴るなら、音源側は動いています。次にシリアル接続と NOTE フレームを確認します。
  • g で疑似 NOTE フレームを注入すると、受信処理から音源までの流れをテストできます。
  • wrong part が増える場合は、送信側の partId と Processing 側の選択パートが違います。
  • dropped が増える場合は、version、type、gate、partId などの値をログで確認します。
  • missing が増える場合は、送信側の seq が飛んでいるか、通信途中でデータが抜けています。

17. このプログラムの良い分け方

初学者が読みやすいポイントは、役割ごとにファイルが分かれているところです。

  • 通信のまとまりを探す処理は SerialFrameReader
  • バイト列を意味のあるデータに変換する処理は Protocol
  • 正しいデータかチェックする処理はメインの handlePacket()
  • 音を鳴らす判断は PartManager
  • 1音ずつの鳴り方は Voice
  • 画面表示は UiView
  • 記録は Logger

このように「通信」「変換」「検証」「音源」「画面」「ログ」が分かれていると、 不具合が起きたときにどこを見ればよいか判断しやすくなります。
