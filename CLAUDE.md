大学2年生（Git・共同開発未経験）が対象読者。様々なプロジェクトで流用できる汎用テンプレート。

## ドキュメントの言語

すべて日本語で記述する。コード内のコメントも日本語。

## ディレクトリ構成のルール

- 各ディレクトリには README.md を置き、「何のフォルダか」「不要なら削除OK」を明記する
- サンプルコードは最小限だが動作する状態を維持する
- `.gitkeep` は空ディレクトリの維持用。ファイルが追加されたら削除してよい

## ドキュメント整合性

以下のドキュメント間で記述が矛盾しないよう注意する：

- `README.md`（ルート）: テンプレートの全体像
- `CONTRIBUTING.md`: Git の使い方・開発ルール
- 各ディレクトリの `README.md`: そのフォルダ固有の説明
- `report/README.md`: LaTeX 報告書の使い方

機能の追加・削除（CI ワークフロー等）を行った場合、関連するすべての README を更新すること。

### LaTeX 報告書

```bash
cd report
docker run --rm -v "$(pwd):/workspace" -w /workspace \
  ghcr.io/paperist/texlive-ja:debian latexmk main.tex
```

### LaTeX 編集時の運用ルール（重要）

`.tex` ファイルを編集・新規作成した場合は、以下をワンセットで必ず行う。
LaTeX 報告書だけでなく、`work/shiozawa/ai_declaration/` 配下の生成AI利用申告書など、
このリポジトリ内の **すべての `.tex` プロジェクト** に適用する。

1. **Docker で自動コンパイルしてエラーゼロを確認**

   `.tex` が置かれているディレクトリ（`main.tex` と同階層）で:

   ```bash
   # Docker Desktop が未起動なら自動起動して待つ
   docker info > /dev/null 2>&1 || (open -a Docker && until docker info > /dev/null 2>&1; do sleep 2; done)
   # コンパイル実行
   docker run --rm -v "$(pwd):/workspace" -w /workspace \
     ghcr.io/paperist/texlive-ja:debian latexmk main.tex
   ```

   コンパイルが通らないままコミットするのは禁止。

2. **生成 PDF も `.tex` と同じコミットに含める**

   - 生成された `main.pdf`（または提出用にリネームした PDF）を必ず `git add` する
   - `.tex` の変更と PDF の更新は **同一コミット** にまとめる（分けない）
   - プッシュまでを 1 セットとして完了させる

   理由: 提出物 PDF やレビュー用 PDF がリポジトリに入っていないと、
   Docker 環境を持たないメンバーが中身を確認できず、提出・レビューが止まる。

3. **中間ファイルはコミットしない**

   `.gitignore` で除外済み: `*.aux` / `*.log` / `*.fls` / `*.fdb_latexmk` /
   `*.synctex.gz` / `*.dvi` / `*.out` / `*.toc` など。
   コミット対象は **`.tex` 一式（`main.tex`, `sections/*.tex`, スタイルファイル等）と `*.pdf`** のみ。

## Git ワークフロー（AI 補助作業のルール）

AI（Claude）が編集作業を行う際は、以下のフローを変更ごとに遵守する。

### 1. 変更ごとのブランチ運用

- 以下に当てはまる変更は、**変更内容ごとに適切なブランチ**を切る
  （例: `feature/imu-gesture`、`docs/simplify-specs`、`fix/udp-broadcast`）
    - **大規模な変更**（複数ファイルにまたがる機能追加・設計変更など）
    - 他メンバーが同じ箇所を触っていて**衝突が予想される変更**
    - レビューを通してから取り込みたい変更
- それ以外の**軽微で影響範囲が局所的、かつ衝突の恐れがない変更**
  （誤字修正・設定ファイルの小修正・小さなドキュメント追記など）は、
  `main` に直接コミットしてよい
- 判断に迷う場合はブランチを切る側に倒す
- 1 つのブランチには**独立した変更**だけを載せる。無関係な修正が混ざりそうになったら
  別ブランチに分ける

### 2. コミット → プッシュまでの標準フロー

変更をコミットするときは**毎回**以下を順に実行する：

1. **プル（衝突確認）**: `git pull --rebase origin <branch>` でリモートを取り込み、
   衝突があればその場で解消してから進む
2. **コミット**: グローバルルールのメッセージフォーマット（`[種別] 概要`）に従って
   機能単位で細かく分割する
3. **プッシュ**: `git push origin <branch>` でリモートに反映
4. 作業ブランチなら PR を作成（または既存 PR に追従）

プッシュまでを**1 つの変更の完了条件**とみなし、コミットしただけで止めない。

### 3. 手動変更の取り扱い

AI が編集していないファイルにも **`git status` / `git diff` で差分が残っていないか必ず確認**し、
以下の手順で扱う：

1. 差分がある場合、その内容を要約してユーザーに提示する
2. **コミットしてよいか・別コミットに分けるか・破棄するか**を確認する
3. コミットする場合は、AI が編集した変更とは**別コミット**にする（無関係な変更を
   混ぜない）
4. 上記「標準フロー」に従ってプッシュまで行う

`.vscode/`、`.env`、ローカル実験用ファイルなど**コミット対象か判断が迷う**ものは
必ず確認する。無断でコミット対象に含めない。
